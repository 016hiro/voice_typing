# v0.6.4 — ASR keep-alive (anti-compressor)

> **状态**：scope locked 2026-04-29，待开工（独立分支 `v0.6.4`）
>
> **主题**：阻止 macOS compressor 把 Qwen ASR (MLX) 权重压缩成"冷页"。今天 dogfood 暴露：闲置 1-2 小时后 1.7B 第一次 ASR 要 9-30s（基线 99-550ms），慢 17-56 倍。根因是 macOS unified memory compressor 把 mmap 的 2.5 GB 权重压缩，下次 access 要解压。Whisper backend 不受影响（CoreML 权重在 ANE 池里）。
>
> **触发**：v0.6.3 #R4 spike 排查中无意诊断出。这是 v0.6.0 切到 MLX backend 那天就存在的存量 bug，不是 v0.6.3 引入。

## 目标

- ASR loaded 状态下，**不论闲置多久**，下次 Fn 按下 ASR 推理延迟保持 baseline (~100-500ms)
- 保活成本可忽略（CPU < 0.5%，无可感发热 / 风扇）
- Refine 模型**不**保活（按用户决策：refiner 冷解压可接受，ASR 不可接受）

## 不做

- ❌ Refiner keep-alive —— 用户明确说 refiner 冷解压可接受
- ❌ Whisper backend 保活 —— CoreML 不走 mmap RSS 路径，不受 compressor 影响
- ❌ `mlock()` 钉物理内存 —— 需 root，不实际，且 24 GB Mac 钉 2.5 GB 占太大
- ❌ 让用户配置 keep-alive 频率 —— 默认值就行，省心智负担
- ❌ 启动时立即开 —— 等首次 ASR ready 后再开，避免冷启动期间的复杂状态

## 设计决策

### 1. 保活机制：定期 dummy transcribe

后台 timer，每 N 秒（默认 90s）触发一次极短 dummy transcribe，让 MLX runtime 真实触摸全部权重 page。compressor 看到"recently accessed" → 不会瞄准这些页。

```swift
// 简化伪代码
class ASRKeepAlive {
    private var timer: Timer?
    private weak var recognizer: QwenASRRecognizer?
    private let interval: TimeInterval = 90  // seconds

    func start(recognizer: QwenASRRecognizer) {
        self.recognizer = recognizer
        timer = Timer.scheduledTimer(withTimeInterval: interval, repeats: true) { _ in
            Task.detached(priority: .background) {
                try? await self.tick()
            }
        }
    }

    func stop() { timer?.invalidate(); timer = nil }

    private func tick() async throws {
        guard let r = recognizer, r.state == .ready else { return }
        // 200ms 静音 buffer：足够走完整推理 path 触摸 weights，
        // 又短到不会感知发热 / 占据 ANE 队列阻挡用户真实请求
        let silent = [Float](repeating: 0, count: 16000 / 5)  // 200ms @ 16kHz
        _ = try? await r.transcribeSync(samples: silent, language: .auto)
    }
}
```

### 2. Cadence 选择：90 秒

权衡（待 dogfood 数据 finalize）：

| 间隔 | CPU 成本 | 防压缩有效性 |
|---|---|---:|
| 30s | ~1.0% 持续 | 强（远早于压缩窗口） |
| **90s** | **~0.3% 持续** | **足够（compressor 决策窗口典型 5-10 分钟）** |
| 180s | ~0.15% | 边缘，可能失效 |
| 300s | <0.1% | 不可靠 |

90s = 默认。Cadence 写常量不入 Settings，省 UI 代码 + 心智负担。Dogfood 数据如显示太频繁 / 太松就调常量。

### 3. 启停时机

| 事件 | 行为 |
|---|---|
| App 启动 | 不开 |
| ASR backend 变为 `.ready`（首次 prepare 完成） | 开 timer |
| ASR backend swap (1.7B → 0.6B → Whisper) | stop 旧 timer，新 backend 若是 Qwen 则 start 新 timer |
| App 进入后台 / 失焦 | **不停** —— 用户切 app 写代码、回头 Fn → 必须命中 warm |
| App 即将退出 (`applicationWillTerminate`) | stop |
| Mac 进入 sleep | OS 自动暂停 timer，wake 后自动恢复（NSTimer 默认行为） |

特殊：用户主动按 Fn 录音期间，timer tick 会被 `state != .ready` 跳过（recognizer 在 transcribing 状态）。

### 4. dummy input 选 200ms 静音

- 太短（< 100ms）：可能某些 backend 短路 path，不真正过 weights
- 太长（> 500ms）：占 ANE 队列，可能阻挡同时发生的真实 request（罕见但可能）
- 200ms：mostly silent → VAD 会输出空段 → transcribe 返回空字符串。但**关键的 weight access 已发生**

### 5. 能不能验证它真的有效？

`pmap`/`vmmap PID` 看 MLX 权重 region 是否仍标 `[ active ]` vs `[ compressor ]`：

```bash
vmmap $(pgrep VoiceTyping) 2>&1 | grep -i "MALLOC\|compressed\|MLX"
```

或在 dogfood 期跑一周后看 `tail asr_ms` log —— 健康 baseline 应保持 100-500ms 范围，无 9000-30000ms outlier。

### 6. 跟 v0.6.3 refiner 的关系

v0.6.3 实施 LocalMLXRefiner 时，**不要**把 refiner 也接进 ASRKeepAlive。Refiner 完全可以被 compressor 收，第一次 refine 慢几秒可接受（refine 本来就有"等"的心智模型）。Settings UI 上加一句 "First refine after long idle may take 5-30s due to macOS memory management."

## 风险

- **dummy transcribe 影响 mic / audio pipeline 状态**：keep-alive 直接调 model 层（不经 LiveTranscriber / VAD），跳过 mic capture，应无影响。但要验证 `Qwen3ASRModel.transcribe(samples:)` 是 stateless wrt streaming session
- **多 backend 并存场景**：用户在 Settings 里同时切多个 backend，旧 timer 和新 timer 短暂重叠。state machine 简单：`stop` 立即 invalidate timer，新 start 之前一定 stop
- **ANE 队列争用**：极小 dummy transcribe + 90s 间隔 ≈ 不会撞用户真实请求；万一撞了，用户请求会 lock 到 transcribeLock 后面，多等 ~50ms（dummy 的执行时长），可接受
- **电池**：MacBook 上每 90s 唤醒一次 ANE，估算 < 0.1% 电池/小时影响。Dogfood 期注意
