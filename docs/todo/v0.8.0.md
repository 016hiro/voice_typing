## 副线 #S1：短句 refine skip heuristic

> **状态**：scope + 数据 + Python prototype 已就绪，待 Swift 实现（2026-05-15 更新）。

**动机**：v0.7.3 #B1 telemetry 跑出来 inline local refine 里 **~50% 是 no-op**（input == output），平均每次仍付 ~1700ms LLM round-trip。pre-LLM 的 input-side heuristic 应该能把这部分跳过去。

### 阶段 1：rule heuristic（已数据验证，2026-05-15）

离线 replay 工具 `Scripts/analysis/skip_refine_replay.py` 在 303 条真实 refine 上扫了 7 个变体，**变体 C** 是 sweet spot：

任一命中即走 refine（全 miss → skip）：

| 条件 |
|---|
| 长度 ≥ 40 字符 |
| 含中文 filler：`啊 嗯 呃 唉 哦 嘛 呢 那个 就是 这个` |
| 含英文 filler：`um / uh / er / hmm / like / you know / kinda / sorta / basically / literally / i mean` |
| 含中文 stutter：正则 `(.{1,2})\1` |
| 含英文 stutter：正则 `\b(\w+)\s+\1\b` |
| 含中文数字串：≥2 个 `零一二三四五六七八九十百千万亿点` |
| 含 unspaced code-switch：`[CJK][A-Za-z]` 双向 |
| 含全角标点紧贴 ASCII：`[A-Za-z0-9][。？！]` 双向 —— v0.7.3 #B2 落地的 ASCII . 规则的对偶 guard |

数据：precision 95.7%（47 skipped / 45 TP / 2 FP），节省 14% 累计 LLM 时间。剩 2 个 FP 中 1 个是 semantic 重排（`还覆盖的不够全 → 覆盖的还不够全`，任何规则算法都搞不定），另 1 个 code-switch 边缘 case。

### 阶段 2：hotword 回声 guard（必须，2026-05-15 加入 scope）

**为什么必加**：变体 C 单独跑会**错误跳过**短热词 mishearing 场景。手测 11 个 case，8 个 false-skip：

| input | C alone | 期望 |
|---|---|---|
| `Cloud Code` | skip ❌ | refine → Claude Code |
| `曲文` | skip ❌ | refine → Qwen |
| `千问怎么样` | skip ❌ | refine → Qwen 怎么样 |
| `钱问可以吗` | skip ❌ | refine → Qwen 可以吗 |
| `cloudcode` | skip ❌ | refine → Claude Code |
| `曲文是个好工具` | skip ❌ | refine → Qwen 是个好工具 |
| `我想用 Qwen 试一下` | skip ❌ | refine → 加空格规整 |
| `kuwen` | skip | (算法搞不定，需要 hint) |

这些 case 在 dogfood 数据里"碰巧"被 length / codeswitch / filler 规则盖住了——所有用户对话都比较长或代码混合。产品上不能依赖这种巧合。

**两层 guard 设计**：

- **Layer 1（substring，零新代码）**：复用 `GlossaryBuilder.matchedEntryIDs(in: input, entries:)`（`GlossaryBuilder.swift:148`，v0.3.0 起服务于 `lastMatchedAt` LRU）——term 或 `pronunciationHints` 任一 substring/word-boundary 命中 → 不 skip。
- **Layer 2（phonetic fuzzy）**：jieba 切词等价物 + 拼音化 + Levenshtein。覆盖用户**没填 hint** 的 mishearing case（"Qwen → 曲文" 即使 `pronunciationHints=[]` 也能匹配）。Swift 实施：
  - 切词：`NLTokenizer(unit: .word)`（Foundation 自带，零依赖）
  - 拼音化：`CFStringTransform(..., kCFStringTransformMandarinLatin) + kCFStringTransformStripDiacritics`
  - 距离：手写 Levenshtein ~30 行
  - 候选形态：1-token + 2-gram + 3-gram concat（保 jieba 把 `曲文` 切成 `["曲","文"]` 后仍能聚合匹配 `qwen`）
  - 阈值：`distance / variant_form_length ≤ 0.3`
  - min_form_len = 4（单字中文拼音太短，避免 `文` 一字伪命中 `qwen`）

Python prototype 已写并验证：`Scripts/analysis/hotword_phonetic_replay.py`。jieba 切词解决了"自然中文短语跨字伪音节匹配 hotword"的核心 FP 问题（如 `之前应该` 不再被切成 `qianyin` 误命中 `千问`）。

Layer 2 已知 trade-off：dogfood 上加 1 个误拦（`原文删除了吗？` 触发 `原文` vs `千问` ratio 0.286 真音节碰撞）。代价仅是用户多等 ~1.5s 看到一样输出，**不丢数据**——跟 "Cloud Code 被注入" 相比明显是后者更痛，对称性破坏可接受。

## ~~副线 #S2：Skip-refine memoization cache~~ — 数据否决

> **状态**：scope 后被数据驱动否决（2026-05-15），不实施。

**否决依据**：`Scripts/analysis/skip_refine_replay.py` 在 303 条 dogfood refine 上跑 #S2 simulation，**cache 命中率 0%**——voice typing 用户在 269 个 session 里没说过任何完全重复的句子。memoization 的前提（input 高频重复）在这个场景下不成立。

#S1 + hotword phonetic guard（见 #S1 阶段 2）已经足够覆盖 short-skip 收益。replay 工具留作回归监控——将来某次 dogfood 跑出非 0% 命中率即可重启评估，但目前无此信号。
