# v0.5.1 — UX 小补丁 + 性能基线 + Debug 数据捕获 + Live mode dogfood gate

> 本版本目标：四条线并进。
> 1. **UX 小补丁**：修两个 backlog 老 bug（首次启动检测不完整模型 + 录音时长视觉提示）
> 2. **性能基线**：(A) 量化 Qwen 加载耗时；(B) 修上游 weights 双读；(C) 三 backend × 全 fixture 的转写速度 benchmark。整体目的是把"快慢"从感觉变成数字
> 3. **Debug 数据捕获 toggle**：用户开了之后录音/转写/注入全数据落盘，便于回溯分析。**捕获范围已拍 7 项**（见 "Debug 数据捕获 toggle" 段）。**Refine I/O 捕获不在本版本**——独立成 refine 专版处理。
> 4. **Live mode dogfood gate**：不在 v0.5.1 加 live mode 新特性，期间收集真实数据给 v0.5.2 输入
>
> **状态**：决策已锁，执行中。
>
> **预估范围**：~700 行代码改动 + 单元/E2E 测试。3-5 天，主要时间花在 debug capture 框架和 dogfood 等待上。

## 设计决策（已确认）

### Live mode 范围

- **不在 v0.5.1 加 live mode 新特性**。VAD auto-stop / Live UI / Live + refine 全部留给 v0.5.2，**等真麦克风 dogfood 数据稳了再上**。理由：v0.5.0 fixture benchmark 是 100% 但真世界噪声未验，再叠新特性会让回归源难定位。
- **Live mode dogfood 期间不加 toggle UI**。继续 UserDefaults 隐藏开关。等 v0.5.2 决策。

### UX 补丁

- **首次启动检测策略**：检查 `models/<backend>/AudioEncoder.mlmodelc/weights/weight.bin`（WhisperKit）/ 关键 safetensors（Qwen）的存在 + 大小是否 ≥ 100KB。任一失败 → 标记半成品 → 删掉那个子目录让上游重下。**不删整个模型目录**（可能误删别的 backend 的好文件）。
- **录音时长视觉提示触发点**：
  - Live mode 上限 600s，60 秒内用户感知不到，但接近 540s 时提示
  - Batch mode 上限 60s，接近 50s 时提示（剩 10s 倒计时）
  - 提示形态：胶囊文字短暂 flash "10s 剩余"（同 v0.4.5 的 flashInfo 机制），不强制中断录音
  - 录音时长来源：`audio.maxDuration` 已是 per-start 参数，胶囊订阅 levels stream + 自己算 elapsed

### 性能基线（A + B + C）

诊断 + 量化三件套，互相独立但归一类。目的：把"加载快慢 / 转写快慢"从感觉变成数字，给后续优化和 backend 默认值选择提供 evidence。

**A. Qwen 加载量化 (`prepare()` 加 timing log)**：在 `QwenASRRecognizer.prepare()` 各阶段（vocab load / audio encoder weights apply / text decoder weights apply / `MetalBudget.pinMemory` / 第一次 inference warmup）打 wall-clock 毫秒。可见在 `Log.app.info` 默认级别。无 dogfood 数据收集，纯诊断输出。

**B. 修上游双读**：上游 `WeightLoader.loadWeights(into: audioEncoder)` 和 `loadTextDecoderWeights(into: textDecoder)` 各自把所有 safetensors 文件 mmap + 解析头 + 构造 MLXArray 字典一遍，然后 filter 关心的 prefix 用、其余丢。这是确认的浪费——读了同一个文件两遍，做了两份元数据解析。
- **预期收益修正**：MLX `loadArrays` 是 mmap-based（C++ 层 `mlx_load_safetensors`，thin Swift wrapper），所以**不是 2x 物理 I/O**——主要省的是元数据解析 + Swift 字典 + MLXArray 包装的双倍分配。1.7B 模型 ~1000-2000 tensors 估省 100-500ms。**先 A 测，再决定值不值真改 B**。
- **如果决定改 B，路径选择**（按风险 / 维护成本排）：
  1. **上游 PR**：给 `soniqo/speech-swift` 提 patch 把 load+filter 合一。低 LOC、低风险、merge 周期未知。**首选**
  2. **下游 wrapper**：自己写 `loadAllOnceThenApply`，单 load → 分别 filter+apply。难点是上游 `applyConv2dWeights` / `applyEncoderLayerWeights` / `applyQuantizedDecoderLayerWeights` 等可能是 fileprivate；如果是，需要镜像实现一份（重）或绕过。**只有 PR 卡住才走**

**C. 三 backend 速度 benchmark**：用 `Tests/Fixtures/` 里的 16 个 fixture，分别跑 Whisper large-v3、Qwen 0.6B 4bit、Qwen 1.7B 8bit 的批量转写，测 wall-clock + RTF（real-time factor = transcribe_ms / audio_ms，<1 表示快于实时）。每 backend 第一个 fixture 当 warmup（discard 计时），消除 cold start 干扰。
- **目的**：(1) 给"backend 默认值选 1.7B 还是 0.6B"提供 evidence；(2) 给后续讨论 "本地 MLX refiner 用哪个 size" 提供 baseline；(3) 上游升级 / 自己改 backend 后能 regress check
- **输出格式**（recap 段）：
  ```
  fixture                          dur     │ Whisper        │ Qwen 0.6B 4b   │ Qwen 1.7B 8b
  ─────────────────────────────────────────────────────────────────────────────────────────
  fleurs_zh_short_3s_f_1554        3.3s    │ 950ms / 0.29×  │ 180ms / 0.05×  │ 290ms / 0.09×
  ...
  ─────────────────────────────────────────────────────────────────────────────────────────
  average                                  │ X ms / X.XX×   │ Y ms / Y.YY×   │ Z ms / Z.ZZ×
  ```
- **gating**：`VT_BENCHMARK=1` env，跟 VAD benchmark 一样，不进 `make test-e2e` 默认路径
- **入口**：`make benchmark-speed`
- **跳过策略**：模型未下载就 `XCTSkip` 该 backend 的整段，其他 backend 继续跑（用户可能只装了 Qwen，没装 Whisper）
- **内存**：3 backend 串行 load + bench + unload，避免同时占住 GPU 内存。Whisper backend 上游不 expose unload，放最后跑

### Debug 数据捕获 toggle

**目的**：用户主动开启后，把若干次"按 Fn → 出文字"的全过程数据落盘，便于事后离线分析（不同 VAD 配置 replay、不同 backend 对比、找高频误识别词等）。**默认关**，**Settings → Advanced 一个明显的 toggle + 警告**。

**七个待讨论项**（按"先拍 → 后实现"顺序，每项独立讨论 + 勾选确认）。先大致约定：以下"轻量元数据"四类**默认开**且不另立讨论项 —— 体积 < 1 KB / session，价值高，无隐私争议：
- VAD 事件时间线（speechStarted / speechEnded / force-split）
- 段级 transcribe raw text + 耗时 + HallucinationFilter 决策
- LatencyTracker 全字段 + 模型加载各阶段耗时（A 档自然产物）
- Inject 焦点 app bundleID + 是否成功

下面 7 项是**真正需要拍板**的：

#### 待讨论项 #1：音频原始 buffer 存不存

- **选项**：默认开 / 默认关 / Settings 子开关让用户细控
- **取舍**：
  - 价值最高——所有 ASR/VAD/segmentation replay 都依赖原始音频，没它整个 capture 就只是文本日志
  - 隐私敏感度也最高——存的是用户对所有焦点 app 说过的所有话
  - 但用户已经主动开了 debug capture toggle，自负责任
- **倾向**：默认开。Settings UI 文案明确："Audio is captured for offline analysis. Files stored locally only, never uploaded. Auto-purges after N days."
- **决策**：[x] 拍：默认开。理由：toggle 本身默认关，用户主动开启即自负责任；不存 audio 等于 capture 没价值（VAD replay / 跨 backend 对比都需要原始音频）

#### 待讨论项 #2：Refine（LLM）输入/输出存不存

- **选项**：开 / 关 / 默认关
- **取舍**：
  - 价值：能离线复盘 refine 的 prompt → 改 prompt 工程
  - 隐私 + 安全：LLM 调用的 request body 可能携带 API key 在 Authorization header；需要先确认 `LLMRefiner` 实现**绝不**把 key 落盘
  - 文本量：每次 refine 输入输出 ~200-2000 chars，累计很轻
- **倾向**：默认开 prompt + response 文本，**绝不**存 API key / Authorization header。需要先验证 `LLMRefiner.refine()` 不暴露 credentials 到日志路径
- **决策**：[x] 拍：**v0.5.1 不做**。理由：refine 整体优化（含 prompt 工程、credential 路径审计、capture）独立成一个版本（暂记 v0.5.3 或更靠后）；现在 v0.5.1 范围只覆盖 ASR / VAD / inject 链路的数据捕获，refine I/O capture 不进 `DebugCaptureWriter` schema 的 v1

#### 待讨论项 #3：存盘格式

- **候选 1**：`<support-dir>/debug-captures/<yyyy-mm-dd_hh-mm-ss_uuid>/{meta.json, audio.wav, segments.jsonl}` 每 session 一个目录
  - 优点：Finder 直接浏览，可直接 zip 发送，schema-flexible
  - 缺点：几百次录音就是几百个小目录
- **候选 2**：单 SQLite `<support-dir>/debug-captures.sqlite`，audio 用 BLOB 或外链
  - 优点：query 方便（"找所有 hallucination filtered 的段"）
  - 缺点：依赖 SQLite client 才能看，audio BLOB 让 db 文件膨胀
- **候选 3**：混合 —— 候选 1 落盘 + 后期 CLI 工具按需聚合到 SQLite 做 query
  - 优点：分析工具可以独立演进，不影响落盘 schema
  - 缺点：两套代码
- **倾向**：候选 1（最简单 + 用户能直接看）。SQLite query 留给以后分析工具自己解决
- **决策**：[x] 拍：候选 1（每 session 一个目录）。理由：Finder 浏览 + zip 发送 > query 方便；schema 还会变，SQLite 改起来重；CLI 聚合工具留给 v0.5.2 看真实需求再立

#### 待讨论项 #4：自动清的"N 天"

- **选项**：N = 7 / 14 / 30 / 不清
- **磁盘估算**（前提 #1 拍成"默认存音频"）：
  - 1.7B 用户 / 100 次录音/天 / 平均 8s 音频 = 51MB/天
  - N=7：360MB
  - N=14：720MB
  - N=30：1.5GB
- **倾向**：N=7。如果用户主动想留长期数据，Settings → Advanced 加一个 "Auto-purge after" 选项让 7/14/30/never 切换
- **决策**：[x] 拍：N=7 默认，Settings → Advanced 提供 `Auto-purge after: 7/14/30/never` dropdown。理由：日常 ~360MB 可控，重度用户有逃生口

#### 待讨论项 #5：磁盘容量硬上限

- **选项**：X = 1 GB / 5 GB / 不限 / 跟 N 天联动
- **取舍**：自动清是按时间，但如果用户某天疯狂录 1 小时 + 1.7B，单天可能 > 400MB，N 天到期前已经塞爆磁盘
- **倾向**：X = 5 GB 硬顶（远超日常 N=7 的 360MB），到顶后清最旧 capture 直到 < X 的 80%
- **决策**：[x] 拍：5 GB 硬顶 + 触顶清最旧到 < 80% (4 GB)。理由：N=7 日常 14× 头空；触顶清旧而非拒新避免静默丢数据

#### 待讨论项 #6：分析方法

需要支持的分析路径决定了存什么、怎么存。下面是候选清单：

| 分析 | 需要存什么 | v0.5.1 是否要支持 |
|---|---|---|
| **VAD config replay**：audio + 不同 `StreamingTuning` → 看分段差异 | audio.wav + meta（lang/context）| 是，主用例 |
| **ASR backend 对比**：相同 audio → 三 backend → diff transcript | audio.wav + 当时用的 backend | 是 |
| **HallucinationFilter ±**：filter 拦的段是不是真该拦 | 段级 raw text + filter 决策日志 | 是 |
| **加载耗时分布**：从 A 档 timing log 聚合 | 模型加载各阶段 ms | 是（A 档自然产物）|
| **Live mode drain 时间分布**：Fn↑ 后到最终 inject 完成的延迟 | 每段 inject 时间戳 | 是 |
| **Focus-change drop 频率**：live mode 段被 focus 检查丢的频率 | inject 段日志 + 焦点 bundleID | 是 |
| **高频误识别词**：哪些词 ASR 错了 vs 用户改了 | 还需要"用户修正捕获" —— 很难做 | **不做**，留给后续 |

- **决策**：[x] 拍：上表前六项都做。**高频误识别词不做** —— 需要"用户修正捕获"是另一个独立 feature，v0.5.1 不碰

#### 待讨论项 #7：分析工具形态

- **选项**：
  - **Swift CLI binary** `vt-analyze` 子命令：跟 app 一起 ship 在 `Contents/MacOS/`，用户终端调用
  - **独立 Python notebook 模板**：放 `Scripts/analyze.ipynb`，用户开 Jupyter 自己跑
  - **不做工具，只 ship 数据格式 + 文档**：让用户拿 `jq` / `sqlite3` 自己 query
- **取舍**：v0.5.1 实现深度受限——schema 还没稳定，工具跟着改成本高
- **倾向**：v0.5.1 **不做工具**，只 ship 数据 + 文档（`docs/debug-captures.md` 说明 schema + 几个常用 jq one-liner 示例）。工具留给 v0.5.2 或之后看实际需要再做
- **决策**：[x] 拍：v0.5.1 不做工具。只 ship `docs/debug-captures.md`（schema + jq one-liner 示例）。理由：schema v1 还会演进，工具会跟着改两次成本高

#### 实测数据（M-series Mac，2026-04-21）

cached=yes 启动，单位 ms：

| Run | Backend | total | dl_init | tokenizer | audio_w | text_w_pin | warmup |
|---|---|---|---|---|---|---|---|

- **warmup 头次 ≥ 2× 后续**：1.7B 首次 598ms（Metal JIT），后续 180ms（kernel 已 cache）

## 风险与未知

- **首次启动检测的"完整性"判定可能误报**：WhisperKit / Qwen 的内部文件结构未来可能变。文件大小阈值是粗糙信号；如果上游升级模型变体导致文件大小自然下降到阈值以下，会误删触发重下。Mitigation：阈值给得保守（100KB 远低于真实 ~MB 级权重），上游变更时改阈值。
- **录音时长提示触发频率**：用户大部分录音都在 < 30s 范围，cap 提示触发率本来就低。如果 dogfood 期间没人触发，证明这个 UX 的实际价值有限，留作纯防御性 feature。
- **B (修双读) 实际收益不确定**：MLX `loadArrays` 是 mmap-based，所以双读不等于 2x I/O。预期收益 100-500ms 但有可能更小（如果 MLX 内部已经做了 mmap-level 复用）。**A 测出来才知道值不值**。如果 A 显示双读耗时 < 200ms，B 直接砍掉，工时给别处。
- **C (速度 benchmark) 在不同硬件上数字不同**：M1 / M1 Pro / M2 / M3 / M4 内存带宽和 GPU 都差几倍。本机数字只是参考，不是产品指标。devlog 里写数字时要标注硬件型号（CPU 字符串 + RAM）。如果有 CI 跑 benchmark 要单独评估硬件波动 noise，目前 v0.5.1 不上 CI
- **Debug capture 数据可能很大**：1.7B 模型 + 用户日常 100 次录音 / 天，每次平均 8s 音频 = 800s × 16kHz × 4B = 51MB/天的 audio。一周 ~360MB。**N 天保留策略要选对**——默认 7 天可能就 GB 级，需要明显 Settings UI 说明。
- **Debug capture 隐私**：用户可能自己不记得开了 toggle，然后就一直在录所有语音。Mitigation：每 N 次启动 / capture 数到达阈值时弹一次温和提醒"You have X captures in storage, manage them in Settings → Advanced"。决定阈值之前先 ship 一版无提醒看 dogfood 反馈。
- **上游 PR (B 的首选路径) merge 周期**：未知。upstream `soniqo/speech-swift` 不是 Anthropic 官方，但活跃 maintainer。如果一周不响应，走本地 fork 路径（已纳入决策树）。
- **Live mode dogfood 周期**：建议至少 5 个工作日 + 横跨多种使用场景（IM / 邮件 / 文档 / 代码注释）。如果不到一周就发现 P0 bug，先发 v0.5.1 修，dogfood 时钟重置。
