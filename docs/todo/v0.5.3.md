## 设计决策

### Hands-free 模式（v0.5.3 主线）

**状态**：2026-04-23 /think 拍板。7 个 UX 问题 + 1 个抽取重构决策已固化。

#### 7 个决策

| # | 问题 | Decision | 理由 |
|---|---|---|---|
| 1 | tap-vs-hold 阈值 | **200ms**，在 Fn↑ 时判定（音频从 t=0 一直录） | 80-100ms 典型 tap × 2 倍头空间；决策放 Fn↑ 不丢音频 |
| 2 | Hands-free 录音上限 | **跟着 timing × backend 浮动**：见 `RecordingPolicy.maxDuration(timing:backend:)` | hands-free 不独立配置 cap，跟 hold 路径同源 |
| 3 | 静默判定 | **1.5s post-speech 静默** 触发 auto-stop | 舒适的句尾停顿；不是 VAD min-silence 倍数（语义不同）|
| 4 | 取消手势 | **再 tap 一次 Fn** = cancel + discard | 对称入口；单权限；不引入 esc |
| 5 | 状态 UI | **capsule 颜色变 + "HF" 角标 + 前 3s overlay "tap Fn to cancel"** | 与 hold mode 视觉区分，防 "我在被录吗" 焦虑 |
| 6 | 适用范围 | **Qwen 后端的 3 种 timing 模式都支持**（one-shot / post-record / live）；Whisper 不支持（toggle 灰掉）| 不强制用户切 live 才能用；Whisper 等 v0.6.0 协议化主题 |
| 7 | 默认开/关 | **OFF**（dogfood opt-in），v0.5.4 验证后再考虑翻 ON | tap 之前是 no-op，改语义需要验证 |
| 附 | no-speech 自动取消 | **10s** 内未触发 speech-start → cancel | 5s 太严（用户翻笔记 7s 起念会被误杀）|

#### RecordingPolicy 抽取（顺带做的事）

当前 `AppDelegate.swift:368` 是 inline 决策 `useLive ? 600 : 60`。加 hands-free 就成第二处复制——趁机抽出 single source of truth：

新文件 `Sources/VoiceTyping/Support/RecordingPolicy.swift`：
```swift
enum RecordingPolicy {
    static func maxDuration(timing: TranscriptionTiming, backend: ASRBackend) -> TimeInterval {
        (timing == .live && backend.isQwen) ? 600 : 60
    }
}
```

调用点：`startRecording`（hold 路径）和新 hands-free 路径都查它。改 cap 只能改这一处。文档同步写到 `architecture.md` "Recording duration cap" 段。

#### 状态机（高层）

```
Fn↓ (any duration)             Fn↑
  │                             │
  ├─ startRecording (t=0)       ├─ duration = now - Fn↓ time
  ├─ start audio + (live? VAD-pump : VAD-watchdog if hands-free)
  └─ if in_handsfree → CANCEL   ├─ if handsFreeEnabled
                                │   AND duration < 200ms
                                │   AND backend.isQwen:
                                │     → enter hands-free
                                │     → arm 10s no-speech timer
                                │     → arm silence watchdog (idle until VAD .speechEnded)
                                │     → swap capsule UI
                                │
                                └─ else: stopRecording (current behavior)

silence watchdog (only in hands-free):
  on VAD .speechStarted → cancel no-speech timer; cancel pending stop-after-silence
  on VAD .speechEnded → schedule "stop in 1.5s"
  on stop fires → stopRecording (normal flush + inject path)
  on Fn-tap during hands-free → cancel (discard audio, no inject)
  on RecordingPolicy.maxDuration cap → stopRecording
```

### Debug Capture writer 修补（v0.5.2 dogfood 揪出）

#### Bug 1：meta.json 缺失（87% session 失数据）

`DebugCaptureWriter.begin()` 只创建目录不写 meta。任何不走 finalize/abort 的退出路径（crash / force-quit / early stopRecording bail）都让 meta 永远缺席。dogfood 数据里 303 个 session 只有 37 个有 meta（**12% 数据完整率**）。

**修法**：`begin()` 立即写 partial meta（`startedAt` / `backend` / `language` / `liveMode` / `frontmostBundleID` / `profileSnippet` / `asrContextChars` 都已知）。`finalize()` / `abort()` update 加上 `endedAt` / `totalAudioSec` / `totalSegments` / `totalInjections`。

落地点：`Sources/VoiceTyping/Support/DebugCaptureWriter.swift:130` `private init` 后 enqueue 一个写 meta 的 task。约 15 行。

#### Bug 2：timestamp 秒精度

```swift
encoder.dateEncodingStrategy = .iso8601    // ← 默认无 fractional seconds
```

写出来的是 `2026-04-21T19:13:19Z`（秒级），导致 `endedAt == last_inject.timestamp` 经常落同一秒 → `live_drain.py` 报 0ms 全失效。

**修法**：

```swift
let formatter = ISO8601DateFormatter()
formatter.formatOptions = [.withInternetDateTime, .withFractionalSeconds]
encoder.dateEncodingStrategy = .formatted(formatter)
```

落地点：所有 `encoder.dateEncodingStrategy = .iso8601` 调用（appendJSONL / finalize / abort，3 处）。约 5 行。

注意：分析脚本 `_common.py.parse_iso` 已经能解析有/无 fractional seconds，无需改。

#### Bug 3 配套：profile snippet 写入 meta

`DebugCaptureWriter.Meta` 有 `profileSnippet` 字段，但 dogfood 数据里全是空。检查 `AppDelegate.startRecording` → `DebugCaptureWriter.begin(profileSnippet:)` 调用链是不是传 nil。如果没传，wire up `state.profiles.currentSnippet` 之类。

约 10 行（要看上下文）。

### Qwen 0.6B prompt echo 调研

dogfood 数据：拦掉的 20 段（占总 50 段 **40%**）**全部**是 dictionary 列表 `'热词：Python、Qwen3-ASR、VAD、E2E、Rust。'` 的字面回显。HallucinationFilter 抓得准（误杀率 0%），但 echo 频率本身离谱。

观察：
- 问题集中在**短音频段**（多数 0.3-1.6s）
- 100% 落在 Qwen 0.6B（dogfood 没用 1.7B 数据可比）
- 现有 user 100% 走 0.6B（自己挑的）

候选方案（按可能性 / 成本排）：

| 方案 | 描述 | 成本 | 风险 |
|---|---|---|---|
| (a) 短音频跳 dictionary prompt | < 2s 段不注入 dictionary 上下文 | 小 | dictionary 命中率下降 |
| (b) dictionary 改 logits bias | 不进 prompt 就不会被 echo | 大（要碰 ASR generation 层）| 兼容性 |
| (c) 默认升 1.7B | 0.6B 仅在 streaming 用 | 极小 | 加载时间 + 显存翻倍 |
| (d) HallucinationFilter 加 dictionary substring match | 现在已经做了——只是 baseline 高 | 0 | 已生效 |

v0.5.3 决定方案前要：
1. 跑 1.7B 一段时间看 echo rate 是否同样高
2. 测 (a) 方案 dictionary 命中率影响

如果 1.7B echo 显著低 → (c) 最简单。如果都高 → (a) 或 (b)。

## 风险与未知

- **Hands-free + Live mode 协议冲突**：两个独立 toggle 都开会怎样？scope 时要明确（建议 Hands-free 强制走 live）
- **tap timing threshold 在不同键盘 ergonomic 上差异**：用户的 keyboard 有人重按，threshold 太短误判 hold；太长 hands-free 入口不灵敏。需要 dogfood 校准
- **Qwen 1.7B 数据缺**：决定 (c) 之前必须要有 1.7B 实测数据，不然只是猜
- **partial meta wire-up 跨多 task race**：begin + 立即 enqueue 写 partial meta，要确保 segment/inject append 排在它后面（serial queue 保证应该没问题，单测验一下）
