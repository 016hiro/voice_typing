## #B1 — 修 inline refine telemetry bug

**问题**：v0.7.0 引入 `LocalLiveSegmentSession`（`LocalMLXRefiner.swift:252`）走 inline per-segment refine 路径后，每段 refine 跑了 `chatSession.streamResponse(...)`、`totalInferMs` 也算了，**但没调用 `DebugCaptureWriter.appendRefine`**。

`AppDelegate.swift:755` 的 `if !liveResult.refinedInline && willRefine` gate 直接跳过 batch refine block —— refines.jsonl 在 live + local 这条路径上一行不写。

**影响**：
- cloud↔local refine 质量 A/B 没数据（`refine_quality.py` 等了 5 个版本）
- 没法看 refine 改了 ASR 原文什么（input/output 配对丢失）
- 没法测 pin 前后 refine 延迟变化（→ #B5 阻塞）

**修法**：`LocalLiveSegmentSession` 接受一个 `appendRefine: (RefineRecord) -> Void` 闭包（在 `LocalMLXRefiner.makeLiveSession(...)` 被构造时由 AppDelegate 注入）。`runSegment` 跑完一段后构造 `RefineRecord` 调闭包。注入侧从 line 716 附近的 captureWriter 拿。

## #B3 — VAD silenceDuration 微调

**当前**：`StreamingTuning.production` = `minSpeech 0.3s, minSilence 0.7s, maxSeg 25s`（v0.5.0 shipping default，`QwenASRRecognizer.swift:411`）。

**待调**：minSilenceDuration 0.7s 在 live mode + clean up 下偶尔切得太碎（短停顿被切成两段，inline refine 各自跑一次，多花 ~1s）。

## #B5 — 决定是否 pin refiner（数据驱动决策点）

> 依赖 #B4 数据。

**判据**：
- 过夜 first segment > 10s → pin（值得吃 2.5GB wired）
- 过夜 first segment < 5s → 不 pin（边际收益太小）
- 5-10s 之间 → 看 refine 频率是否高 / 用户是否真的有"一早起来立刻 refine"心智

**如果决定 pin**：
- 复用 #B7 模式，`LocalMLXRefiner.ensureLoaded()` 拿到 container 后挂 `WiredMemoryTicket(.active)`，size 估 2.5GB
- 跟 ASR ticket 加起来 process-wide ~4GB wired —— 16GB Mac dogfood 跑一轮看 OS 是否压力反弹（free RAM、swap、其他 app 卡顿）
- ADR-0003：refiner 与 ASR 的 pin 策略对齐
