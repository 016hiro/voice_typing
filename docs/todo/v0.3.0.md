# v0.3.0 — 智能后处理 + 自定义词典 + 延迟优化

> 本版本目标：把 LLM refiner 从"保守修正 ASR 错误"升级到"四档可切换强度"，加入用户自定义词典（ASR bias + LLM glossary 双层注入 + LRU 排序 + token budget 动态截断），并提供"Raw-first 注入"作为延迟优化选项。
>
> **状态**：代码完成，手动端到端验证进行中。对标目标：[Typeless](https://www.typeless.com/)（智能清洗 + 自定义词典两大核心能力）。

## 设计决策（已确认）

- **RefineMode 四档**：`off` / `conservative`（v0.2 等价，默认）/ `light`（+filler+结巴）/ `aggressive`（+自我纠正识别+列表格式化+语义润色）。默认保持 `conservative` 不打扰老用户。
- **Refiner backend**：本版本只做云端（OpenAI 兼容），本地 MLX refiner 推到 v0.5.0。
- **字典结构**：`{id, term, hints[], note?, createdAt, lastMatchedAt?}`。刻意不做 language tag / category / caseSensitive / 动态 relevance 过滤。
- **字典存储**：`~/Library/Application Support/VoiceTyping/dictionary.json`（debounced 5s flush）。不用 UserDefaults 是为了导入导出即存储本身、iCloud 零成本迁移。
- **字典容量**：软上限 500 条防爆，无硬条数 cap。真正的限制是注入时的 token budget。
- **条目形态的语义分桶**（注入时分情况处理）：
  - 只 `term`（无 hints）→ **纯热词**：进 ASR context + LLM glossary 的 `Preserve` 段（"don't paraphrase"）
  - `term + hints` → **重写规则**：进 ASR context（只用 term 锚定）+ LLM glossary 的 `Rewrite` 段（"`hint / hint → term`"）
  - 只 hints / 空 → 非法，三层挡住（`DictionaryEntry.hasContent` + `upsert` guard + UI Add 按钮 disabled）
- **双层注入**（必做）：
  - **ASR bias**：
    - Qwen context（zh-\*）`热词：X、Y。`，其他语种 `X, Y`（对齐 Alibaba CLI 例子 `-c "Qwen-ASR, DashScope, FFmpeg"`）
    - Whisper prompt：空格分隔 term 列表，通过 `WhisperTokenizer.encode` 转成 `DecodingOptions.promptTokens`
  - **LLM glossary**：接在 RefineMode system prompt 后，拆 Preserve + Rewrite 两段指令
- **注入算法**：LRU 排序（`max(lastMatchedAt, createdAt)` desc）+ 贪心填到 backend budget × 90% 后截断。**不淘汰数据**，只决定本次注入谁。
- **匹配检测**：ASR 输出 + LLM 输出两处扫，英文 `\b{term}\b`，中文 substring。命中更新 `lastMatchedAt`。
- **Raw-first 模式**（全局开关，默认关）：ASR done 立刻 inject raw，refiner 后台跑完判断 focus + 用户未继续打字 → simulate Cmd+Z + paste refined。
- **长度约束**：aggressive system prompt 加硬约束 `output between 0.9× and 1.5× input chars` 减 LLM 生成 token。
- **埋点**：`LatencyTracker` 三段 ms（ASR / LLM / inject）→ os_log。

## 风险与未知

- **Aggressive prompt 实际质量未知**：需要 5-7 天 prompt 工程迭代 + 50 条中英 ground-truth 对比集。预算留出来了，最终质量要实测。
- **Whisper prompt 实测效果弱于 Qwen**：WhisperKit 的 `DecodingOptions.promptTokens` 对 ASR 的影响未实测（Qwen context 已从源码确认是 system prompt 槽）。预期生效但可能偏弱。
- **Qwen context 的作用边界**：实测确认 Qwen context 是**声学锚定**而非**发音替换**——"配森→Python"这种跨语种重写是 LLM refiner 的活。已通过 LLM glossary 的 Preserve/Rewrite 分桶落实。
- **中文 substring 匹配的误判率**：可能误触发（"超配森" 算 "配森"命中），长期看是否歪 LRU 排序。v0.3 发布后看 stats 再定是否 v0.3.1 优化。
- **Raw-first 的 focus 检测在 Electron / Web 内嵌输入框的准确性**：Slack / Notion / VSCode 各跑 5 轮验证，实现阶段第一周完成。
- **长度约束 prompt 是否真能省 token**：aggressive 理论上生成变长，约束能抵消多少未实测。
