## 设计决策

### Transcription timing Picker（单选三档）

**从双 Toggle 改为 Picker** —— /think 回合收到用户反馈"两个 streaming 会混淆"后拍板。双 Toggle 方案的根源问题：
- `LiveTranscriber` runtime 本来就 bypass post-record 路径（见 `AppDelegate.startRecording`），双 Toggle 假装独立
- 用户第一次看到两个 "Streaming" 相关开关要先搞清 "Post-record vs Live" 的技术差异才能选
- Off/Postrecord/Live 本质是一维选择（**什么时候出字**），硬拆成两个开关反而难用

**当前 → v0.5.2 映射**：

| 当前 | v0.5.2 |
|---|---|
| "Streaming (experimental)" SectionCard + Toggle "VAD-segmented transcription" | "Transcription timing" SectionCard + radio Picker（三选一）|
| `state.streamingEnabled` 绑定 Toggle | `state.transcriptionTiming` computed property 绑定 Picker |
| `state.liveStreamingEnabled` 隐藏（只能 `defaults write`）| 暴露为 Picker 的 `.live` 选项 |

**三个 option 文案**（都在 Picker 里，单 Text line，disabled 时整个 Picker 灰掉）：

- `After recording — one shot (default)`
- `After recording — segmented (VAD)`
- `While speaking — live (experimental)`

**Picker 下方 footer**（动态）：
- 非 Qwen backend 时："Only Qwen backends support segmented or live transcription. Switch to a Qwen model to enable."
- `oneshot` 选中：解释"Fn↑ 后一次性出字，兼容所有 backend"
- `postrecord` 选中：解释"Fn↑ 后 VAD 分段，逐段出字，Silero VAD 已 bundle"
- `live` 选中：解释"说话中出字，跳过 refine，Qwen only experimental"

**UserDefaults 映射**（无新 key，零迁移）：
```
getter:  live=true → .live
         streaming=true → .postrecord
         else → .oneshot

setter:  .oneshot    → streaming=false, live=false
         .postrecord → streaming=true,  live=false
         .live       → streaming=false, live=true
```

setter 设另一个为 false 而非保持——保证从 live 切回 postrecord 状态干净，不遗留歪值。

**向后兼容**：
- `defaults write com.voicetyping.app liveStreamingEnabled -bool true` 仍然有效（UserDefaults 本就是 backing store）
- `streamingEnabled` key 不变（老用户 `defaults write streamingEnabled=true` 升级后 Picker 自动显示 "postrecord"）
- 如果同时为 true（legacy dual-state）：getter 返回 `.live`（运行时优先级匹配）

### 分析脚本

按 `todo/v0.5.1.md` 决策 #7 "v0.5.1 不做工具" 的尾巴："工具留给 v0.5.2 或之后看实际需要再做"。现在拍板做，但范围收窄。

- **位置**：`Scripts/analysis/`（不污染 `Sources/`，不进 .app bundle，不参与 Swift build）
- **语言**：Python 3 stdlib only —— 不引 pandas / matplotlib / numpy。理由：
  - 用户运行不需要额外 `pip install`
  - 输出数据量小（典型一周 capture < 5 GB / < 1000 sessions），不需要 DataFrame
  - 第三方依赖会拖累"开箱即用"承诺
- **输入约定**：positional arg 指向 capture root 或单 session 目录。脚本自适应（如果是 root 就遍历所有 session；如果是单 session 就只算那一个）
- **输出约定**：文本 summary 到 stdout。脚本 `--csv` flag 可选输出 CSV 给后续 spreadsheet 处理（v0.5.2 暂不实现 CSV 选项，留接口）

**首批 5 个脚本**（在反思回合从 4 → 5：补 `segment_latency.py` —— `live_drain` 只看最后段，但用户中段感受是每段 ASR 延迟 + RTF）：

| 脚本 | 输入 | 输出 | 覆盖 v0.5.1.md #6 表 |
|---|---|---|---|
| `summary.py` | capture root | sessions / audio + p50-p99 / segments / injections / on-disk + per-backend / language / profile-hit / skipped split | overview，新增 |
| `hallucination_review.py` | capture root | 所有 `filter: hallucinationFiltered` 段列出（rawText + 位置 + session），per-backend filter rate，可选 `--sample N --seed S` 随机抽样 | 第 3 行 "HallucinationFilter ±" |
| `live_drain.py` | capture root | live mode session 的"最后段 inject 完成时间 - Fn↑时间"分布，percentile + ASCII histogram | 第 5 行 "Live mode drain 时间分布" |
| `focus_drop.py` | capture root | `status: focusChanged` + `skipped` 双重 drop，按 target bundleID 排序，`--min-sessions N` 过滤长尾 | 第 6 行 "Focus-change drop 频率" |
| `segment_latency.py` | capture root | per-segment `transcribeMs` 分布 + per-backend RTF (transcribeMs/audio_ms) + cold/warm 对比 (first vs rest segment) | 新增（反思回合补 —— mid-recording 体感由这个决定） |

**不进首批**（v0.5.1.md #6 表里的另两类，需要 Swift 跑模型）：

- **VAD config replay**（audio + 不同 `StreamingTuning` → 看分段差异）：需要 Swift LiveTranscriber 实例 + Silero VAD 模型加载。要做就单独成 Swift CLI binary，不该用 Python 弄
- **跨 backend ASR 对比**（同 audio → 三 backend → diff transcript）：同上

这两类挪到 v0.5.3 或更晚，需要决策"独立 CLI binary 还是扩展 audio CLI"。

### dogfood polish

具体内容由 dogfood 数据定，**TBD by data**。候选清单（按可能性排）：

- **HallucinationFilter 调参**：如果 `hallucination_review.py` 显示 filter 拦掉的段里 ≥ 20% 是真实有效语音 → 阈值/规则需要松
- **Live drain 卡顿修**：如果 `live_drain.py` 显示 30s 录音的 drain time p95 > 500ms → 调查瓶颈（可能是 maxTokens=448 在长段被打满，或 Metal 队列堆）
- **Capsule overlay 阈值调整**：v0.5.1 dogfood 已经发现 540s 警告窗口实测难触发。如果用户感受是"想测又测不到"，加 Settings 一个 dev-only override
- **IME 切换体感**：如果 CJK 用户报 IME 闪烁明显，研究 segment-injection 时的 IME save/restore 是否每段都做（应该可以缓存）
- **Focus drop 体感**：如果 `focus_drop.py` 显示 > 5% 段被丢，调查 frontmost detection 是否有 race
- **dl_init 修法 ship 验证**：v0.5.1 ship 后用户重测 cached prepare 时长，如果还是 > 2s 说明 `ModelStore.isComplete` 检查比 upstream `weightsExist` 严，需要对齐

## 本版本不做

- ❌ **VAD auto-stop (hands-free 模式)** —— 推到 v0.5.3 单独成版。需要专门的 UX 设计：tap-vs-hold 检测、模式状态 UI 指示、abort 手势、和 Fn-hold 模式共存的心智模型

- ❌ **Refine I/O capture** —— 同上，绑在 refine 专版
- ❌ **VAD config replay** + **跨 backend ASR 对比** —— 需要 Swift 跑模型，独立 Swift CLI 设计
- ❌ **Debug capture Swift CLI binary (`vt-analyze`)** —— Python 脚本够用；如果 v0.5.2 dogfood 期间发现需要跨 session SQLite query 再立项
- ❌ **WhisperKit live 路径** —— Whisper 没有 streaming engine，沿用 v0.5.0 限制
- ❌ **快捷键可配置 / 历史转录** —— v0.6+ backlog

## 风险与未知

- **dogfood polish 内容空**：本 doc 起草时 v0.5.1 ship 还没满 24 小时，dogfood 数据几乎为零。"polish" 段需要真实数据填实，否则 v0.5.2 实际会变成"只做 UI 公开 + 脚本"。这是预期状态——polish 是 placeholder，不是漏写
- **Picker 公开会拉高 dogfood pool**：之前只有显式 `defaults write` 的用户在用 live mode；Picker 公开后默认入口存在，dogfood 数据量会增加，但也会引入"以前没碰过 live 的用户首次接触"的反馈噪声（vs 老用户的 informed feedback）
- **option 文案改 vs 老用户**：现有 `streamingEnabled = true` 的老用户在 v0.5.2 升级后看到 Picker 显示为 `After recording — segmented (VAD)` 而非 "VAD-segmented transcription"。功能未变，只是 UI 表达。如果担心混淆，devlog v0.5.2.md 显式说明
- **双 true legacy state**：老用户如果手动同时 `defaults write` 过两个 key 为 true，升级后 Picker 显示 `.live`（getter 优先级）。一次切回其他档会把 streaming 也清 false——这是故意的（见 setter 决策），但需要在 devlog 里点名避免"我的 segmented 设置被吞了"的困惑
- **Python stdlib only 的限制**：`live_drain.py` 算 percentile / histogram 需要手写小算法（statistics.quantiles 在 3.8+ 可用，足够）。如果以后想画图必须升级到 matplotlib，那时就破"零依赖"承诺
- **分析脚本面对的实际场景未知**：首批 4 个里可能有不用的也可能缺的。v0.5.2 末尾 retrospective 决定第二批方向（是不是该把 VAD replay 提上来？是不是该加 `inject_latency.py`？）
- **`liveStreamingEnabled` 老用户**：v0.5.2 ship 后老用户看到 toggle 已经是 ON 状态（因为 UserDefaults 有值），不需要二次操作。但如果他们之前 `defaults write false` 过又忘了，会奇怪 "Live transcription" 默认是关的。文案不强调默认值，只说功能行为，避免误导

## v0.5.3 advancement criteria

dogfood 数据攒到用户认可"够"才启动 v0.5.3。无硬性 5 天 / 30 Fn 指标（v0.5.1.md 里写的那套指标继续作为参考，不作为 gate）。建议在启动 v0.5.3 之前用本版本的分析脚本至少跑一遍：

- [ ] `summary.py` 跑一遍知道用 live 多深 + 哪个 backend 主流 + audio length 分布
- [ ] `hallucination_review.py --sample 20` 人工判断里面真实语音 vs 训练尾巴比例（≥ 20% 真语音 → 松阈值）
- [ ] `live_drain.py` 跑一遍看 p95 drain time 分布是否在可接受范围（< 500ms）
- [ ] `focus_drop.py` 跑一遍确认 focus drop 比例 < 5%（单 app < 10%）
- [ ] `segment_latency.py` 跑一遍确认 Qwen RTF p95 < 1.0（live 不堆积）+ first/rest delta < 50%（warmup 已消除）

如果五项任一明显异常，先发 v0.5.2 polish patch 修对应问题，再 dogfood 一阵，再启动 v0.5.3。

## 决策矩阵：脚本输出 → 行动

详细的"看到什么数字 → 做什么"映射见 [`Scripts/analysis/README.md`](../../Scripts/analysis/README.md) + 反思回合产物（每脚本对应一张表）。核心 gate 数字摘要：

- **Drain p95 > 500ms** → 查 maxTokens / Metal queue / dl_init 残留
- **Filter rate > 5% 且抽样真语音 ≥ 20%** → 松 HallucinationFilter，可能要 per-backend 阈值
- **Total focus drop > 5% 或单 app > 10%** → frontmost detection race，必修
- **Qwen RTF p95 > 1.0** → live mode broken，要么换 backend 要么调 force-split
- **First/rest delta > 50%** → warmup 仍在，prepare() 还需优化
- **Live 占比 < 20%** → Picker UI 公开后用户没 adopt，文案/入口有问题
