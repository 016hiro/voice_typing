# v0.6.3 — 本地 MLX refiner (Qwen3.5-4B)

> **状态**：scope locked 2026-04-29，待开工
>
> **主题**：让 refine 摆脱云依赖。引入本地 MLX LLM refiner（选型 `mlx-community/Qwen3.5-4B-MLX-4bit`），与现有 cloud LLMRefiner 并存；按 Mac RAM tier 决定是否暴露 / 默认行为。同版顺手收两笔 refine 历史欠债：refine I/O capture (#R9) + Live×Refine Cmd+Z (#R10)。
>
> **决策依据**：[`../perf/baseline.md`](../perf/baseline.md) — Qwen ASR 1.7B (~2.5 GB RSS) + Qwen3.5-4B refiner (~2.6 GB RSS) = ~5.1 GB 总；refine 质量 ★★★★（zh-EN 混合）；推理 ~0.9s @ 30-50 tok/s；ANE 不争用（refine 在 ASR 完成后顺序跑）。

## 目标

- 24 GB+ 用户可在 Settings 里开本地 refiner，零云依赖、零 API key
- 16 GB 用户能 opt-in（看到警告"may swap on heavy multitasking"）
- 8 GB 用户不暴露入口（避免误开 → 严重 swap 体验崩盘）
- 现有 cloud refiner 路径不动（既有用户 Settings、API key、行为零变化）
- 顺带：refine I/O capture（v0.5.1 punted）+ Live×Refine Cmd+Z chain 跨段问题

## 不做

- ❌ Bundle 模型进 DMG —— 2.6 GB 太大；走 ModelStore 下载（同 Qwen ASR 路径）
- ❌ Gemma 4 E4B —— 当前 mlx-vlm 0.4.4 不支持 Matformer 架构（audit 已验，详见 baseline.md）
- ❌ ANE refiner（Apple Foundation Model 等）—— 选型已定 MLX，本版不做对比
- ❌ Refine 流式输出（token-by-token 喂胶囊）—— 0.9s 延迟可接受，复杂度不值
- ❌ 让 16 GB 用户默认开 —— audit 数据是 24 GB 外推，先 opt-in 收数据再考虑改默认
- ❌ Settings "RAM tier 自动检测覆盖"开关 —— sysctl 读到的就是真值，不给用户骗自己的入口
- ❌ Whisper Loading UI (#P1) —— hold（CoreML 编译慢是 backend 一次性问题，本版优先做 refiner 主线）

## 设计决策

### 1. Refiner 协议抽象

当前 `LLMRefiner` 是 final class、cloud-only。改造为协议 + 两实现：

```swift
protocol LLMRefining: Sendable {
    func refine(_ text: String,
                language: Language,
                mode: RefineMode,
                glossary: String?,
                profileSnippet: String?) async -> String
    func test() async -> LLMTestResult
    var requiresCredentials: Bool { get }   // cloud=true, local=false
}

final class CloudLLMRefiner: LLMRefining { ... }       // 现 LLMRefiner 改名 + 协议
final class LocalMLXRefiner: LLMRefining { ... }       // 新增
```

`refine` 签名里 **去掉 `config: LLMConfig`** —— config 注入到实现的 init，对调用方透明。AppDelegate 持的就是 `any LLMRefining`，runtime 按 Settings 选 cloud 或 local。

### 2. LocalMLXRefiner 实现

- **底层**：`mlx-swift-examples` 仓库的 LLM 子模块（`LLMEvalSwift`），新增 SwiftPM 依赖。`speech-swift` 已经走 mlx-swift 路径，整套依赖图相熟。
- **模型**：`mlx-community/Qwen3.5-4B-MLX-4bit`，~2.6 GB（参见 baseline.md S 段）
- **System prompt**：复用现有 `RefineMode.systemPrompt`（不需要为本地写新 prompt）
- **Chat template**：必须传 `enable_thinking=False` 关掉 Qwen3 系列默认 chain-of-thought（audit 验过：不关会先吐 `<think>...</think>` 块吃光 max_tokens）。mlx-swift 走 `MLXLMCommon.UserInput` + `tokenizer.applyChatTemplate(messages, tools:, additionalContext: ["enable_thinking": false])`，需要确认上游支持该 key
- **Generation params**：`temperature=0`、`max_tokens=512`（refine 输入通常 < 200 tok，输出长度等同）
- **Output cleanup**：复用 `LLMRefiner.stripQuotesAndCode` 静态方法（提到 enum/utility 里）

### 3. Tier 检测 + Settings 暴露策略

启动时一次 sysctl：

```swift
enum RAMTier {
    case low      // < 16 GB  →  本地 refiner 入口隐藏
    case mid      // 16-23 GB →  入口可见 + opt-in 带警告
    case high     // ≥ 24 GB  →  入口可见 + opt-in 不警告（默认仍 OFF）
    static let current: RAMTier = ...   // sysctl hw.memsize 一次决定
}
```

- 边界用 16 / 24 整 GB（hw.memsize 返回值除以 1024^3）
- 不把 tier 写进 UserDefaults —— 每次启动读 sysctl，避免用户换机后旧值留尸
- Settings → LLM tab 加 "Local refiner (Qwen3.5-4B)" 子段：
  - `.low` tier → 整段隐藏 + 在 LLM tab 底部加一句 "Local refiner unavailable on this Mac (requires ≥ 16 GB RAM)" 提示
  - `.mid` tier → toggle + ⚠ "Will use ~2.6 GB RAM. May cause swapping on heavy multitasking."
  - `.high` tier → toggle，无警告
- toggle 状态键 `voice_typing.localRefinerEnabled`，默认 false

### 4. Lazy-load + warm-up（注：refiner 不做防压缩 keep-alive，与 ASR 不同）

冷加载 ~6.5s（baseline 数据），不能阻塞 Fn↑（用户感知延迟会很糟）。策略：

| 时机 | 行为 |
|---|---|
| 启动 / Settings 切换为 ON | **不**立即加载（避免空跑成本） |
| 第一次 `refine` 调用前 | 同步加载 + 1 个 dummy refine 预热（吃当次延迟） |
| 之后 | 权重常驻，refine 直接用 |
| **闲置 N 分钟后再 refine** | macOS compressor 可能已压缩 ~2.6 GB 权重 → 第一次 refine **5-30s 解压**。胶囊 UI 上明确"warming up..."，用户预期管理 |
| Settings 切换为 OFF | 释放权重（`weights = nil`） |
| Settings 切回 cloud refiner | 释放权重 |

**与 ASR 的关键区别**：refiner **不做** v0.6.4 风格的 keep-alive timer。决策依据（用户 2026-04-29）：refine 本来就有"等几秒"的心智模型，5-30s cold decompress 可接受；ASR 是 critical path 每次 Fn 都触发，必须 warm。

权重常驻 ~2.6 GB RSS，是已知开销 —— Settings UI 的警告文案要明确这点。

### 5. 模型下载 / 缓存

- 复用 `ModelStore` 的下载流程：`Application Support/VoiceTyping/models/local-refiner/Qwen3.5-4B-MLX-4bit/`
- 复用 `HFEndpointResolver`（v0.6.1 落地）—— 中国用户自动走 hf-mirror
- 第一次启用 toggle 时弹 confirm dialog（同 Qwen3-ASR onboarding 模式）：
  > "Download Qwen3.5-4B (~2.6 GB)? Used for local refining."
- ModelStore.isComplete 检测：mlx-swift LLM 模型典型文件 = `config.json` + `tokenizer*.json` + `model.safetensors.index.json` + N 个 `model-*-of-*.safetensors` shard。粒度到 shard 校验

### 6. 现有 cloud refiner 不动

- `LLMConfig` 里的 baseURL/apiKey/model 字段保持
- Settings → LLM tab "Cloud refiner" 子段（重命名 + 视觉分组）
- 默认 backend 仍是 cloud（既有用户零感知）
- toggle 互斥：local ON → cloud row 显灰 + "Disabled while local refiner is enabled"

### 7. Refine I/O capture (#R9，v0.5.1 punted)

Developer logging 段已在 v0.4.4 落地（capture audio + ASR）。本版加 refine 段：

- 开关：现有 `voice_typing.developerLoggingEnabled`（不再多加 toggle）
- 落盘：`~/Library/Application Support/VoiceTyping/captures/<session>/refine-<idx>.json`
  ```json
  {"input": "...", "output": "...", "mode": "natural",
   "backend": "local|cloud", "latency_ms": 920,
   "glossary": "...", "profile_snippet": "..."}
  ```
- 调用点：`AppDelegate` 两处 `await self.refiner.refine(...)` 后立即 dump
- 对 local refiner 尤其重要 —— 出 dogfood 才能比较 cloud vs local 质量差

### 8. Live × Refine Cmd+Z chain 跨段问题 (#R10)

**现象**（既存 bug）：Live mode 下，N 个 segment 各自 refine + replace-injection。用户按 Cmd+Z 只撤销最后一段，无法一次撤回整次录音的 refine。

**原因**：每段 refine 完独立调 `Injector.replace(raw → refined)`，N 个独立 undo group。

**修法**：
- `Injector` 加 `beginUndoCoalescing()` / `endUndoCoalescing()` —— live session 开始/结束包一层
- 实施手段：每段 replace 前压一个 zero-width hint（`AXTextChangedNotification` 不会显示），或更朴素 —— 段间 paste 时用同一个 NSUndoManager group token
- macOS AX/CGEvent 路径下没有 cross-app NSUndoManager，所以更可能是"靠 timer 紧贴的 paste 系统会 coalesce"。验法：实测 TextEdit / Notes / Cursor 三处 baseline，确认下界
- **若发现 system 行为不可控** → 退化方案：live mode 下整段录音结束后再统一 refine + 一次性 replace（牺牲段级 progressive 反馈）。本版决策：先调研 1-2 天，搞不定就退化

## 数据流

```
[启动]
  applicationDidFinishLaunching
    ├─ RAMTier.current = sysctl hw.memsize / 1024^3 → .low/.mid/.high
    └─ 按 voice_typing.refinerBackend 设置创建：
         "cloud" → CloudLLMRefiner(config: LLMConfig.fromUserDefaults())
         "local" → LocalMLXRefiner(modelPath: ModelStore.path(for: .localRefiner))
                   ├─ 不立即加载权重
                   └─ 第一次 refine() 触发 sync load + warm-up dummy

[refine 调用]
  AppDelegate.refine(text, ...)
    └─ await self.refiner.refine(...)         // 不知道 cloud / local
        ├─ CloudLLMRefiner: HTTP POST → 返回
        └─ LocalMLXRefiner:
            ├─ if !loaded { sync load + dummy "ping" warmup }
            ├─ generate(prompt, max_tokens: 512, temperature: 0)
            └─ stripQuotesAndCode → return

[Settings toggle local ON, .high tier]
  └─ if !ModelStore.isComplete(.localRefiner) {
        NSAlert "Download Qwen3.5-4B (~2.6 GB)?" ["Download", "Cancel"]
        user OK → ModelStore.download(.localRefiner) [HFEndpointResolver 已 race]
     }
  └─ refinerBackend = "local"
  └─ 替换 self.refiner = LocalMLXRefiner(...)
```

## 风险

- **mlx-swift LLM 集成成本**：仓库已经有 mlx-swift（speech-swift 依赖），但 LLM 推理是新路径。`mlx-swift-examples` 不是 SwiftPM-friendly 的"库"形态，它是 example app 集合，需要 cherry-pick `MLXLMCommon` / `MLXLLM` 子 target 当依赖。spike (#R4) 没跑通就要重新评估方案（candidate：直接 Process exec mlx-lm Python 子进程 —— 但 ~750 MB venv 不可接受）
- **chat template `enable_thinking` key 在 mlx-swift 路径上能否传**：Python `tokenizer.apply_chat_template(enable_thinking=False)` 是 transformers 的特性。mlx-swift 走的 `Tokenizers` Swift 包是否透传该 kwarg 待验。spike (#R4) 必测项 —— 不行就需要在我们这边手动构造 prompt template、跳过 jinja
- **16 GB 真实表现**：baseline 是 24 GB 外推，没在真 16 GB 机器测过。极端情况：refiner 加载即触发 swap，UX 崩盘。Mid tier 的警告文案是兜底；dogfood 期 #R8 capture 数据帮判断
- **Live × Refine Cmd+Z 实测可能搞不定**：跨 app NSUndoManager 不可控，AX 路径下 Cmd+Z 行为由 target app 决定。退化方案是"段终再统一 refine"，会折损 live progressive 体验。最坏情况 #R9 这条 punt 到 v0.6.4
- **2.6 GB 模型下载 = 不小流量**：confirm dialog 已经做了，但中国用户即便走 hf-mirror 也是 ~10 分钟（10 MB/s 测算）。Onboarding 文案要明确预期时间
