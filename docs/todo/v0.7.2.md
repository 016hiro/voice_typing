## TODO

- [x] **#B6** Live-mode VAD hang 根因 + workaround (done 2026-05-03)
  - 触发：dogfood `be8ac70b` orphan capture——首段 145 ms transcribe 后第二段无声卡死，session 没 finalize
  - 根因：`relu` 算子在 `Qwen3AudioEncoder` 和 `SileroVADNetwork.forward` 间共享，MLX per-`CompiledFunction` `NSRecursiveLock` 跨 `eval()` → Metal `IOSurfaceSharedEvent waitUntilSignaledValue:` 持有；Silero MLX 冷路径 30+ s 占住整个推理链
  - 修法：(1) `TranscribeWatchdog` 5 s 阈值 + 自动 `sample(1)` 抓 hung-thread 栈；(2) live-mode VAD 切 `.coreml` 走 ANE+CPU 跟 MLX 分开；(3) `MLXWorkGate` 串行化 user MLX work（这条 #B8 后撕掉）；(4) 落盘 pumpStallMaxMs / chunkLagMaxMs / vadProcessSumMs / lockWaitMs / ingestCount。Spike 文档 `docs/spike/v0.7.1-vad-hang.md`
- [x] **#B7** Cold-compressor root-cause kill via `WiredMemoryTicket` (done 2026-05-05)
  - 触发：v0.7.1 #B3 修后 ≥5 s 率仍 5.3 %（v0.6.1 baseline 2.2 %）。Dogfood `2026-05-04_15-24-38` 显示 keep-alive tick 自身就跑 30-45 s——silent buffer 没 EOS 信号，decoder 跑满 448-token budget。13.5 h overnight 后 RSS 5 GB → 62 MB
  - 根因：macOS unified-memory compressor 把 mmap'd MLX 权重 page 评估为 "cold" 压成 60-80 MB compressed segment。v0.6.4/v0.7.1 keep-alive 路线本身就在 cold path 上，治标不治本。MLX 社区 canonical fix 是 `mlx_set_wired_limit` (Metal residency_set)
  - 修法：`Sources/VoiceTyping/ASR/QwenASRRecognizer.swift` 在 `model = loaded` 后挂 1.5 GB `WiredMemoryTicket(.active)`，**永不 end**（除 unload 时）。`.active` 不是 `.reservation`——后者 idle 时不 keep limit elevated
  - 验证（56 sessions / 84 segments / 2 天）：≥5 s 率 5.3 % → 0.00 %；max 30-51 s → 3.56 s；TranscribeWatchdog 触发 18/50min → 0；11.5 h overnight 首段 37-51 s → 1.08 s
  - ADR：[`decisions/0002-pin-mlx-weights-not-keep-alive.md`](../decisions/0002-pin-mlx-weights-not-keep-alive.md)
