# v0.7.1 #B6 follow-up — Live-mode VAD hang spike

**Status**: ✅ root cause identified (2026-05-03) via auto-captured `sample(1)` stacks. Mitigation in progress (live VAD → CoreML, MLXWorkGate, drop VADKeepAlive). See `## Root cause (confirmed)` near the bottom; original metric-only analysis preserved above for record.

**Build under test (initial signal)**: `0.7.1@0b81381d803b`.
**Data window**: 14 live sessions captured 2026-05-02 17:43 .. 18:53 UTC, of which 3 are sick (>3 s pump stall) and 1 borderline.

## Symptom (what the user sees)

User presses Fn, speaks for a few seconds, releases Fn. Capsule stays in `transcribing` state for **30+ seconds** before any text injects. After the injection finally lands, the next session works normally.

In the live pump, the entire utterance comes out as **one segment with `flushTriggered=true`** — i.e. no segment was emitted in real time during the speech; everything was bottled up until `sampleStream.finish()` (Fn↑) forced an EOF flush.

Reproduces under terminal apps (Warp, Ghostty); not yet observed in editor / chat apps in this dataset, though sample size is small (14 sessions).

## Key path

Hang is **inside `StreamingVADProcessor.process(samples:)`**, specifically the inner `model.processChunk(chunk)` call:

- `LiveTranscriber.runPump` (`Sources/VoiceTyping/ASR/LiveTranscriber.swift:272`) — `for await (ingestedAt, chunk) in sampleStream`
- `processor.process(samples: chunk)` (`LiveTranscriber.swift:296`) — wraps a Date around it (this is where `vadProcessSumMs` is measured)
- → `StreamingVADProcessor.process` (`speech-swift/Sources/SpeechVAD/StreamingVADProcessor.swift:83`) — pure-Swift state machine
- → `model.processChunk(chunk)` (`StreamingVADProcessor.swift:91`) — **the suspected hang site**
- → `SileroVADModel` runs through MLX (Metal/GPU). `SileroVAD.swift` confirms `.mlx` is the default backend; `.coreml` is an alternative ("frees GPU").

Lock contention, model.transcribe, audio tap delivery, and pump task scheduling are **all ruled out** by the math below.

## Evidence

### v0.7.1 #B6 instrumentation overview

Captured per chunk inside `runPump`:

| Metric | Definition | What it would say if it were the bottleneck |
|---|---|---|
| `chunkLagMs` | `drainAt - ingestedAt` (queue residency) | Audio tap was producing but pump wasn't draining |
| `pumpStallMs` | `drainAt(N+1) - drainAt(N)` (gap between drains) | Anything making a single drain iteration slow — includes time inside `process` |
| `vadProcessMs` | wall around `processor.process(samples:)` only | The VAD layer itself is slow / blocked |
| `lockWaitMs` | per segment: time to acquire `transcribeLock` | Concurrent batch / keep-alive transcribe contending |
| `transcribeMs` | per segment: time inside `model.transcribe` | The Qwen ASR call itself is slow |

### The 3 sick sessions

|---|---|---|---|---|---|---|---|---|
| `df46656b` | 40.1 s | 5.5 s | 55 | 39307 ms | **39402 ms** | **39448 ms** | Warp | 505 s |

### The math that pins the hang to `processChunk`

For every sick session: **`pumpStallMax ≈ vadProcessSum ± 100 ms`**.

That equality only closes if **one single call to `model.processChunk` accounted for ~all of `vadProcessSum`** — i.e. one chunk's VAD inference took the entire stall window:

- `pumpStallMax` is `drainAt(N+1) - drainAt(N)` — the longest single gap between consecutive drains. That gap includes everything iteration N does (buffer append, `processor.process`, event handling).
- `vadProcessSum` is summed across all drains over the session.
- If one drain's `processor.process` took 39 s and the rest were sub-ms, then both numbers land at ~39 s. Any other distribution (many slow drains; or a wholly-idle pump) breaks the equality.

Ruled out by the same data:
- **Lock contention**: `lockWaitMs = 0` everywhere.
- **`model.transcribe`**: `transcribeMs` is sub-second on all sick sessions.
- **Audio tap pause**: would imply `chunkLag ≈ 0` (chunks delivered after resume have fresh `ingestedAt`). We see `chunkLagMax ≈ pumpStallMax` instead, meaning chunks were queued during the entire stall, the pump just wasn't draining them — because the pump was stuck inside `processChunk` on the prior iteration.
- **Pump scheduling starvation** (`Task.detached` not getting CPU): would imply `vadProcessSum = 0` (no `process` calls completed). We see `vadProcessSum ≈ pumpStallMax`, so `process` *was* called and blocked there.

### Idle-gap correlation (partial only)

```

✓  idle=    9s  pumpStall=  1893 ms     ← borderline; vadSum=6367 ms on a 35.6 s session
🔴 idle=   28s  pumpStall=  7761 ms     ← 28 s gap, sick anyway
✓  idle=    5s  pumpStall=   334 ms
✓  idle=    2s  pumpStall=   337 ms

```

- All 3 worst-case sick sessions follow ≥9 min idle. **Not the same as: every long idle is sick** — the 27 min idle session was healthy.
- `vadProcessSum` also creeps up under sustained use even when not "sick" (1666 ms / 6367 ms / 6511 ms on the long-audio sessions vs. <1 s on short healthy ones), suggesting some warm-state pathology too, not purely cold-start.
- 1 of 3 sick sessions had only 28 s of idle before it, but immediately followed a 35.6 s session with already-elevated VAD time (6367 ms) — the prior session may not have fully "cooled down" the issue.

So the idle correlation is real but not deterministic. A model that says "every cold session is sick AND only cold sessions are sick" is wrong.

## Root cause — current hypotheses (none confirmed)

### H1: macOS compressor on MLX weights

Same shape as the v0.6.4 `ASRKeepAlive` bug. macOS memory compressor pages out the model's MLX weights after idle; first inference triggers a sluggish decompression. Fits the cold-session correlation. Doesn't explain the 27 min healthy outlier or the 28 s sick session.

### H2: GPU command queue / Metal cold path

Metal command queue or shader pipeline goes cold under GPU idle (low-power state, JIT recompile, command buffer setup cost). Different mechanism from H1, similar symptom shape. Would not be helped by keeping weights resident — would need to keep GPU work flowing.

### H3: GPU power management state transition

macOS clocks GPU back down after idle; first MLX op stalls while GPU ramps. Hard to distinguish from H2 without sampling.

### H4: Something not GPU-related at all

`SileroVADModel.processChunk` is a pure-Swift Sendable wrapper around `forward(_ samples: MLXArray, h:, c:)`. Could in principle hang on MLXArray construction, mmap fault on weights file, lock inside MLX itself, etc. No evidence either way.

### Why none are confirmed

We have enough data to locate the **site** of the hang (one `processChunk` call) but no data about what that call is doing internally during the hang. All four hypotheses produce the same v0.7.1 #B6 metric signature.

## Disambiguation plan (ordered by ROI)

1. **Catch a stack on the next sick session.** When `transcribing` capsule persists >5 s, capture `sample <pid>` mid-hang. The frame on top of the MLX call disambiguates immediately:
   - `MTLCommandBuffer waitUntilCompleted` → H2/H3 (GPU side)
   - `mmap` / `madvise` / page-fault frames → H1 (compressor / mmap)
   - MLX internal lock → H4
   - Could be added as a self-`sample` script triggered from the `TranscribeWatchdog` once it fires, since we already have a 5 s passive timer there.

2. **Differential test: switch SileroVAD to `.coreml`.** speech-swift exposes the engine choice on `SileroVAD`; if hangs disappear, it's GPU-side (H2/H3). If hangs persist, GPU is exonerated, suspect H1/H4. Cheap to flip behind a build flag.

3. **(If 1 inconclusive) Add per-call MLX layer timing.** Bracket the components inside `SileroVADModel.forward` to localize whether the cost is in MLXArray construction, the LSTM forward, or the post-processing. Last-resort granularity.

## What to do in the meantime

A symptomatic mitigation — independent of which hypothesis is right — is **`VADKeepAlive`**: a 90 s timer that runs one 512-sample silent VAD chunk to keep the GPU pipeline / MLX weights / GPU power state warm. Mirrors the v0.6.4 `ASRKeepAlive` pattern (`Sources/VoiceTyping/ASR/ASRKeepAlive.swift`) which solved the analogous problem on `Qwen3-ASR` weights.

Caveats:
- Helps H1/H2/H3 to varying degrees; does **not** help H4 if the cause is e.g. an internal MLX lock.
- Must use a separate `SileroVADModel` instance, or wrap with `resetState()` before/after, to avoid polluting the live session's hidden-state hysteresis.
- Should be gated on `liveTranscriber == nil` (don't tick during a live session).

Open question: ship as a v0.7.2 follow-up immediately (symptom-driven), or block on disambiguation step #1 above so we don't mask the real bug under a workaround. Not yet filed in [`backlog.md`](../todo/backlog.md) — pending this decision.

## Root cause (confirmed, 2026-05-03)

The disambiguation step #1 above (auto-`sample(1)` from `TranscribeWatchdog`) was added in build `0.7.1@0b81381d803b-dirty` second iteration; on the next reproduction the watchdog wrote a stack to `~/Library/Application Support/VoiceTyping/hang-stacks/2026-05-03T15-02-15Z_live-vad-process.txt`. **Three threads converge on the same `NSRecursiveLock` inside MLX `CompiledFunction.innerCall`**:

| Thread | Stack tail | Lock state |
|---|---|---|
| Live VAD pump (`Task.detached`, .userInitiated) | `LiveTranscriber.runPump` → `StreamingVADProcessor.process` → `SileroVADModel.processChunk` → `processChunkMLX` → `SileroVADNetwork.forward` → `relu` → `CompiledFunction.innerCall` → `NSRecursiveLock.lock` → `__psynch_mutexwait` | waiting |
| ASR keep-alive (Task.detached, .utility) | `ASRKeepAlive.tick` → `transcribeSegmentSync` → `Qwen3ASRModel.transcribe` → `Qwen3AudioEncoder` → `relu` → `CompiledFunction.innerCall` → `NSRecursiveLock.lock` → `__psynch_mutexwait` | waiting |
| **VAD keep-alive** (Task.detached, .utility) | `VADKeepAlive.tick` → `SileroVADModel.processChunk` → `processChunkMLX` → `eval(_:)` → `mlx::core::eval` → `mlx::core::array::wait()` → `mlx::core::Event::wait()` → `IOSurfaceSharedEvent waitUntilSignaledValue:` → `iokit_user_client_trap` | **HOLDING** lock, blocked on GPU event signal |

### Mechanism

1. `mx.compile()` produces a process-global `CompiledFunction` cache. Each entry has its own `NSRecursiveLock` taken in `innerCall` and **held across `eval()`** — which means held across the synchronous wait for the GPU work to complete (`waitUntilSignaledValue:`).
2. `relu` is a standalone compiled function shared by every MLX caller in the process — `Qwen3AudioEncoder`, `SileroVADNetwork.forward`, the local refiner, anything else.
3. When ANY caller's `eval` is slow because the GPU event takes 30+ s to fire (cold path / power state / memory pressure — pick your H1/H2/H3, mechanism for *that* is still ambiguous, but it's now a secondary question), all OTHER callers of `relu` in the process serialize behind it.
4. In the captured hang: VADKeepAlive's tick is the **lock holder**. Live VAD pump and ASR keep-alive are the visible victims. The user sees "transcribing capsule stuck" because the live pump can't drain its first chunk.

### Why the v0.7.1 #B6 first-pass fix (`SharedVADBox` lock + isolated keep-alive instance) didn't work

- The `SharedVADBox` lock + keep-alive isolation assumed the contention surface was the `SileroVADModel` instance state.
- Reality: the contention surface is **the compiled function cache**, which is process-global. Two separate `SileroVADModel` instances both call into the same compiled `relu` and share the same `NSRecursiveLock`.
- Hence isolating the model instance does nothing; the second iteration of post-fix dogfood (34 sessions) showed the hang come back at `p95 = 38 s`, identical to pre-fix.

### Why VADKeepAlive is net-negative

- Without VADKeepAlive, the only background MLX caller was ASRKeepAlive; the lock-blocking window was bounded by Qwen transcribe's own GPU event wait (still bad on cold paths, but at least one source of contention).
- VADKeepAlive added a SECOND background MLX caller. After the first dogfood iteration we observed both: (a) sessions where live pump's `relu` blocked on VADKeepAlive's lock (the hang), and (b) sessions where the new `SharedVADBox` lock added a *third* failure mode (`vad.resetState()` blocked at the box level even before MLX, signature `chunkLag ≫ pumpStall`).

### What ALSO holds across the GPU event wait

The `relu` lock isn't unique. Any compiled MLX function exhibits the same shape — the `CompiledFunction.innerCall` path is taken for every `compile`d op. The contention is per-compiled-function, but `relu` happens to be the most-shared op (every transformer block, every conv stack, every Silero forward). Other compiled functions that are also shared across audio encoder / refiner would create the same hang under different combinations.

## Mitigation plan (in progress)

Short-term hot fix (v0.7.2 #B?, building now):

1. **Delete VADKeepAlive entirely** — its presence demonstrably introduces more hang sources than it prevents.
2. **Switch live VAD to CoreML backend** (`SileroVADEngine.coreml`, ANE+CPU) — Silero leaves MLX entirely, so VAD is no longer a participant in the `relu` lock contention.
3. **Add `MLXWorkGate`** — try-acquire gate around all MLX call sites:
   - User-visible (Qwen batch / streaming / live segment, refiner) wraps with `runUser(callsite:)` — increments user-inflight counter.
   - ASR keep-alive wraps with `tryRunKeepAlive(callsite:)` — refuses to enter MLX when user-inflight > 0. Cannot interrupt a tick already inside `eval()`, but stops *new* concurrent ticks during the user's recording / transcribe / refine pipeline.
4. **Continue auto-`sample(1)`** — the watchdog stays wired so any future hang lands a stack on disk for analysis.

Mid-term (the user already sketched, parked here for follow-up):

- Surface MLXWorkGate state (in-flight callsites, denied counts) in dogfood metadata so `pump_health.py` can tag every session with "did keep-alive get denied during this session?".
- Replace ASRKeepAlive's mechanism if cold path turns out to be primarily memory-compressor-driven — possibly mlock pages or accept cold first-press in exchange for never holding the global MLX lock from a background thread.
- Upstream issue to mlx-swift: the lock semantics around `eval()` deserve a discussion — the lock arguably should protect cache mutation only, not the GPU sync wait.

## Lessons

- "Metric closure" (the `pumpStall ≈ vadSum ≈ chunkLag` equality from the original analysis) correctly localized the hang to a single `processChunk` call but couldn't tell us *what within* `processChunk` was blocked. The answer turned out to require a stack, not better metrics. Prior conjectures about compressor / mmap / GPU power state were all symptoms-of-the-same-disease at most: the proximate cause was a global lock around a slow GPU sync.
- "Add more warming" was the wrong instinct. Warming inherently means "background MLX work runs"; combined with a process-global lock that's held across GPU sync, more warming → more contention sources → more hang opportunities.
