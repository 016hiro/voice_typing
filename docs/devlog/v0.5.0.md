# v0.5.0 — Live-mic 流式 + 25s force-split

> 主线版本。两件事:
> 1. **真 live-mic 流式**：ASR 在按住 Fn 的过程中就跑（按 VAD 分段），松开时只剩尾段 + drain。中长录音的"松开后等待"从 `ASR(整段)` 降到 `ASR(最后一段) + drain`。隐藏 toggle，UserDefaults 手动开。
> 2. **VAD force-split 阈值 10s → 25s**：v0.4.5 沿用的 10s 是 v0.4.2 留下的保守值；查 `AudioPreprocessing.swift` 才发现 Qwen3-ASR 硬上限是 1200s，实际约束是 `maxTokens=448`，25s 中文连讲 ~120 tokens，离 budget 远得很。一口气说 12-25s 的录音不再被切坏词。

UI 没动。`Resources/Info.plist` 0.4.5 → 0.5.0，build 10 → 11。

## 背景

v0.4.2 的 "streaming" 是 **post-record** —— 用户松开 Fn → AudioCapture 返回完整 buffer → walk 整个 buffer 跑 VAD → 逐段 Qwen → yield 累积文本。"流式"只是 UI 渐进；ASR 工作整体在 Fn 松开**之后**才开始。中长录音用户感知延迟 = `ASR(整段)`。

v0.4.5 把 VAD 调参 + HallucinationFilter 落地，质量已经稳。下一步攻延迟。

## 调研

### Qwen3-ASR 的硬上限不是 30s

`StreamingTuning.maxSegmentDuration` v0.4.2 拍了 10s，理由是 "保守"。这次先查上游：

```swift
// .build/checkouts/speech-swift/Sources/Qwen3ASR/AudioPreprocessing.swift:299-313
// Qwen3-ASR encoder handles arbitrary-length audio via windowed attention.
// The chunkLength=30 from preprocessor_config.json is inherited from the HuggingFace
// WhisperFeatureExtractor class but is not enforced by the official Qwen3-ASR pipeline.
// Cap at 1200 seconds (120000 frames) to match the official pipeline's upper bound
let maxFrames = 1200 * sampleRate / hopLength  // 120000 frames at 16kHz/160hop
```

**1200 秒 = 20 分钟。** Whisper preprocessor 里的 `chunkLength: 30` 是从 HuggingFace 配置继承的，**Qwen3-ASR pipeline 不强制执行**。

但 20 分钟是上游硬上限，不是实用上限。真正卡住 force-split 的是我们自己设的 `maxTokens=448`：

| 语种 | 30s 连讲 token 估算 | 距 448 budget |
|---|---|---|
| 中文 | 80-120 (150-200 字) | 4× 余量 |
| 英文 | 80-110 (60-80 词) | 4× 余量 |

**25s 是一个安全的上限**：30s preprocessor 标称值减 5s buffer，避开 maxTokens 边界，并且实测人类一口气说话很少超过 25s（自然停顿 → VAD 自己切）。

### Benchmark 验证 25s 严格更优

`make benchmark-vad` 加 `prod10s` preset 做对照（v0.4.5 shipping），跑 14 个 fixture：

| fixture | prod10s | production (25s) |
|---|---|---|
| Tianshan (32s 中文) | 100% / 4 partials | 99% / **2 partials** |
| fleurs_zh_xlong_18s (18s 连讲) | 100% / 2 partials | 100% / **1 partial** |
| librispeech_1919 | 99% | **100%** (回收 prod10s 的小回归) |
| 其余 12 个 fixture | 100% | 100% (无变化) |
| **平均** | 99% / 829ms / 1.4 | **99% / 813ms / 1.1** |

中长 fixture (10-25s 连续语音) 的 partial 数显著下降，避开了 v0.4.5 在词中间切的情况。短 fixture (< 10s) 行为不变（force-split 本来就没触发）。

### Live 化的关键挑战

读完代码，瓶颈是 4 个：

1. **AudioCapture 只在 stop() 时返回 buffer**，没法边录边消费 —— 需要加一路 `AsyncStream<[Float]>`
2. **`maxDuration = 60s` 硬限**对 live 是浪费 —— samples 边到边消费，不会堆积
3. **VAD 是按 chunk 顺序 stateful 的**，不能乱序丢；live 路径需要单 producer-consumer
4. **`runStreaming` 把 `transcribeLock` 持有整个 walk**（同步循环里） —— live 是 async 循环，OSAllocatedUnfairLock 不能跨 await，必须改成 per-segment 锁

## 变更

### 1. `AudioCapture` 多吐一路 samples + per-start maxDuration

```swift
// Sources/VoiceTyping/Audio/AudioCapture.swift
public struct StartOutputs {
    public let levels: AsyncStream<Float>      // 给胶囊波形（不变）
    public let samples: AsyncStream<[Float]>   // 16k mono float 块，给 LiveTranscriber
}

public func start(maxDuration: TimeInterval = 60.0) throws -> StartOutputs
```

- `levels` 流不变，胶囊照旧
- `samples` 流是新增 —— 每个 `[Float]` 块是音频 tap 经过 16kHz 转换后的样本（48kHz 源 → ~341 样本/块）
- `maxDuration` 改成 per-start 参数：batch 路径继续用 60s（防 stuck Fn 触发 OOM 单段 transcribe），live 路径用 600s（10 分钟）
- 没人订阅 samples 流时，`.unbounded` continuation 缓冲后由 `stop()` 一并 finish，无副作用

### 2. `StreamingTuning.production` 升 25s

```swift
// Sources/VoiceTyping/ASR/QwenASRRecognizer.swift
public static let production = StreamingTuning(
    minSpeechDuration: 0.3,
    minSilenceDuration: 0.7,
    paddingSeconds: 0,
    maxSegmentDuration: 25.0   // 10 → 25
)
```

旧值留作 `prod10s` benchmark preset 做回归对照。**这一步同时改善 batch 流式和 live 流式** —— 用户即使不开 live，已有的 `streamingEnabled` 也吃到 25s 红利。

### 3. `LiveTranscriber` 协调器

新文件 `Sources/VoiceTyping/ASR/LiveTranscriber.swift` (~140 行)。架构:

```
AudioCapture.samples  →  ingest(chunk:)
                          ↓ AsyncStream
                         pumpTask:
                           for await chunk in sampleStream {
                               liveBuffer.append(chunk)
                               events = vadProcessor.process(chunk)
                               for event in events {
                                   case .speechEnded(seg):
                                       transcribeSegment(start..end)  ← per-segment lock
                                       accumulated += text
                                       outputContinuation.yield(accumulated)
                               }
                               check force-split (25s)
                           }
                         finish() → flush() → tail segment → output.finish()
```

设计要点：

- **Per-segment lock**：`recognizer.transcribeSegmentSync` 内部 `transcribeLock.withLock` 一次调用一释，跨段间不持锁。这样和 batch transcribe 的并发是安全的（事实上 AppDelegate 的 pipelineTask 单 flight 也不会真并发，但锁设计要正确）
- **liveBuffer 全保留**：不做滑窗。10 分钟 × 16k × 4B = 38MB，Apple Silicon 不在乎；省了"段切片时绝对索引 vs 相对索引"的复杂度
- **HallucinationFilter 复用**：v0.4.5 的过滤器照样挂在 transcribeSegment 后面
- **VAD reset 双端**：进入 / 退出都 `vad.resetState()`，防上次 session 的 RNN 隐状态污染
- **空 buffer fallback**：如果 VAD 全程没 fire（音频太短/太弱），fallback 到把整个 liveBuffer 当一段转写（同 batch 路径行为）

### 4. `transcribeSegmentSync` 同步入口

```swift
// QwenASRRecognizer.swift
func transcribeSegmentSync(samples: [Float], language: String, context: String?) -> String {
    guard samples.count >= 400 else { return "" }
    return transcribeLock.withLock {
        guard let model = self.model else { return "" }
        return model.transcribe(audio: samples, sampleRate: 16000,
                                language: language, maxTokens: 448, context: context)
    }
}
```

`internal` 范围（同 module 给 LiveTranscriber 调）。**返回 `""` 而不是 throw** —— mid-stream 单段错误不应该把整个 live session 撕掉，相比之下 silently 丢段 + log 更稳。

### 5. AppDelegate 集成

新文件 `Sources/VoiceTyping/AppDelegate+Live.swift`，把 live wiring 跟 batch 路径分开避免 AppDelegate.swift 进一步膨胀。

新增字段（在 AppDelegate.swift 主类里，因为 extension 不能加存储属性）：

```swift
var activeLiveTranscriber: LiveTranscriber?
var liveIngestTask: Task<Void, Never>?
var liveSnapshot: LiveRunSnapshot?
var cachedVADBox: SharedVADBox?

struct LiveRunSnapshot {
    let backend: ASRBackend
    let language: Language
    let dictEntries: [DictionaryEntry]
    let asrContext: String?
    // ...
}
```

**关键时机**：
- **VAD 预热**：`activateBackend` 时 fire-and-forget `vadActor.get()` → 存到 `cachedVADBox`。Qwen 准备好的时候 VAD 也好了，`startLiveTranscriberIfEnabled` 同步取，无 await race
- **Snapshot 时机**：live 模式下 ASR-side state 在 Fn↓ 锁定（防止用户编辑字典污染段间 ctx），refine/inject 仍在 Fn↑ 锁定（latest user choice）
- **Cleanup**：`buffer.samples.count < 400` 提前返回路径手动 `cleanUpLiveState()`

`startRecording` 决策：

```swift
let useLive = state.liveStreamingEnabled
    && activeBackend.isQwen
    && (recognizer is QwenASRRecognizer)
    && cachedVADBox != nil
let cap: TimeInterval = useLive ? 600 : 60
let outputs = try audio.start(maxDuration: cap)
// ... capsuleWindow.show(levels: outputs.levels) ...
startLiveTranscriberIfEnabled(samples: outputs.samples, useLive: useLive)
```

`stopRecording`:

```swift
if let lt = liveTranscriber {
    await liveIngest?.value          // wait for samples drain (audio.stop() 已 finish 上游)
    lt.finish()                      // signal LiveTranscriber 走 flush + drain
    for try await text in lt.output { transcript = text }   // 最后一次 yield = 最终
} else {
    transcript = try await self.runASR(buffer: buffer, ..., useStreaming: useStreaming)
}
```

```swift
// AppState.swift
@Published var liveStreamingEnabled: Bool {

```

**默认 `false`**。开法：

```bash
defaults write com.voicetyping.app liveStreamingEnabled -bool true
# 重启 app
```

**为什么不暴露 UI**：v0.5.0 是 dogfood 阶段。v0.4.5 fixture benchmark 全 99-100% 通过，但真麦克风背景噪声（键盘声、空调、其他人说话）跟干净 fixture 不一样，VAD 分段质量需要实测一周。等数据稳定再加 Settings UI。

## 明确不做

- **VAD auto-stop**（detect 长时间静默自动结束录音）—— 拆到 v0.5.1。原因：live 模式下 ASR 一直在跑，"Fn 松开"已经是最自然的 stop 信号；auto-stop 跟 live drain 触发的耦合点会让 v0.5.0 太复杂
- **Whisper live 路径** —— Whisper backend 仍走 batch；live toggle 对 Whisper 无效（沿用 v0.4.2 streaming 同样的限制规则）
- **段级 cancel**：MLX 没有 cancel API；Fn 双击 cancel 只能"停止 submit 新段"，已 in-flight 段跑完。能力缺失而不是设计选择
- **胶囊 partial 文本** —— v0.4.4 已经主动去掉，不加回（v0.5.0 主要的延迟红利来自 drain，不需要可视化）
- **Live mode Settings UI** —— 上面说了，等 dogfood
- **Padding 0.3s 给 force-split** —— 25s 之后 force-split 触发频率本来就低，padding 收益边际化；如果实测仍有词被切坏再加

## 快速自测

```bash
make build
defaults write com.voicetyping.app liveStreamingEnabled -bool true
open build/VoiceTyping.app
```

测试场景:

1. **短录音**（<5s）：live vs batch 行为接近 —— 都是按一下 Fn → 出文字。延迟红利不明显
2. **中等录音**（10-25s 连讲）：live 应该明显更快出文字。这是 v0.5.0 的主要受益场景
3. **长 monologue**（30s+）：live 大幅更快。前面 25s 在按住期间已经 transcribe 完，松开后只跑尾段
4. **错觉验证**：开了 live mode 短录音可能感觉"等待时间几乎 0" —— 因为 25s 内整段都在 push 期间转完了，松开瞬间就 inject

观察 `Log.dev` 输出（开 Settings → Advanced → Developer logging）：

```bash
log stream --predicate 'subsystem == "com.voicetyping.app"' \
           --style compact --level info | grep -iE "live|hallucin"
```

应该看到:
- `Live: VAD pre-warmed and cached` (启动后 + 切 backend 时)
- `Live: started with bias context (...)` (Fn↓)
- `Live drain returned <N> chars` (Fn↑)
- 不应该看到 `Live: VAD load failed` 除非 SpeechVAD bundle 缺失

关掉 live 验证回退路径不破:
```bash
defaults write com.voicetyping.app liveStreamingEnabled -bool false
# 重启，验证 batch + (可选) post-record streaming 行为完全等价 v0.4.5
```
