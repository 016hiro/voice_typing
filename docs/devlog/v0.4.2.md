# v0.4.2 — 流式转录（opt-in experimental）

**发布日期**：2026-04-20
**状态**：本地编译 + signed bundle 通过；Settings toggle / Whisper 自动禁用 / VAD 首载路径手工验过；质量对比留给用户盲测
**定位**：v0.4 路线的 Step 2。流式转录作为**可选开关**落地，默认关闭；为后续 Step 3（v0.5，真 live-mic）打地基。

## 用法

1. `make build` → `make run` 升级到 0.4.2
2. Settings → **Models** → 页底新增 **"Streaming (experimental)"** section，勾选 "Show transcript as it forms"
3. backend 必须是 Qwen3（0.6B 或 1.7B）—— Whisper 下 toggle 自动灰掉且描述切换为 "Only Qwen3 backends stream"
4. 首次开启按 Fn 说话松手 → 下载 Silero VAD（~2MB，走 HuggingFace 默认缓存）→ 胶囊里文字逐段浮现
5. 关掉随时退回 batch 路径，行为等同 v0.4.1

**升级无 migration**：`streamingEnabled` 是新 UserDefaults key，默认 false；老用户行为零改动。API key / Keychain / 词典 / profile 都不动。

## 背景

v0.4.1 ship 后去掉了阻挡流式改造的几条债（API key、stable sign、CI）。这一步要做的是流式本身。

设计阶段在 `/think` 里讨论了三个方案：

- **A**：post-record 流式（用上游 `StreamingASR` 高层 API，录完整段后逐段吐文）
- **B**：真 live-mic（自己拼 live mic → VAD → ASR）
- **C**：先做 A 把 UI/toggle/refiner 交互跑通，Step 3 再升级引擎到 B

最终选 C。用户明确说 "流式直接替代非流式风险很大，感觉先做成一个选项比较，我需要深度测试" —— 不是要最大化真实时感知，是要有能做对比实验的底座。

## 范围（最终落地 5 项）

1. **`QwenASRRecognizer.transcribeStreaming`** —— 自建 streaming runner（VAD + Qwen 直拼），不走上游 `StreamingASR`
2. **`AppState.streamingEnabled`** —— UserDefaults 开关，默认 off
3. **AppDelegate 分叉** —— `stopRecording` 里根据 toggle + backend 路由 batch vs streaming
4. **Capsule 尾部截断显示** —— 流式 partial 超过 30 字符时，显示 `…latest30chars`
5. **Settings UI toggle** —— Models tab 新增 "Streaming (experimental)" section，Whisper 下自动禁用

## 为什么不用上游的 `StreamingASR`

计划阶段假设直接用 `soniqo/speech-swift 0.0.9` 的 `StreamingASR.transcribeStream(audio:) → AsyncThrowingStream<TranscriptionSegment, Error>`。读源码时发现问题：

```swift
// speech-swift/Sources/Qwen3ASR/StreamingASR.swift
public func transcribeStream(...) -> AsyncThrowingStream<TranscriptionSegment, Error> {
    AsyncThrowingStream { continuation in
            // 同步循环：VAD chunk → detect segment → asrModel.transcribe → yield
            while offset < samples.count { ... }
            continuation.finish()
    }
}
```

`AsyncThrowingStream.init(_:_:)` 的 closure 在 init 阶段**同步执行**（Swift stdlib 的 `storage.build(Continuation(...))` 就是直接调用）。所以这份代码里调 `transcribeStream(audio:)` 相当于：

1. `AsyncThrowingStream` init 触发 closure
2. Closure 在 caller 线程跑完所有 `yield`
3. 函数返回时 stream buffer 已经**装满所有段**
4. 消费者 `for try await` 秒刷完 buffer

—— 等价于一个同步批量接口，UI 看不到段级进度。上游这层"streaming"是语义化的，不是行为化的。

### 解法：自拼 runner

`QwenASRRecognizer` 里新增一段 runner，直接用低层积木（`StreamingVADProcessor` + `Qwen3ASRModel.transcribe`）：

```swift
// 简化版，实际见 Sources/VoiceTyping/ASR/QwenASRRecognizer.swift
public func transcribeStreaming(...) -> AsyncThrowingStream<String, Error> {
    AsyncThrowingStream { continuation in
        Task.detached {                       // ← 这里是关键：消费者在 main actor，
            let box = await Self.vadActor.get()  //   producer 在 detached thread，
            try self.transcribeLock.withLock {   //   yield 真的唤醒 for-await。
                let vad = box.model
                let processor = StreamingVADProcessor(model: vad)
                // chunk-loop: VAD → .speechEnded → asr.transcribe → yield
                for event in processor.process(...) {
                    if case .speechEnded(let seg) = event {
                        let text = asr.transcribe(audio: segAudio, ...)
                        accumulated += trimmed
                        continuation.yield(accumulated)
                    }
                }
            }
            continuation.finish()
        }
    }
}
```

关键：consumer（`for try await partial in stream { ... }` 在 main actor）和 producer（detached task 里的 `continuation.yield`）在**不同 Task** 上。每次 `yield` 真的切一次调度 → main actor 拿到 partial → 写 `capsuleText` → SwiftUI 重绘。progressive UI 成立。

`StreamingASR` 上游那份代码的核心逻辑（state machine、force-split at maxSegmentDuration、flush）我们对齐复刻了一份（~100 行）。upstream 更新时要同步 port（或者等上游把 `transcribeStream` 改成异步）。

## Sendable 体操

Swift 6 strict concurrency 下，跨 `Task.detached` / actor 边界传递的值必须 `Sendable`。但：

- `Qwen3ASRModel`：非 Sendable。`QwenASRRecognizer` 已经整体 `@unchecked Sendable`，内部通过 `transcribeLock: OSAllocatedUnfairLock` 串行化 —— batch 路径就这么过的。
- `SileroVADModel`：非 Sendable。需要跨 `VADActor.get() async` 返回给 detached task。

`Qwen3ASRModel` 沿用老办法（`@unchecked Sendable` 的 self 穿透）。`SileroVADModel` 单独加一个 box：

```swift
final class SharedVADBox: @unchecked Sendable {
    let model: SileroVADModel
    init(_ model: SileroVADModel) { self.model = model }
}

actor VADActor {
    private var box: SharedVADBox?
    private var inflight: Task<SharedVADBox, Error>?
    func get() async throws -> SharedVADBox { ... }
}
```

Box 是 `@unchecked Sendable` 的空壳 —— 运行时还是得确保 `.model` 只在串行区域（`transcribeLock.withLock`）里解包访问。上游 `SileroVADModel` 的状态是 RNN hidden state，我们在 run 前后都 `vad.resetState()` 避免跨 run 污染。

## VAD 模型生命周期

- **来源**：`SileroVADModel.fromPretrained()` 默认 `aufklarer/Silero-VAD-v5-MLX`，~2 MB
- **存放**：HuggingFace 默认缓存路径（`~/Library/Caches/huggingface/...`），不走我们的 `ModelStore`。刻意分开 —— VAD 是 backend-agnostic 的共享资源，不该和 Qwen 0.6B/1.7B 的"每 backend 一个目录"放一起
- **加载时机**：lazy on first streaming press。首次开 streaming 按 Fn → capsule 显示 "Transcribing"（状态机不变）→ `VADActor.get()` 下载 + 加载 → 进 runner
- **生命周期**：`static` in `QwenASRRecognizer`，跨 backend swap（Qwen 0.6B ↔ 1.7B）存活；app 退出才清
- **首次下载失败**：`VADActor.get()` 抛错 → stream 带 error 结束 → AppDelegate catch 里打 log，capsule 直接隐藏。**没有专门 UI**（设计里记了，v0.4.2 先简化）。后续若常见可加 "VAD 未就绪" 的 toggle 禁用态

## UI / UX 决策

### Capsule 截断

流式 partial 会越来越长，但胶囊宽度封顶 520pt（Morse indicator 吃掉 ~120pt，剩 ~400pt，mono 16pt + tracking(2) 大约 12pt/char）。硬截断点设在 30 字符。

显示策略：**head truncation**（`"…last30chars"`），最新一段始终可见。SwiftUI 原生 `truncationMode` 是 per-Text 的静态属性，要做成动态（只在 `.transcribing` 下用 head）会给 CapsuleView 加 param。改用"在 AppState 层手动加 `…` 前缀"更 localized —— `labelTextForCapsule` 现在对 `.transcribing` 长字符串返回 `"…\(s.suffix(30))"`。

Morse indicator 保留。流式态和 batch 转写态看起来一样（都是 Morse 动 + 某段文字），唯一区别是 capsuleText 在刷新。这是刻意的 —— 状态机没多一档，减少 Settings/文档的解释负担。

### 注入策略

**Commit-on-end，不做 progressive inject**。

想清楚过：段级注入（边转边打到 focused app 的 textfield）在理论上能让用户看到文字实时出现在 Xcode/Terminal/微信里。实际上：

- 用户在流式转写过程中可能已经在该 app 里打字（尤其 raw-first 的心智）—— 段级注入会插在他的 cursor 处，干扰
- 段与段之间有空格还是无空格是 model 输出决定的，跨段拼接会有奇怪空格
- ⌘Z undo chain 会变成 N 段，而不是一次

流式的用户价值定义为"胶囊里看到文字形成"，不是"文字提前进入 textfield"。最终 commit 的行为（raw-first / wait-refine）和 batch 路径完全一致。

### Whisper 不支持

`ASRBackend.isQwen` 是唯一的 gate。当前 backend 为 Whisper 时：

- Settings toggle 被 `.disabled(!state.asrBackend.isQwen)` 灰掉
- 描述文字换成 "Only Qwen3 backends stream. Switch to a Qwen model to enable."
- 即使 `UserDefaults.streamingEnabled` 为 true，`useStreaming` 计算为 false，静默走 batch 路径

用户切回 Qwen 时 toggle 自动可用，保留的 UserDefaults 值自然生效。**没有持久化两份 state**（"用户意图"和"当前可用"）—— 过度设计。

## 代码路径

```
User presses Fn
  ↓
AppDelegate.stopRecording()
  - let useStreaming = state.streamingEnabled && backend.isQwen
  - pipelineTask = Task { ... runASR(..., useStreaming) ... }
  ↓
runASR(useStreaming: true)
  - let qwen = recognizer as? QwenASRRecognizer
  - for try await partial in qwen.transcribeStreaming(...)
  -     state.capsuleText = partial  (on MainActor)
  ↓
QwenASRRecognizer.transcribeStreaming
  - AsyncThrowingStream { continuation in
  -     Task.detached {
  -         box = await VADActor.get()  (lazy-loads first time)
  -         runStreaming(buffer, ..., vadBox: box, continuation)
  -     }
  - }
  ↓
runStreaming (sync, under transcribeLock.withLock)
  - chunk-loop: VAD processor.process → .speechEnded events
  - each segment: asr.transcribe → continuation.yield(accumulated)
  - force-split at 10s maxSegmentDuration
  - flush at EOF
  - fallback: if accumulated still empty, transcribe whole buffer
  ↓
Back in runASR: `latest` = last yielded partial = final transcript
  ↓
injectWaitingForRefine / injectRawFirstThenRefine (unchanged)
```

## 已知限制 / 取舍

### 不是真实时

v0.4.2 的流式发生在 Fn 松开之后。用户按 Fn 说话 10 秒，松开，**然后**看到段在胶囊里一个一个出来。对比 batch：松开后胶囊显示 "Transcribing" 静态文字，转写完整段后一次性 paste。

流式的好处是：
- 转写过程**可见进度**，心理等待时间短
- 长录音（>60s）能完整转出来（VAD 切段，每段 <10s，不触发 Qwen `maxTokens` 限）
- batch 的 60s 硬限来自 `AudioCapture.maxDuration`，**v0.4.2 暂未解除**（`AudioCapture` 该不该解除要看 Step 3 的 live mic 改造思路）

真实时（边说边出文）是 Step 3 的事。engine 已经接近 ready —— `StreamingVADProcessor.process(samples:)` 本来就是 live-feedable 的 chunked API。主要差距：
- `AudioCapture` 需要暴露 "streaming chunks out" 的 API（当前只有 "stop() → full buffer"）
- Fn 中途松开 + in-flight segment 的语义
- Refiner 要不要段级介入（决定不做，和 v0.4.2 一致）

### VAD 切段对 code-switch 质量的影响

**未验证**。假设："I don't know 那个事情" 被 VAD 按静默切成 ["I don't know", "那个事情"] 两段，Qwen 段内语境丢失，可能转写质量下降。需要用户盲测对比。

这是 Option A 最大的未知风险。如果盲测结果显著差，得重新评估"流式当默认"的路线。对冲：默认 off + experimental 标注。

### Silero VAD 下载失败

当前错误处理：stream 抛错 → AppDelegate catch → log → 隐藏胶囊。没给用户可见提示。

容忍原因：
- VAD 是 experimental 功能的依赖，首次启用即下载
- 2MB，网络条件正常时秒级完成
- 失败也不影响 batch 路径（用户只是不能用 streaming）

后续若常见，加 Settings 层的 "VAD not downloaded" 提示。

### 段间空格

`accumulated += " " + trimmed` 简单空格分隔。对英文 OK，对中文会在段边界留奇怪的空格（"你好 今天"）。refiner 开的话会清掉（conservative 模式也会改空格）；refiner off 的话用户看到空格。

不是正确性问题，是观感问题。v0.4.2 不处理。理由：启用 streaming 的用户大概率也开 refiner（experimental 叠加 experimental 心智一致），refiner off + streaming on 是窄场景。

### Capsule head-truncation 简单粗暴

30 字符截断没考虑 unicode grapheme boundary（虽然 `String.suffix(_:)` 是 Character 级的，所以 emoji / CJK 不会 half-glyph），但也没做按词/字截断。长连续无空格字符串（URL、超长 CJK 句子）会就地砍。实用层面够用。

## 验证矩阵（用户深度测试清单）

我这一层只做了编译 + bundle 构建通过。用户做的验证：

- [ ] **质量对比**：同一 15-20 句中英混盲测 batch vs streaming，主观可读性 + refiner 后准确率
- [ ] **短命令**（<2s 单词）：走 streaming 时 VAD fallback（accumulated empty → 整段兜底）是否触发，延迟是否劣化
- [ ] **中长**（10-30s）：段级显示效果；refiner 后文本是否正常
- [ ] **长**（>60s）：**必须解 `AudioCapture.maxDuration` 才能测出效果**，当前限 60s。这是 v0.4.2 的 gap，暂不改
- [ ] **静默间隔 >1s 的多 utterance**：单次录音里有中间停顿，看 VAD 是否切对
- [ ] **Backend 切换**：Qwen 切到 Whisper → toggle 禁用 + 描述更新；切回 → toggle 恢复，上次状态保留
- [ ] **Refiner on/off × streaming on/off** 四矩阵：行为都应该合理
- [ ] **Refiner API key 无效** + streaming on：streaming commit 不受 refiner 失败影响
- [ ] **Silero VAD 首次下载**：网络正常 / 断网两种路径
- [ ] **Raw-first + streaming**：raw-first 拿到的是最终 accumulated（而不是第一段），paste 时行为对

## 不做（明确）

- **真 live-mic 流式**：v0.5 / Step 3
- **段级 refine**：定义为反模式，见上面"注入策略"
- **段级 progressive inject**：同上
- **`AudioCapture.maxDuration` 解除**：和 live mic 改造一起，Step 3 做
- **Whisper 流式**：上游不支持
- **VAD auto-stop**（检测长静默自动结束录音）：单独 ticket，live mic 稳定后评估
- **段间空格智能处理**：价值小，留给 refiner
