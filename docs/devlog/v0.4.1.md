# v0.4.1 — 安全 + 开发流程硬化

**发布日期**：2026-04-19
**状态**：本地验证完成（Keychain migrate round-trip、stable-sign rebuild、debug build smoke 通过）
**定位**：v0.4 流式转录（streaming）前的地基一步。没有用户可见的功能新增，全部在修安全/开发流程的债。v0.4.0 版本号留给流式旗舰。

## 背景

v0.3.2 封版后做了一轮代码审查，发现几条非功能性的遗留问题：
- API key 明文存 UserDefaults（`LLMConfigStore`）—— 任何读 `~/Library/Preferences/com.voicetyping.app.plist` 的进程都能拿到
- 每次 `make build` 产生新的 ad-hoc 签名，cdhash 变化 → macOS TCC 每次都要求重新授权 Microphone + Accessibility
- backlog 里躺了几条一直没排期的 dev-flow 小活（`reset-perms`、CI smoke）

这次把这些一起收掉，为下一步流式改造留出干净的地基 —— 流式会动 `CapsuleView`、`AudioCapture` 的状态机，不想背着"API key 明文"这种债一起动。

## 范围（最终落地 4 项）

1. **API key → Keychain**
2. **稳定自签 codesigning identity**（修 TCC 重新授权）
3. **`make reset-perms` target**（TCC 一键重置）
4. **GitHub Actions `swift build` smoke**

讨论过但**没做**的（连带决策记录留在这里，避免下次重复讨论）：

- ~~LLMRefiner 共享 URLSession~~：原 v0.3.2 review 提的"每次 new URLSession 漏 delegate"问题，看代码发现当前 `URLSession(configuration: .ephemeral)` **没传 delegate**，Apple 文档说无 delegate 的 ephemeral session 释放即回收，没泄漏。唯一剩下的收益是 HTTP keep-alive 复用（~100ms/次），但 refine 间隔 > 1 min 的场景下连接早就死了。整体是 micro-optimization，不值一个 slot。
- ~~录音倒计时 HUD~~：60s 上限是 Qwen `maxTokens: 448` 导致的后端约束，不是 UI 问题。v0.4 下一步流式会走 VAD 分段，60s 上限自然消失 —— 现在给 batch 路径做倒计时 UI 等于给即将退役的模型打补丁。
- ~~可配置快捷键~~：单独中等规模 UX 项，不和本次"安全 + dev flow"的主题一起做，留 Step 2 或 v0.4.x 其他点 release。

## 1. API key → Keychain

### 问题

`LLMConfig` 是个 `Codable struct`，字段 `apiKey: String` 和 `baseURL/model/timeout` 一起 JSON 编码塞进 `UserDefaults.standard` 的 `llmConfig` key。这意味着 OpenRouter/OpenAI 的 key 以明文形式落在 `~/Library/Preferences/com.voicetyping.app.plist`，任何能读这个文件的进程都看得到。

### 修法

把 `apiKey` 从 UserDefaults 拆出，移到 macOS Keychain 的 `kSecClassGenericPassword` slot：
- **service**: `com.voicetyping.app`（bundle id）
- **account**: `openai-api-key`（固定名；未来多 LLM backend 时再加新 account）
- **accessible**: `kSecAttrAccessibleWhenUnlocked`（refiner 只在用户按 Fn 时触发，用户在场即设备解锁）

### 关键设计决策

- **只迁敏感字段**：`baseURL`/`model`/`timeout`/`enabled` 仍 UserDefaults。key 是唯一的敏感信息，Keychain 面越小越好。
- **`LLMConfig.apiKey` 仍是 struct 字段**：保留 in-memory API 不破，避免全 codebase 改 call site。通过 `CodingKeys` 排除，JSON 编码跳过；`LLMConfigStore.load()` 解码完 struct 后再从 Keychain 读 key 填上。
- **`save()` 对空 key 主动 `deleteAPIKey()`**：用户清空 Settings 里的 key 字段 → Keychain slot 也被清掉，保持 UserDefaults 和 Keychain 的"有/无"状态对齐。

### Migration（一次性）

v0.3.x → v0.4.1 启动时，老 UserDefaults JSON 里有 `apiKey` 字段。流程：

1. `main.swift` 在 `AppDelegate()` 构造前调用 `LLMConfigStore.migrateIfNeeded()`
2. 读原始 UserDefaults data，**不走 `Codable`**，用 `JSONSerialization` 解成 `[String: Any]` —— 因为 struct 的 `CodingKeys` 已经排除了 `apiKey`，正常 decode 会直接吞掉明文字段看不到
3. 字典里查 `apiKey` 字段：
   - 非空 → `KeychainStore.writeAPIKey(key)`
     - 成功：继续
     - 失败：记 `LLMConfigStore.migrationFailure = reason`（MainActor-isolated），给 AppDelegate 读
   - 空 → 直接跳过
4. 无论成功失败，**从字典里把 `apiKey` 字段删掉、重编码写回 UserDefaults** —— 明文绝不留
5. 迁移失败路径：AppDelegate 在 `applicationDidFinishLaunching` 末尾检测到 `migrationFailure != nil`，弹 `NSAlert`（"API key couldn't be migrated … please re-enter in Settings → LLM"），"Open Settings" 按钮直接打开 LLM tab

### 为什么 migration 必须在 AppDelegate 构造**之前**

`AppDelegate` 有 stored property `let state = AppState()`，它的 init 会调 `LLMConfigStore.load()`。如果迁移放在 `applicationDidFinishLaunching` 里，`state.llmConfig` 已经按"Keychain 空"装进了内存（`apiKey = ""`），迁移完成也不会回填。放在 `main.swift` 的 `MainActor.assumeIsolated { ... }` 块里，顺序是：迁移 → AppDelegate 构造 → AppState init → load() → 已经从新 Keychain 读得到 key。

### Keychain 读失败 fail-soft

`KeychainStore.readAPIKey()` 吞 error 返回 nil（只 log）。这是刻意的 —— 读失败的后果是 `hasCredentials = false`，refiner 直接返回原文（v0.1 以来的 fail-open 契约），不影响 ASR 核心流程。UI 层再补一个"API key missing"提示是 backlog 里的事。

## 2. 稳定自签 codesigning identity

### 问题（详见 devlog v0.1.0 issue 7）

`make build` 原来用 `codesign --sign -`（ad-hoc）。ad-hoc 签名的 code requirement 长成 `identifier "com.voicetyping.app" and cdhash H"xxxx"`，cdhash 来自二进制内容哈希 —— 每次 build 都变。macOS TCC 用 code requirement 做授权匹配，所以 Microphone、Accessibility 权限**每次 rebuild 都要重新点一遍**。开发期高频 rebuild 的痛。

### 修法

引入 `make setup-cert`（一次性）：生成本地 self-signed codesigning identity `VoiceTyping Dev`，放到**专属 keychain** 里。`make build` 检测到 identity 存在则用它签名，否则 fallback 到 ad-hoc（老行为）。

### 关键设计决策

- **专属 keychain**（`~/Library/Keychains/voicetyping-dev.keychain-db`）而不是 login keychain：
  - **空密码**的专属 keychain 让脚本能非交互地调 `security set-key-partition-list`（给 codesign 白名单访问权）
  - 塞进 login keychain 的话，第一次 `make build` 时 codesign 会弹 GUI "codesign wants to use key X, Always Allow?" —— 用户必须手点一次才能解除
  - 专属 keychain 里只有这一个 self-signed cert，没有其他秘密，"空密码 + 所有 app 可访问"的 posture 不扩大风险面
- **脚本幂等**：`security find-identity -p codesigning | grep "VoiceTyping Dev"` 先查，命中就 skip
- **Makefile 条件分支**：`HAVE_SIGNING_IDENTITY := $(shell ...)`，识别不到则 ad-hoc；不强求用户跑 `make setup-cert`

### 遇到的 2 个坑

- **OpenSSL 3.x PKCS12 MAC 失败**：macOS `security import` 用的是 legacy PKCS12 PBE（RC2+SHA1），OpenSSL 3 默认 AES-256+SHA256，`security` 读不了。必须 `openssl pkcs12 -export -legacy`，同时用**非空密码**（空密码另有一条 MAC verification 边缘 bug）
- **`set -u` + bash 3.2 + 多字节字符**：脚本里原来用中文省略号 `…`（`$KC_NAME…`），macOS 默认 bash 3.2 的 identifier 解析器把 UTF-8 continuation byte 当成变量名的一部分读进去，报 `unbound variable`。全部换成 ASCII `...` 解决

### 副作用

这次 `make build` 产出的 bundle cdhash 从 ad-hoc 换成 `VoiceTyping Dev` 签名 —— **存量用户升级后第一次启动会重新要求 Microphone + Accessibility 授权一次**，之后稳定。

## 3. `make reset-perms`

trivial，一条命令：

```makefile
reset-perms:
	-tccutil reset Accessibility com.voicetyping.app
	-tccutil reset Microphone   com.voicetyping.app
```

开头的 `-` 让 tccutil 没找到授权记录时不炸 target。

## 4. GitHub Actions `swift build` smoke

单 job、`macos-14` runner（public repo 免费的 arm64）、`swift build -c debug --arch arm64`。

范围明确砍掉的：
- 不装 Metal Toolchain（要 GB 级下载 + Apple Developer 登录）
- 不编 MLX metallib（需要 Metal Toolchain）
- 不跑 release build（debug 已足够抓 Swift 6 并发/语法回归）
- 不跑 test（还没 test 套件）

加了 SwiftPM checkout 的 cache（`~/Library/Caches/org.swift.swiftpm` + `.build/checkouts`）和 per-branch concurrency cancellation（新 push 取消老 run），CI 反馈速度不会太慢。

目标是**"这玩意能编过"的 smoke test**，不是"功能正确"的回归套件 —— 后者等 test 套件真正有了再加。

## 踩过没落地的方向

**尝试过的 keychain 方案，没选**：
- `security import` 到 login keychain + `-A` flag：第一次 codesign 仍会 GUI 弹窗（partition-list 问题）
- `security set-key-partition-list` 到 login keychain：要求用户 keychain 密码（≈ login 密码），自动化不友好

**尝试过的 stable sign，没选**：
- ad-hoc 签名 + 固定 identifier：cdhash 仍变化
- entitlement-based 授权固定：不改变 cdhash 计算，无效

## Code Review：额外发现（未修，留给后续）

- **migration failure UX**：`migrationFailure: String?` 是 process-lifetime 的，下次重启就清了。如果用户点了 "Later" 不重入 Settings，第二次启动这个 alert 就再不显示了 —— 但这时候 UserDefaults 已经没 key 了，Settings → LLM 的 field 会是空，用户测 refine 时会被 fail-open 吞掉（原文 paste）。**实际上不会丢数据**，但体验不好。Step 2 可以在 Settings 里做"Key missing" banner。
- **KeychainStore 是单 slot 设计**：`apiKeyAccount` 硬编码 `openai-api-key`。v0.4.x 如果加 Claude / Gemini / Ollama backend，应该按 backend 分账号而不是复用这一个。
- **专属 keychain 在 uninstall 时不清理**：用户 `make clean` 不会删 `voicetyping-dev.keychain-db`。不是大问题（没秘密），但做一个 `make teardown-cert` 对称会更干净。
