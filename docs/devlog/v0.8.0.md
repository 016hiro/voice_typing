## 做了什么

- **#B5 主线 — Per-app independent hotwords**：`ContextProfile` 在本版本内连续重塑两轮（详见"遇到什么"）。最终形态：砍掉 `systemPromptSnippet`（6 个月 dogfood 零使用），换成 `entries: [DictionaryEntry]`（app 私有热词）+ `includeGlobal: Bool`（是否也用全局词，默认 true）。生效集 = `(includeGlobal ? global : []) + entries`，helper `effectiveEntries(global:)` 单点 resolution。模型是"全局共享 baseline + app 私有追加"——全局 Dictionary 装"到处都要的词"，每个 app 额外加自己的私有词，并能单独关全局。Settings tab "Profiles" → "App hotwords"；profile editor sheet 把 snippet TextEditor 换成 "Also use global hotwords" 开关 + app 私有热词内嵌小表格（增删改）+ 二级 entry editor sheet（term/hints/note，复用全局 Dictionary 形态）；表格 "Hotwords" 列显示 `global` / `global + N` / `N only` / `none`。决策依据见 `decisions/0005-per-app-independent-hotwords.md`（supersedes `0004`）。
- **协议链 + telemetry 清理**：`profileSnippet: String?` 参数从 `LLMRefining` 协议、`CloudLLMRefiner`、`LocalMLXRefiner`（含 `LocalLiveSegmentSession`、`makeLiveSegmentSession`）、`LLMRefiningHelpers.compose`、`RefineRecord`、`Meta` 全删；3 个 resolution 点（Fn↓ live / Fn↑ batch / debug capture begin）改为吃 `profile?.effectiveEntries(global: state.dictionary.entries) ?? global`，下游 14 个 `GlossaryBuilder` 调用点收到的是已 resolve 的 effective 列表。`RefineSkipHeuristic.evaluate` 也吃 effective entries——app 私有热词同样能拦短句 skip。
- **#S1 副线 ship —— Refine skip gate**：v0.8.0 #S1 (Swift `RefineSkipHeuristic` + `PhoneticMatcher` + `RefineRecord.gate` 四态遥测 + `Scripts/analysis/skip_gate_report.py`) 已经在 5/20 落地。dogfood 第二轮 11 条 verify 出来 4 个 gate state 全触发，其中 Layer 1 一次实战拯救（`千问。` → `Qwen。`）。完整设计走 `docs/todo/v0.8.0.md` 阶段 1+2。
- **Refine prompt cleanup-vs-rewrite 冲突修复**：用户 dogfood 体感发现 cleanup mode 行为不稳。grep 出根因——`RefineMode.lightPrompt` 写 `MUST NOT: rewrite, paraphrase, translate, or summarize`，而同一个 system prompt 后面 append 的 glossary block 写 `Rewrite these user pronunciations to the canonical spelling`。模型看到同一文档里同一个词 "rewrite" 同时是禁令和指令。修法：① `lightPrompt` 把 `rewrite` 从禁令清单去掉，换成 `paraphrase, restructure, translate, or summarize`（保留禁的意图，去除冲突 token）；② `GlossaryBuilder.buildLLMGlossary` 段落标题改 `Replace these user pronunciations with the canonical spelling on the right`，并加补一句明确"这是 job 1 的 ASR error fix，不受 MUST NOT 影响"。
- **测试**：`ContextProfileTests` 覆盖 `effectiveEntries`（global-only / global∪private / private-only / 空 四态 + 默认值匹配未配置 app）+ 旧 JSON 解码（v0.7.x snippet 和 pre-release whitelist `dictionaryFilter` 都被忽略）+ roundtrip 不带 legacy 字段；`DebugCaptureWriterTests` / `LLMRefiningTests` / `LLMRefiningDefaultStreamTests` / `LocalMLXRefinerIntegrationTests` fixture 全更新；`GlossaryBuilderTests` 加 negative-assert 防止 "Rewrite" 标题再溜回来。233 tests pass, 4 skipped (model-dependent integration tests).

- **dogfood 验证**：build 后真机录入确认 per-app 生效——Claude desktop（配了私有词 NAS + includeGlobal）的 refine glossary 头部含 `- NAS`、ASR bias context 81 字符；同期 Warp（无 profile）glossary 无 NAS、context 77 字符。差值正好是私有词 NAS 只在对应 app 出现，且同时进了 ASR bias 和 refine glossary 两条路。
- **Info.plist bump**：0.7.3/24 → 0.8.0/25.

## 遇到什么

- **per-app 模型在本版本内翻了两次（snippet → whitelist → independent）**：
  - 第一翻（→ whitelist，ADR-0004）：最初列三个 schema 候选——(a) `dictionaryFilter` 挑全局子集；(b) 全新 `PerAppDictionaryStore`；(c) `dictionaryMode` 枚举。本来要保守上 (a) 的"snippet + filter 并存"，用户中途观察"那个页面我自己从来没用过 snippet"，于是决策从"叠加 hotword 字段"翻成"砍 snippet 换 filter"。当时明确**否决了 (b) 独立词典**，理由是"没有 app 专属拼写的需求 + 单一 SSOT"。
  - 第二翻（→ independent，ADR-0005）：whitelist build 第一天 dogfood 就撞墙——whitelist 只能"做减法"，无法给 app 加专属词。要让某词只在微信生效，得先塞进全局再在所有别的 app 白名单里排除它，词住错地方、到处管理它的缺席。ADR-0004 否决 (b) 时假设的"没有 app 专属需求"被第一天真实使用证伪。改成 (b) 的演化版：全局共享 baseline + app 私有追加 + per-app 关全局开关。
  - 关键时机：v0.8.0 两翻都发生在 `gh release` **之前**，没有任何已发布 build 带 whitelist schema，所以是 pre-release 修订不是 migration。代价是两天内翻两次模型的 churn——但第二翻是 concrete dogfood 失败驱动，不是拍脑袋，值得在发版前翻而不是把已知受限的 whitelist 发出去。详见 `decisions/0005-per-app-independent-hotwords.md`。
- **Codable backward-compat 不靠 schema bump 靠语言特性 + 自定义 `init(from:)`**：旧 `profiles.json` 可能带 `systemPromptSnippet`（v0.7.x）或 pre-release `dictionaryFilter`（v0.8.0 第一翻），新结构都没这些字段——CodingKeys 不列它们即被忽略。新字段 `entries` / `includeGlobal` 用 `decodeIfPresent` + 默认值（`[]` / `true`），旧 JSON 自然落到"用全局、无私有"。DiskLayout v1 没动，零迁移代码。`ContextProfileTests` 的 `testCodable_LegacyV073SnippetIgnored` + `testCodable_LegacyPreReleaseDictionaryFilterIgnored` 锁住这两条不变量。
- **14 个 GlossaryBuilder 调用点没有自动 check**：filter 必须在 Fn↓（live）和 Fn↑（batch）两个 resolution 点统一应用，否则有 leak 风险但 unit test 难以覆盖。Mitigation 是 single helper `profile.filteredEntries(from:)` + grep 警觉，靠 review 自我约束。
- **Prompt 修复是 dogfood 反馈驱动的小需求**：原本不在 v0.8.0 scope，体感观察到 cleanup mode 改写热词时模型偶尔拒绝执行。Grep 才看到 `MUST NOT rewrite` + glossary `Rewrite these` 同时存在的内部矛盾。这种"提示词内部不一致"靠 #E1 的 refine CER eval（v0.8.1 skeleton）能更早抓到，先记下来作为 #E1 上线后的回测样本。

## 为什么这么做

- **最终模型选"全局共享 + app 私有"不选"完全独立"也不选"全局子集"**：纯独立（每 app 一张表、无全局）会逼着"到处都要的词"（你名字、Claude）复制进每个 app；全局子集（whitelist）连 app 专属词都加不了。两级模型让两种真实需求各归其位——通用词住全局不重复，专属词住 app 不污染全局。per-app 关全局开关覆盖"这 app 别用全局"的粗粒度抑制；per-term 黑名单（option c）留作未来 additively 扩展，dogfood <20 词时不需要。详见 `decisions/0005-per-app-independent-hotwords.md`。
- **pre-release whitelist profile 不迁移、直接重置**：whitelist 的"只用这几个全局词"语义无法干净映射到新模型，硬迁要把全局条目快照进 profile 造成重复 + drift。dogfood 只有一两个测试 profile，clean reset 比 lossy migration 简单。旧 JSON 的 `dictionaryFilter` / `systemPromptSnippet` 字段 Codable 静默忽略，新字段 default 到"用全局 + 无私有"。
- **删 snippet 不留**：用户明确说"自己从来没用过"——dogfood-only blast radius 一个人。半 ship 半 deprecate 在小型项目里只会成为别人下一年来 debug 的包袱。如果真有需要，重新设计基于届时数据更准。
- **Prompt 修法选"删 token"而非"加 exception clause"**：原本想加一句 `Replacing misheard tokens (job 1) is fine — that's the job` 解释 rule 的例外。但研究 LLM prompt 经验：删除冲突 token 比叠加例外说明更可靠——例外语言会被模型理解成"this rule has a wiggle"，松散整套 rule。"rewrite" 这个词在 MUST NOT 里被 `paraphrase, restructure, translate, summarize` 完全替代，意图保留、冲突消失。
- **#S1 dogfood 不强行等 1 周再 ship**：第二轮 11 条 verify 已经覆盖 4 个 gate state 各一，包括 Layer 1 实战拯救一例。继续等只是统计意义上的更准，不影响 ship——telemetry 是 always-on 的，dogfood 期间数据继续流入，post-ship 可以反过来回看分布有没有惊喜。
