# v0.5.1 — 性能基线 + dl_init 修 + Debug capture toggle + UX 补丁

> 三件并行：
> 1. **性能基线**：先 instrument、量数据、再决定改哪。`Qwen prepare` 全阶段打 wall-clock + 1s warmup 把 Metal JIT 成本前置；新加 `make benchmark-speed` 三 backend × 全 fixture 的速度 baseline；**实测发现"上游 weights 双读"在 prepare 总耗时里只占 < 1%（mmap 红利）**，`dl_init` 反而占 75-90%。砍掉双读 PR，改成 `offlineMode: ModelStore.isComplete(backend)` 让上游早 return，cached prepare 5s → ~1s。
> 2. **UX 补丁两条**：(a) `ModelStore.isComplete` + `repairIfIncomplete` 启动前自检并清半成品下载，避免 prepare 撞上残缺 mlmodelc 直接 crash；(b) 接近录音 cap 时胶囊文字 flash "Xs left"。
> 3. **Debug Data Capture toggle**（Settings → Advanced，默认关）：开启后每次 Fn 录音落盘 audio.wav + segments.jsonl + injections.jsonl + meta.json 到 `~/Library/Application Support/VoiceTyping/debug-captures/`。7 项设计决策全部在 [`docs/todo/v0.5.1.md`](../todo/v0.5.1.md) "Debug 数据捕获 toggle" 段拍板。**LLM refine I/O 不抓**——独立成版本。
>
> `Resources/Info.plist` 0.5.0 → 0.5.1，build 11 → 12。

## 背景

v0.5.0 把 live-mic 流式 ship 了，dogfood 准备启动。三件事在并行排队：

1. 用户反馈 "Qwen 切 backend 慢" 是真实存在但没量化过——v0.5.0 之前从来没写过 prepare timing。先 instrument 才能讨论"改哪"
2. backlog 里两条老 UX 项 (incomplete-model detect + duration warning) 一直没排进版本，搭车做掉
3. dogfood 期间需要数据基础设施。回看 v0.5.0 ship 后 1 周的反馈，光看 `Log.dev` 输出回不到当时的音频上下文——希望开个 toggle 把每次按 Fn 的全套数据落盘，事后能 replay / 跨 backend 对比 / 验 HallucinationFilter ±

## 调研

### dl_init 才是真凶（实测推翻"双读"假设）

预设：上游 `WeightLoader.loadWeights` 和 `loadTextDecoderWeights` 各自全 mmap + 解析头一遍，scopes 称之为 "B 修双读"，预期省 100-500ms。

instrument 完跑了 5 次 cached prepare（M-series Mac，2026-04-21）：

| Run | Backend | total | dl_init | tokenizer | audio_w | text_w_pin | warmup |
|---|---|---|---|---|---|---|---|
| 1 | 0.6B | 8246 | **7545** | 426 | 4 | **11** | 249 |
| 2 | 0.6B | 3711 | **3129** | 426 | 4 | 9 | 141 |
| 3 | 0.6B | 3335 | **2777** | 402 | 5 | 9 | 141 |
| 1 | 1.7B | 4311 | **3275** | 422 | 5 | 9 | 598 |
| 2 | 1.7B | 4668 | **4064** | 408 | 5 | **10** | 180 |

观察被推翻：
- `text_w_pin` 9-11ms / 总 ~5000ms = **0.2%**——远低于"≥ 15% 才做 B"的门槛
- 真因：MLX `loadArrays` 是 mmap-based（C++ `mlx_load_safetensors` thin wrapper）。prepare 阶段只是建立 mmap + Swift 字典包装，没真物理读盘。物理 I/O 摊到 warmup 的第一次 inference 触发 Metal JIT 时
- `dl_init` (3-7s) 才是大头——这是 upstream "Downloading model..." status 到 "Loading tokenizer..." status 之间的时间。cached=yes 也耗 这么久

### dl_init 是 HuggingFace HEAD 检查

读上游 `HuggingFaceDownloader.downloadWeights`：

```swift
public static func downloadWeights(...) async throws {
    if offlineMode && weightsExist(in: directory) {
        progressHandler?(1.0)
        return                                  // ← 跳过所有 HEAD
    }
    // 否则 GET API 列文件清单 + HEAD 每个 .safetensors / vocab.json / merges.txt 验 ETag
    ...
}
```

`offlineMode: false`（我们之前一直传的）会强制走 HEAD 路径——即使本地完整也要发请求验证，3-7 秒的瓶颈就在这里。

**v0.5.1 已经在做的 `ModelStore.isComplete(backend)` 正好可以当"安全的 offline 信号"**：我们自己的检查通过 → 文件存在且大小合理 → 可以告诉上游 "别 HEAD 了，直接走"。

### Warmup 头次 vs 后续

| Run | Backend | warmup |
|---|---|---|
| 1 | 0.6B | 249 |
| 2 | 0.6B | 141 |
| 1 | 1.7B | 598 |
| 2 | 1.7B | 180 |

第一次启动 warmup 显著高于后续——Metal kernel JIT 编译是**进程级 cache**，重启 app 才会重做。1.7B 的 598ms 是单次最大用户感知（仍在"用户按 Fn 之前"的时段，胶囊里看不到，没所谓）。

## 变更

### 1. `QwenASRRecognizer.prepare()` 全阶段 timing + warmup

```swift
// Sources/VoiceTyping/ASR/QwenASRRecognizer.swift
let prepStart = Date()
let timing = LoadStageTimer(initialStage: "init")

let handler: (Double, String) -> Void = { [weak self] fraction, status in
    guard let self else { return }
    timing.mark(stage: status)
    // ... 原有 setState progress 逻辑 ...
}

let canSkipHEAD = ModelStore.isComplete(backend)   // ← v0.5.1 dl_init 修
let loaded = try await Qwen3ASRModel.fromPretrained(
    modelId: modelId,
    cacheDir: cacheDir,
    offlineMode: canSkipHEAD,
    progressHandler: handler
)
self.model = loaded
timing.mark(stage: "loaded")
let loadMs = Int(Date().timeIntervalSince(prepStart) * 1000)

// 1s silence warmup —— Metal kernel JIT 前置
let warmupMs: Int = await Task.detached { [weak self] in
    guard let self else { return 0 }
    let start = Date()
    let silence = [Float](repeating: 0, count: 16000)
    _ = self.transcribeSegmentSync(samples: silence, language: "en", context: nil)
    return Int(Date().timeIntervalSince(start) * 1000)
}.value

let totalMs = Int(Date().timeIntervalSince(prepStart) * 1000)
Log.app.info("Qwen prepare timing: backend=\(backend) cached=\(alreadyDownloaded) offline=\(canSkipHEAD) total=\(totalMs)ms load=\(loadMs)ms warmup=\(warmupMs)ms stages=[\(timing.summary)]")
```

`LoadStageTimer` 是私有 class，封装 "status 字符串变化 → 记录 elapsed → 进 stages 数组" 的逻辑。Status 字符串映射成短 key (`Loading audio encoder weights...` → `audio_w`) 让 log 行可读。

新加的 warmup 段是关键的 UX 改善：以前用户第一次按 Fn 转写会比后续慢 5-10×（Metal JIT），现在那段成本前置到 prepare 阶段，胶囊里看不到，用户按 Fn 时立刻就是"温好的"模型。

**dl_init 修法的实测验证**（用户提供）：开 `offlineMode: ModelStore.isComplete` 之后，cached prepare 总时长应该从 ~5s 降到 ~1s。pending dogfood 的实测重跑确认。

### 2. `make benchmark-speed` 三 backend × 全 fixture

新文件 `Tests/VoiceTypingTests/E2E/E2EBackendSpeedBenchmarkTests.swift`，结构 mirror `E2EVADTuningBenchmarkTests`。一个 test 函数串行跑：Qwen 0.6B → 1.7B → Whisper（顺序固定，因为 WhisperKit 不 expose unload，放最后避免占内存）。

每 backend 第一个 fixture 当 warmup（discard 计时）。每 fixture 测 wall-clock + RTF（real-time factor = `transcribe_ms / (audio_samples / 16000) / 1000`）。Recap 表带硬件型号 + RAM。

`Makefile` 新增 `benchmark-speed` target，`VT_BENCHMARK=1` gate（同 `benchmark-vad`，不进 `make test-e2e` 默认路径）。未下载的 backend 整段 `XCTSkip`，不阻塞其它 backend 跑。

**v0.5.1 没跑出实际数字**——dogfood 时由用户跑一次，结果写进未来 devlog。这个 task 的产出是**测试基础设施**，不是当下数字。

### 3. `ModelStore.isComplete` + `repairIfIncomplete`

```swift
// Sources/VoiceTyping/Support/ModelStore.swift
static func isComplete(_ backend: ASRBackend) -> Bool {
    return isComplete(backend, atDirectory: directory(for: backend))
}

// Testable variant 接受注入的 root
static func isComplete(_ backend: ASRBackend, atDirectory dir: URL) -> Bool {
    switch backend {
    case .whisperLargeV3:
        // 三个 .mlmodelc，每个 weight.bin ≥ 100 KB
        let critical = ["AudioEncoder", "TextDecoder", "MelSpectrogram"]
            .map { dir.appendingPathComponent("...openai_whisper-large-v3/\($0).mlmodelc/weights/weight.bin") }
        return critical.allSatisfy { fileExists($0) && fileSize($0) >= 100_000 }
    case .qwenASR06B, .qwenASR17B:
        // model.safetensors ≥ 1 MB + vocab.json ≥ 100 KB
        ...
    }
}
```

阈值故意保守（100 KB / 1 MB 远低于真实文件 GB 级）——主要捕捉"完全空 / 几 KB 的 stub"这种被杀掉的下载。如果上游切到更小变体，阈值再调。

`repairIfIncomplete` 删失败的子目录（Whisper 是 per-mlmodelc；Qwen 是整个 variant 子目录）让上游 `fromPretrained` 重下。

**AppDelegate.activateBackend** 在构造 recognizer 之前同步跑这俩，repair 触发时 `Log.app.info` 一行 `"Detected incomplete model for X, cleaning and re-downloading"`。

**单元测试** `ModelStoreTests`（11 个）覆盖：missing dir / truncated weights / missing vocab / 完整状态 noop、Whisper repair 只删失败的 mlmodelc / Qwen repair 删整个 variant subdir 等。用 `atDirectory:` 测试钩子注入 temp dir，不污染用户真实 Application Support。

### 4. 录音时长 overlay

新加 `AppState.capsuleOverlayText: String?`，CapsuleView 用 `state.capsuleOverlayText ?? state.statusTextForCapsule`——overlay 设置时压住状态文字，nil 时回退。**关键设计**：不去动 `state.status`，否则 `stopRecording` 的 `guard state.status == .recording` 会误判（Fn↑ 的瞬间撞上 overlay tick 改了 status → 录音卡死）。

```swift
// AppDelegate.swift
private func startRecordingDurationTimer(maxDuration: TimeInterval) {
    let warningWindow: TimeInterval = maxDuration > 120 ? 60 : 10  // live 60s / batch 10s
    let threshold = max(0, maxDuration - warningWindow)
    let start = Date()
    recordingDurationTask = Task { [weak self] in
        try? await Task.sleep(nanoseconds: UInt64(threshold * 1_000_000_000))
        while !Task.isCancelled {
            let remaining = maxDuration - Date().timeIntervalSince(start)
            if remaining <= 0 { break }
            await MainActor.run { [weak self] in
                self?.state.capsuleOverlayText = "\(Int(ceil(remaining)))s left"
            }
            try? await Task.sleep(nanoseconds: 1_000_000_000)
        }
    }
}
```

`stopRecording` 一进去就 `clearRecordingDurationTimer()` 把 overlay 抹掉 + cancel timer task。`applicationWillTerminate` 同样兜底 cancel。

### 5. Debug Data Capture toggle

新文件三个：

- `Sources/VoiceTyping/Support/DebugCapture.swift` —— namespace：folder 路径解析 + retention 选项 + purge 函数
- `Sources/VoiceTyping/Support/DebugCaptureWriter.swift` —— per-session writer：append segment / append injection / finalize (写 audio.wav + meta.json) / abort

**Layout**:

```
~/Library/Application Support/VoiceTyping/debug-captures/
  2026-04-21_18-30-42_a1b2c3d4/
    meta.json         session 元数据
    audio.wav         16 kHz mono Float32
    segments.jsonl    每 ASR 段（含 HallucinationFilter 拦掉的，filter: kept | hallucinationFiltered）
    injections.jsonl  每注入尝试（status: ok | focusChanged | skipped）
```

**七个设计决策**（[v0.5.1.md](../todo/v0.5.1.md) "Debug 数据捕获 toggle" 段全部 `[x] 拍`）：
1. **音频默认开**——开关已经是 toggle 默认关，进了就 opt-in，不存 audio capture 没价值
2. **Refine I/O 不存**——独立成 refine 专版（v0.5.3 或更靠后）
3. **每 session 一个目录**——Finder 直接看，zip 可发；SQLite query 留给后续 CLI 工具
4. **N=7 默认 + Settings 可调 7/14/30/Never**
5. **5 GB 硬顶 + 触顶清最旧到 < 4 GB**
6. **6 类分析支持**（VAD replay / backend 对比 / hallucination ± / load timing / inject latency / focus drop），高频误识别词不做（需要"用户修正捕获"独立 feature）
7. **不做 CLI 工具**，只 ship `docs/debug-captures.md`（schema + jq 一行命令）

**LiveTranscriber 的 SegmentObserver hook**：

```swift
struct SegmentEvent: Sendable {
    let rawText: String
    let kept: Bool
    let startSec: Double
    let endSec: Double
    let transcribeMs: Int
}
typealias SegmentObserver = @Sendable (SegmentEvent) -> Void
```

LiveTranscriber 接受可选 observer，在 `transcribeSegment` 里**对 kept 和 filtered 段都 fire**——这样 segments.jsonl 能记录 HallucinationFilter ± 的真实样本，事后可以审视过滤是否过激。

**整合点**：
- `AppDelegate.startRecording` 调 `DebugCaptureWriter.begin(state:)` —— state.debugCaptureEnabled 是 false 时返回 nil，调用方无需 if
- `AppDelegate+Live.startLiveTranscriberIfEnabled` 把 writer 包成 SegmentObserver 传给 LiveTranscriber + 在 inject task 里 appendInjection
- batch 路径在 pipelineTask 末尾 appendSegment(全文一段) + appendInjection(全文一次) + finalize
- `stopRecording` 早返回路径（buffer too short）调 writer.abort()
- `applicationDidFinishLaunching` 起手做 `purgeOlderThan` + `purgeIfOverCap`

**Settings UI** 在 Advanced tab 加新 SectionCard：toggle、warning 文案、Auto-purge after dropdown、Open captures folder 按钮、Clear all captures 按钮、实时 on-disk size 显示（2s 自动刷新）。

**Bug 修过一次**（dogfood 早期发现）：on-disk size 显示 "Zero KB"。原因是复用了 `ModelStore.humanReadableBytes` 扩展（`.useMB | .useGB`，专为 GB 级模型设计），180 KB 的 capture 被四舍五入到 "Zero MB"。换成 inline `.useKB | .useMB | .useGB` 后正常显示 "180 KB"。Timer-based 2s 刷新也是这次加的——AdvancedTab 打开期间用户做新录音，数字会动。

## 快速自测

### A. dl_init 修法验证（最大变化）

```bash
log stream --predicate 'subsystem == "com.voicetyping.app"' --level info | grep "prepare timing"
```

切到 Qwen backend（已下载），看 `cached=yes offline=yes`，`dl_init` 应该从 3-7s 降到几百 ms 以内（剩下的是 Swift 调用 + tokenizer 解析）。如果 `offline=no` 但你以为 cached，说明 `ModelStore.isComplete(backend)` 检查失败——把对应 model dir 的 `model.safetensors` / `vocab.json` 文件大小贴上来排查。

### B. 录音时长 overlay

最快验法走 batch 模式（cap 60s）：

1. 关掉 live：`defaults write com.voicetyping.app liveStreamingEnabled -bool false` + 重启
2. 按住 Fn 沉默或胡乱说 50s+
3. 第 50s 起胶囊文字应该开始 flash "10s left" → "9s left" → ... → "1s left"
4. 录音到 60s 自动结束（AudioCapture maxDuration 触发 stop）

Live 模式（cap 600s）窗口是 60s，需要憋 9 分钟，自测嫌长可以临时改 `cap = 30` + `warningWindow = 5` 跑两次。

### C. Debug Data Capture

1. Settings → Advanced → 勾上 Debug Data Capture toggle
2. 按 Fn 录两段
3. 关 toggle、点 "Open captures folder"，应该看到两个时间戳目录，每个含四个文件
4. `cat */meta.json` 看 backend / liveMode / totalAudioSec / totalSegments 是否吻合
5. `cat */segments.jsonl | jq` 看每段的 kept / hallucinationFiltered 决策
6. Settings 上的 "On disk" 应该实时刷新；点 Clear all captures 应该秒清

### D. Incomplete model 修复

1. 关 app
2. `truncate -s 50 ~/Library/Application\ Support/VoiceTyping/models/qwen-asr-1.7b/models/aufklarer/Qwen3-ASR-1.7B-MLX-8bit/model.safetensors`（把权重截到 50 字节）
3. 重启 app，看 `Log.app.info`：`Detected incomplete model for qwen-asr-1.7b, cleaning and re-downloading`
4. 等下载完，转写应该正常

## 补遗 — Silent data loss in DebugCaptureWriter（2026-05-01 发现，commit `38eb581` 修）

> 整段 v0.5.1 → v0.6.3 期间所有 dogfood capture session **都缺 audio.wav 和完整 meta.json**。从来没人发现，因为 `segments.jsonl` 看起来正常，没人对着目录 `ls` 数文件。**之前所有 dogfood 数据可用性减半**——音频回放、跨 backend 对比、live drain 时间分布全部受影响。

### 怎么发现的

v0.6.3 #R8 加 refine I/O capture，让用户去检查 `refines.jsonl` 是否生成。用户 `ls` 目录发现连 `audio.wav` 和 `injections.jsonl` 都没有，只有 `meta.json` + `segments.jsonl`。检查 `meta.json` 没有 `endedAt` / `totalSegments` / `totalInjections`——只有 `begin()` 写的 partial meta（v0.5.3 引入的 fix）。

回看历史 `~/Library/Application Support/VoiceTyping/debug-captures/` 里 200+ 个 session 目录，**几乎全是这个状态**。

### 真因

`DebugCaptureWriter` 的所有 `queue.async` 闭包都用了 `[weak self]`：

```swift
func finalize(audio: AudioBuffer) {
    queue.async { [weak self] in
        guard let self, !self.finalized else { return }
        // write audio.wav + final meta.json
    }
}
```

生产里 writer 只被 `pipelineTask` 闭包内的 `let captureWriter = currentDebugWriter` 局部变量持有强引用（`stopRecording` 在 line 653 已经把 `self.currentDebugWriter = nil`，写得很谨慎防止下个 Fn 手势污染当前 session）。

`finalize()` 是 pipelineTask 的**最后一个**入队动作。调完它之后 Task 立即结束 → 闭包释放 → `captureWriter` 局部释放 → writer 引用计数归零 → **writer 析构**。然后 queue 才轮到执行 finalize 闭包，`[weak self]` → nil → `guard` 直接 return → audio.wav / final meta 都没写。

`appendInjection`（line 837）紧贴 finalize 之前调，同款 race 同款丢。

`appendSegment`（line 788）逃过去了——它在 pipelineTask 早期入队，那时 writer 还被 Task 闭包牢牢持有，queue 在 microsecond 级别就跑完了那个闭包。

### 为啥这么久没人发现

1. 受影响的是"大文件"和"完整 meta"，**不是"全部丢"**——目录里有东西，jq 跑 segments.jsonl 也能出结果，肉眼看分析脚本没明显报错
2. 现有分析脚本（`summary.py` / `hallucination_review.py` / etc.）都基于 `segments.jsonl`，没有跑"audio.wav 缺失率"或"endedAt 缺失率"的 sanity check
3. v0.5.3 加了"`begin()` 立即写 partial meta"作为对"crash 留下完整 meta"的兜底——partial meta 的存在反而让"finalize 没跑"这件事看起来像"只是 partial meta，没事"，没触发 alarm
4. dogfood 用户主要看 capsule UI 行为是否正常，没去翻 capture 目录

### 修法（commit `38eb581`）

去掉所有 `queue.async` 闭包的 `[weak self]`。每个闭包 strong-capture self 直到执行完毕再释放，writer 必然存活到 queue 排空。

不会造成永久 retain cycle：writer 拥有 queue（`private let queue`），queue 短暂 retain 闭包，闭包执行完即释放 self。整个 lifecycle 是有限的。

加了一个 regression 测试 `testFinalize_AfterWriterRefDropped_StillWritesAudioAndMeta`：用 `do { writer.finalize(...) }` 块包住 writer 让局部强引用退出作用域，再 poll 文件系统验证 finalize 工作完成。Pre-fix 代码会失败。

### 教训

- **`[weak self]` 是默认动作但不是默认正确**——尤其当 self 的 lifetime 跟 enqueued work 的 deadline 没有显式约束时
- **看似"局部 silent failure"可能是结构性数据丢失**——下次写诊断/落盘类基础设施，需要一个"完整性 sanity check"脚本（"目录里所有 session 都有 audio.wav 吗？meta 都有 endedAt 吗？"）跑在 dogfood 数据上
- **dogfood 数据本身的健康度需要被监控**——不能只信任"工具看起来在工作"
