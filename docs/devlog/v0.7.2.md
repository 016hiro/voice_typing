## 遇到什么

- **#B3 整段被 dogfood 证伪**：v0.7.1 上的 keep-alive App Nap 修法是 prophylactic（抑制 App Nap / 补 wake hook / 升 QoS / 升日志可见性），dogfood 数据表面看每条都生效——`keep_alive.py` 显示 v0.7.1 build 的 tick `>0` 列压过 `==0`。但**整套打法本身是错的**：保温 timer 自身在 cold path 上（silent buffer 没 EOS 信号 → decoder 跑满 448-token budget → 单 tick 30-45 s），既保不了温也阻塞用户 Fn↓。真正的根因不是 timer 没跑，是 macOS compressor 主动决定压权重——这是设计上的功能不是 bug。修法应该治"compressor 别压"，不是"周期触摸权重"。教训：观测要直接答"修法生效了吗"（≥5 s 率降了吗）而不是"修法的子步骤跑了吗"（timer 跑了吗）
- **`.reservation` ticket 看上去对其实错**：第一次读 mlx-swift API 时直觉 reservation 是"宣告我打算用这么多"的语义，看上去比 active 干净。文档 `Articles/wired-memory.md:28-30` 反复读三遍才注意到 reservation **idle 时不 keep limit elevated**——而 idle window 正是要保护的。一字之差，仅 active 能用
- **mlx-swift 不是 mlx-swift-lm 的 transitive export**：mlx-swift-lm 依赖 mlx-swift 但产品没 re-export。我们 target 没法直接 `import MLX`，必须 `Package.swift` 加直接 dep，pin 到 mlx-swift-lm 自己的 `.upToNextMinor(from: "0.31.3")` 避免 resolver 冲突
- **v0.7.1 已经 ship 出去了**：2026-05-02 那次 close-iteration 直接 tag + GitHub release 出去了 3 次下载。本次 dogfood-driven fix 走 v0.7.2 单独发，不动 v0.7.1 已发的内容——Sparkle 自动给已下 v0.7.1 的用户推 v0.7.2

## 为什么这么做

- **用 process-wide pin 不用 per-region wire**：mlx-swift `WiredMemoryTicket` 走 `mlx_set_wired_limit` → Metal MTLResidencySet，process-wide 一个限。理论上可以单独 wire 某些 buffer，但 (a) MLX 内部分配很多 tensor，手动 wire 会漏；(b) 一个 ASR 后端就一个进程范围，process-wide 已够；(c) `WiredMemoryManager` 是上游推荐 API，不直接调 C
- **1.5 GB ceiling 不按权重精确算**：Qwen3-ASR 1.7B 4-bit 权重 ~850 MB，加 Metal scratch headroom 留 1.5 GB，远低于 `GPU.maxRecommendedWorkingSetBytes()`（最小 8 GB Mac ~5.6 GB）。如果未来加 1.7B 全精度或更大模型再调
- **不留 belt-and-suspenders**：理论上 keep-alive + pin 双保险更安全。实际上 keep-alive 90 s 周期 dummy transcribe 是真实电耗 + 有 Metal queue 占用风险，pin 已经验证能独立 hold 11.5 h overnight，留下来纯成本。`TranscribeWatchdog` 仍保留——被动观察，对未来未知 hang 模式有用
- **ADR Negative 段写实在的代价**：1.5 GB 永久 reserved 是真代价——8 GB Mac 上 19% 物理内存不让 compressor 用。未来如果有人投诉内存压力，要从 ADR-0002 的 Negative 段读到这是为延迟做的取舍
