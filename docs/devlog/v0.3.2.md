# v0.3.2 — Settings bug fixes + UI halo fix + code review

**发布日期**：2026-04-19
**状态**：修复完成；Settings UI 阴影问题已手动验证，其余端到端验证待补

## 修复的 Bug

### 1. Settings 输入框不支持 ⌘A / ⌘C / ⌘V / ⌘X

**现象**：LLM tab 里的 Base URL / API Key / Model 字段、Dictionary 编辑器、Profiles 编辑器里的所有 TextField，粘贴不生效、全选不生效——用户要拷一段 key 进去得手动一个个字符选。

**根因**：`main.swift` 把 app 设成 `.accessory` 激活策略（无 Dock 图标、无 menu bar）。AppKit 的 ⌘A/⌘C/⌘V/⌘X 默认是通过主菜单的 Edit 菜单的 `keyEquivalent` 触发的——没有主菜单，就没有这些快捷键的 responder chain 入口。SwiftUI TextField 自己不处理这些快捷键（它依赖系统路由）。

**修法**：在 `BorderlessKeyWindow.performKeyEquivalent(with:)` 里拦截这五个组合，用 `NSApp.sendAction(_, to: nil, from:)` 转发给 first responder。`to: nil` 会让 AppKit 走 responder chain，field editor（NSTextView）实现了这些 selector，就会正确执行。

关键细节：
- 先调 `super.performKeyEquivalent` 让 SwiftUI 自己的 button shortcuts（Return → Save、Escape → Done）仍然优先匹配
- 只拦带 Command 的、且不带 Control/Option 的组合——避免误吃带 Option 的其他绑定
- ⌘Z 和 ⌘⇧Z 分别走 `undo:` / `redo:`
- 字符串 selector（`Selector(("redo:"))`）仅用于 undo/redo——这俩没有统一的 Swift type-safe 选择器；其他四个用 `#selector(NSText.foo(_:))`

代替方案（评估过、没采用）：
- 安装一个 hidden 主菜单：`NSApp.mainMenu = NSMenu()` + Edit 子菜单。问题是 `.accessory` app 的主菜单仍会在 window 活跃时出现在屏幕顶部——副作用比 performKeyEquivalent 重
- 在每个 TextField 上包一层自定义 NSViewRepresentable：代码量爆炸

### 2. Qwen 切换时闪一下"正在下载中"（模型其实早下好了）

**现象**：菜单里切到 Qwen3-ASR 0.6B 或 1.7B，即使权重已经在本地，menu bar 和 Settings 里都会先闪一下 "Downloading 0%" 甚至短暂跑个百分比，然后才进入 Active。

**根因**：`Qwen3ASRModel.fromPretrained`（来自 `soniqo/speech-swift`）的 progress handler 不区分"已缓存、跳过下载"和"正在真下载"。它无条件地：
1. 先 `progressHandler?(0.0, "Downloading model...")`——触发 UI 显示 "Downloading 0%"
2. 调 `HuggingFaceDownloader.downloadWeights` → 内部调 `hub.snapshot()`，对每个文件做 HEAD 请求验证 ETag，即使缓存完整，progress 会在几百 ms 内从 0% 跑到 80%
3. 才进入真正的权重加载阶段（0.80 → 0.85 → 0.92 → 1.0）

我们 `QwenASRRecognizer.prepare` 把这些 fraction 照直 forward 成 `.loading(progress: fraction)`，UI 因此把 HEAD-check 阶段显示成 "Downloading"——用户体感是"明明下好了为啥还下"。

**修法**：在 `prepare` 入口 snapshot 一下 `ModelStore.isDownloaded(backend)`。缓存存在时，把 progress handler 的所有回调都压成 `-1`（indeterminate），UI 会显示 "Loading…"（已有代码，WhisperKit 走的就是这条 path）。否则走原来的 fraction 映射。

权衡：如果之前下载被中断、存在部分权重（isDownloaded 已经返回 true 但实际还要补下），用户看到的是持续 "Loading…" 而不是 "Downloading 60%"。接受——边缘情况，换来主流程的视觉清爽。

### 3. OpenRouter 等自定义 endpoint 调用失败

**现象**：用户把 OpenRouter 的官方 URL `https://openrouter.ai/api/v1/chat/completions` 粘到 Base URL 字段，直接 curl 同样的参数 OK，但 app 内 Test 按钮报 `HTTP 404: {"error":{"message":"Not Found","code":404}}`。

**根因**：旧代码把"Base URL"当前缀处理，在 `chat()` 里强行 `baseURL + "/chat/completions"` 拼 URL。用户如果粘的是完整 endpoint URL（这是最常见的情况——文档展示的就是完整 URL），请求就打到了 `.../chat/completions/chat/completions`。OpenAI 自己还好因为它的默认前缀 `.../v1` 看起来像 base；但一旦换到 OpenRouter、Groq、DeepSeek 等（文档直接给完整 URL），这个拼接就 404。

**第一版修法（已弃用）**：在 `save()` 里 auto-strip 结尾的 `/chat/completions`，让两种输入都能 work。问题：这是 ongoing magic——用户输入什么、字段显示什么、实际 POST 到哪，三者不一致。审查过程中被点出来"透传用户输入就好了"，方案重写。

**采纳的修法**：纯粹的 literal pass-through。
- 字段语义从"base URL"改成"API URL"（placeholder 也换成完整 endpoint URL）
- `chat()` 用 `config.baseURL` 作为 POST 目标，**不**追加任何后缀
- `save()` 只做 whitespace/newline trim，别的一律不动
- `LLMConfig.default.baseURL` 改成完整 URL `https://api.openai.com/v1/chat/completions`
- `LLMConfigStore.load` 加一次性 migration：老的 `.../v1` 型 baseURL 自动补 `/chat/completions`，之后保存就写全量

权衡：
- 原则"用户输入 == 实际请求"比"减少用户配错"重要——前者是契约，后者用更好的错误提示代偿（HTTP 404 的 body 已经包含在错误消息里）
- Migration 只跑一次（老 schema → 新 schema），不是每次 save 都变换，避免了 ongoing magic 的反直觉

顺带一起改进的错误诊断：
- 所有失败路径都把 response body 前 200–300 字符拼进 error message，Console.app 也记 `Log.llm.error`
- OpenRouter 偶尔 HTTP 200 + `{"error": {...}}` body 的情况（上游 provider 拒绝时）单独识别出来，error message 展示 `error.message` 而不是笼统的 "Unexpected response shape"
- 加了 `HTTP-Referer` + `X-Title` header（OpenRouter 推荐但非必需的归因头）
- `save()` 对 `apiKey` 和 `baseURL` 都改用 `whitespacesAndNewlines`（老代码只 trim `.whitespaces`，不带 `\n`——浏览器复制的 key / URL 常带尾随 `\n`，导致 Authorization header 被 URLSession 拒绝）

### 4. Settings 面板外侧残留半透明灰色直角框

**现象**：Settings 主面板本身是圆角，但在正常圆角边框外面还能看到一层很淡的灰色直角框/阴影。第一次修完后问题会减轻，但仍能隐约看见一层矩形外轮廓。

**为什么第一次修复不够**：旧版本只处理了最显眼的一层问题：
- `NSWindow.hasShadow = true` 会按 window frame 画系统阴影，所以先关掉了系统阴影
- 再用 SwiftUI `.shadow()` 给圆角 panel 自己画阴影

这能去掉最粗的一圈矩形阴影，但还有更淡的一层残留，不是 `NSWindow` 自带 shadow，而是 **SwiftUI/AppKit 合成链路里某个矩形 backing layer 被一起参与了阴影/材质渲染**。

**最终根因**：这次问题不是单点 bug，而是几层因素叠加：

1. 外层 borderless window 比 panel 大一圈，给软阴影留透明空间；方向没错，但如果透明留白太小，圆角阴影会被 window bounds 裁成直线边。
2. `NSHostingView` 最初铺满整个外层 window，而不是只包住可见 panel；这给 SwiftUI 一个“整块大矩形宿主”的合成边界。
3. panel 阴影最初挂在整个 SwiftUI 组合结果上，并且用了 `compositingGroup()`。这很容易触发“先把 subtree 拍平成一个离屏矩形纹理，再对这个矩形纹理做模糊/投影”。
4. `glassEffect(...)` 在这个 borderless window 结构里也会引入额外矩形 backing，虽然不是最终唯一根因，但会加重“外面还有一层框”的观感。

最终表现就是：即使 panel 视觉上已经被 `clipShape(RoundedRectangle(...))` 裁成圆角，只要阴影或材质采样发生在裁剪之前、或者发生在一个更大的矩形离屏层上，用户仍会看到一层很淡的直角灰框。

**排查过程**：这次不是一把修好的，按下面顺序逐层收缩嫌疑：

1. **先放大 `shadowMargin`**  
   从 `28` 提到 `48`，确认最外层那圈不是简单的“阴影撞到 window 边界被裁直”。
   结果：矩形框变淡，但没完全消失。

2. **去掉 `glassEffect(...)`，改成 shape-scoped material**  
   把 panel 背景从系统 Liquid Glass 改成 `RoundedRectangle + .ultraThinMaterial + 渐变叠层`，确保材质采样严格限制在圆角 shape 内。
   结果：矩形框仍在，说明不只是 glass API 自己的问题。

3. **把 `NSHostingView` 从“铺满整个 window”缩成“只包住 panel”**  
   外层透明留白仍由 AppKit `container` 提供，但 SwiftUI hosting 本体不再是大矩形。
   结果：仍有残留，说明宿主范围不是唯一根因。

4. **显式把 `NSHostingView.layer.backgroundColor` 设为透明**  
   避免 hosting view 的 layer 默认/继承背景参与合成。
   结果：有帮助，但还不够。

5. **去掉 `compositingGroup() + 对整个 subtree 做 shadow` 的路径**  
   这是最终有效修复。改成：
   - 单独创建一个 `RoundedRectangle` 作为 shadow caster
   - 阴影只挂在这个圆角 shape 上
   - `panelContent` 自己只负责材质、圆角裁剪和描边，不再承担整个合成组的阴影

   这一步避免了 SwiftUI 先把整棵 panel 子树拍平成矩形离屏层，再对那个矩形做 shadow。用户看到的淡灰直角框在这里终于消失。

**最终修法**：
- 保持 `NSWindow.hasShadow = false`
- `shadowMargin` 增大到 `48`
- `NSHostingView` 只包住 panel 尺寸，不铺满外层 window
- `NSHostingView` layer 明确设为透明
- panel 背景不再用 `.glassEffect(...)`，改为圆角 shape 内部的 material + gradient
- panel 阴影改由独立 `RoundedRectangle` 绘制，取消 `compositingGroup()` 那条矩形离屏合成路径

**为什么这版修法可靠**：  
现在“可见圆角 shape”“材质采样区域”“阴影投射区域”“hosting view 实际尺寸”四者终于对齐了。只要这四层边界一致，系统就没有机会在圆角外再留下一个额外的矩形灰框。

**手动验证**：用户本地重新 `make build` 后确认：
- 第一轮修复后：问题“没那么明显了，但是还是有一层”
- 最终修复后：`“现在终于没了”`

## Code Review：额外发现（未修，留给后续）

1. **API key 明文存 UserDefaults**（`LLMConfigStore`）。应该走 Keychain（`SecItemAdd` 或 `KeychainAccess` 包装）。和 migration 一起做。
2. **Raw-first 的 ⌘Z 会撤销用户中间键入**：如果 raw paste 后、refined 还没回来，用户手动打了字，replace-injection 触发的 ⌘Z 撤销的是用户的打字，不是 raw paste。`focusSafe` 只检查 app 切换，不检查 intra-app edits。缓解方案：只在 raw paste 是"最近一次操作"时 replace（监听 app-level 撤销栈比较难，实际可能通过 clipboard 对比或记录 timestamp 判定）。
3. **LLMRefiner 每次 refine 都 new `URLSession`**：Apple 文档说不 invalidate 会泄漏 session delegate；这里用 ephemeral + 无自定义 delegate，实际影响小但不规范。改共享 session。
4. **Dictionary/Profiles Table 的 `.id(state.xxxTick)`** 会在每次 tick bump 时整个 table 重建，导致 scroll position / selection 重置。改为让 store 发布 `@Published var snapshot: [T]` 可避免，但改动需同步更新所有消费者。
5. **模型下载中切 backend 不会取消网络请求**：`activateBackend` 的 `backendSwapTask?.cancel()` 只取消 Task；HuggingFace 的 snapshot 不响应 Task.isCancelled。小问题——切走的模型的下载继续在后台跑，只是结果被丢弃。
6. **FnHotkeyMonitor 的 event tap 超时恢复**：tap 被 OS `tapDisabledByTimeout` 时我们自动 re-enable——OK。但如果 Accessibility 权限被撤销，tap create 会成功但 callback 永远不触发——需要定期 sanity check。现在只在启动时 check。
