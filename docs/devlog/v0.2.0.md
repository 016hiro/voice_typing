# v0.2.0 — 可切换 ASR 后端

**发布日期**：2026-04-16
**状态**：已验证可用

## 交付的功能

- **三档可切换 ASR 后端**，保持 `SpeechRecognizer` 协议不变：
  - Whisper large-v3（WhisperKit）
  - Qwen3-ASR 0.6B（MLX，4-bit 量化）
  - Qwen3-ASR 1.7B（MLX，8-bit 量化）
- **菜单 Model 子菜单**：每项显示名称 / 预估大小 / 状态（Active / Downloaded / Not downloaded / Downloading N%）。点击即切换，未下载自动触发下载。
- **设置窗口分标签**：
  - **Models** — 三行模型表格，每行 [Switch/Download] 按钮 + 删除按钮 + 实时下载进度条
  - **LLM** — 原 LLM Refinement 配置页
- **真实下载进度**：Qwen 后端接入 `progressHandler` 回调，菜单文字显示 "Preparing Qwen3-ASR 1.7B… 42%"
- **模型文件管理**：
  - 切换后端**不删除**旧模型（下次切回零等待）
  - 用户主动删除：`Manage Models…` 二次确认后清理
  - 删除当前活动后端时，recognizer 重新加载并提示用户
- **迁移路径**：v0.1.0 老用户在 `models/models/argmaxinc/...` 的 WhisperKit 权重会自动迁移到 `models/whisperkit/models/argmaxinc/...`，切回 large-v3 无需重下。
- **MLX 未装时不崩溃**：`MLXSupport.isAvailable` 检测 `mlx.metallib` 是否与可执行文件同位；如果没有，默认后端降级到 Whisper，Qwen 选项保留但点击时显示清晰错误（不再把整个进程带崩）。

## 设计决策

### 默认后端是 Qwen-1.7B（MLX 可用时）/ Whisper large-v3（fallback）

- Qwen3-ASR 1.7B benchmark 上多语言多数据集**优于 Whisper large-v3**，参数更少（1.7B vs 1.55B），量化后磁盘占用 ~1.4 GB vs WhisperKit CoreML 的 ~3 GB。
- 0.6B 的 92ms TTFT 在我们"按 Fn → 一次性 utterance"的模式下用不上，留给后续流式版本。
- MLX 没装时 fallback 到 Whisper，保证装 Xcode 但没装 Metal Toolchain 的用户不会开机就挂。

### 为什么切换不删除旧模型

- 用户可能想对比不同模型准确率、或临时切回 Whisper 做验证 —— 反复下载 1.4 GB 体验太差
- 磁盘成本由用户决定：显式的 `Manage Models…` 页面让用户自己掌控

### 为什么不抽象出"通用下载器"层

- WhisperKit 的下载进度是黑盒（只能 indeterminate），Qwen 走 `HuggingFaceDownloader` 有细粒度回调 —— 两种模式本质不同，强行统一反而复杂
- 进度传递通过 `SpeechRecognizer.stateStream` 的 `.loading(progress:)` 已经足够，UI 层不关心细节

## 实现期遇到的问题与修复

### Issue 1: `speech-swift` 的 macOS 版本要求

**症状**：README 写的"macOS 14+ / iOS 17+"但 `Package.swift` 里是 `macOS("15.0")`，加依赖后 swift build 直接拒绝 macOS 14 target。

**原因**：`CoreMLASRModel.swift` 用了 `MLState` KV cache（macOS 15+ API），所以整个包 floor 必须 15。README 没同步。

**修复**：把 `Package.swift` `platforms: [.macOS(.v14)]` 升到 `.v15`，同时 `Info.plist` 的 `LSMinimumSystemVersion` 14.0 → 15.0，`swift-tools-version` 5.9 → 6.0（`.v15` 枚举值需要 6.0 PackageDescription）。

**影响**：Intel Mac 和 Sonoma 用户无法升级到 v0.2.0，但本项目 v0.1.0 起就仅 Apple Silicon。

### Issue 2: Swift 6 工具链的严格并发检查炸了一堆老代码

**症状**：升到 `swift-tools-version: 6.0` 后 `swift build` 产生 5 个 hard errors，全是 v0.1.0 早就存在但 5.9 工具链不检查的问题：

1. `CapsuleView.SizeKey.defaultValue: CGSize` — static var → 非 Sendable global mutable state
2. `InputSourceManager.shared` — 非 Sendable 类的 static let
3. `Permissions.checkAccessibility` 里的 `kAXTrustedCheckOptionPrompt` — Carbon extern var
4. `QwenASRRecognizer.transcribe` 的 `NSLock.lock()/unlock()` — 在 async 上下文不可用（会死锁）
5. `LLMRefiner.refine()` 从 MainActor 跨到 nonisolated 方法时 self.refiner 不是 Sendable

**修复**：
- `defaultValue: var` → `let`
- `InputSourceManager: @unchecked Sendable`（所有调用都在 `TextInjector` 的 `@MainActor` 上）
- Carbon 常量换成硬编码字符串 `"AXTrustedCheckOptionPrompt"`
- NSLock → `OSAllocatedUnfairLock` 的 `.withLock { }` API
- `LLMRefiner` 标 `Sendable`（本身是无状态的 URLSession wrapper）

**教训**：第一次"成功"的 v0.2.0 commit 其实是因为 binary 没覆盖 —— swift build 失败，但 build 目录的旧 .app（v0.1.0 cached）仍在原地，测试时打开的是那个。下次工具链大版本升级时先验证版本号。

### Issue 3: MLX 运行需要 `mlx.metallib`，而 SwiftPM 构建不生成

**症状**：`swift build -c release` 成功，app 能启动，但 Qwen 后端在 `prepare()` 阶段抛 `Failed to load the default metallib`。

**原因**：`mlx-swift` 只把 Metal 源码放进 C++ target，**不在 SwiftPM 构建期自动编译成 `.metallib`**。`speech-swift` 仓库提供了 `scripts/build_mlx_metallib.sh`，需要手动调用 `xcrun metal` 编译 33 个 metal kernel 源文件。MLX C++ 运行时按以下顺序查找：
1. `mlx.metallib` 在可执行文件旁边（`Contents/MacOS/mlx.metallib`）
2. `Resources/mlx.metallib`
3. SwiftPM Bundle `default.metallib`
4. `Resources/default.metallib`
5. 编译时 `METAL_PATH` 宏指向的路径

**修复**：Makefile 加 `metallib` target，在 `make build` 时自动调用脚本，把产物放到 `.build/release/mlx.metallib`，然后 `cp` 到 `Contents/MacOS/`。

文件：[`Makefile`](../../Makefile)

### Issue 4: Metal Toolchain 不在默认 Xcode 安装里

**症状**：`bash scripts/build_mlx_metallib.sh` 报 `missing Metal Toolchain; use: xcodebuild -downloadComponent MetalToolchain`。

**原因**：Xcode 16/26 之后，Metal shader compiler 作为可选组件，默认不装。

**修复**：Makefile 加 `make setup-metal` target 调用 `xcodebuild -downloadComponent MetalToolchain`。如果这个命令本身因为 Xcode plugin 状态不一致失败（DVTPlugInLoading），需要先 `sudo xcodebuild -runFirstLaunch`。README 已说明。

### Issue 5: MLX 找不到 metallib 时 C++ 异常直接杀进程

**症状**：如果用户没装 Metal Toolchain，应用启动就挂 —— 不是 Swift throw 进 .failed 状态，是整个进程 crash。

**原因**：MLX C++ 里找不到 metallib 时 throws std::runtime_error，Swift 侧没有 catch 边界，一路传到 C runtime 最后 abort。

**修复**：在 `QwenASRRecognizer.prepare()` 前加一个 `MLXSupport.isAvailable` preflight：手动检查 `mlx.metallib` 是否与 exe 同位，不存在则直接 `setState(.failed("MLX shaders missing..."))`，永远不碰 MLX。同时 `ASRBackend.default` 和 `AppState` 初始化都做同样检查，MLX 不可用时默认后端降级到 Whisper。

文件：[`ASRBackend.swift`](../../Sources/VoiceTyping/ASR/ASRBackend.swift), [`QwenASRRecognizer.swift`](../../Sources/VoiceTyping/ASR/QwenASRRecognizer.swift), [`AppState.swift`](../../Sources/VoiceTyping/AppState.swift)

### Issue 6: 语言代码不兼容

**症状**：调 Qwen 传 `language: "zh"` 识别退化严重。

**原因**：Qwen3-ASR 期望的是**全拼英文单词**（`"chinese"`, `"english"`, `"japanese"`），不是 ISO 639 代码。它把字符串拼进 system prompt token。

**修复**：`Language` enum 新增 `qwenName: String` 属性。`whisperCode` 保留给 WhisperKit 用。

文件：[`Language.swift`](../../Sources/VoiceTyping/Support/Language.swift)

### Issue 7: WhisperKit 把 `models/` 加进 downloadBase，我的迁移少了一层

**症状**：v0.1.0 用户升级后菜单显示 "Whisper large-v3 · ~3.0 GB · Not downloaded"，但磁盘上明明有 large-v3 文件。

**原因**：`WhisperKitConfig.downloadBase` 是"放置模型的父目录"，WhisperKit 自己会在里面再建一层 `models/<repo>/<model>/`。v0.1.0 传 downloadBase = `<appsupport>/models/`，真实路径是 `<appsupport>/models/models/argmaxinc/...`（双 `models/`）。我的 v0.2.0 迁移代码只处理 `<appsupport>/models/argmaxinc/`，找不到就当没文件，isDownloaded 同样错过一层。

**修复**：`ModelStore.migrateV010WhisperLayoutIfNeeded` 改成 move `<modelsURL>/models/` → `<modelsURL>/whisperkit/models/`；`isDownloaded(.whisperLargeV3)` 的 sentinel path 也加上那层 `models/`。

文件：[`ModelStore.swift`](../../Sources/VoiceTyping/Support/ModelStore.swift)

### Issue 8: Qwen `cacheDir` 参数期望的是最终模型目录，不是 base

**症状**：Qwen 文件下载成功（菜单看得到进度条从 0 → 53%），但模型加载失败 `No safetensors files found in: <path>/qwen-asr-0.6b`。

**原因**：读 `Qwen3ASRModel.fromPretrained` 源码才发现，它的 `cacheDir` 参数不是"放置模型的父目录"，而是"模型的最终目录（要以 `/models/<org>/<model>` 结尾）"。库内部 `makeHubApi` 从 `cacheDir` 里反推 downloadBase：如果 `cacheDir.hasSuffix("/models/<id>")` 就去掉这段作 base；否则 fallback 到 `~/Library/Caches/`（结果就是下载实际去了 cache 目录，而 vocab.json / safetensors 的 load 还从我传的 cacheDir 找，两边完全对不上）。

**修复**：`QwenASRRecognizer.init` 构造 cacheDir 时主动拼上 `models/<modelId>/`：

```swift
self.cacheDir = cacheDir
    .appendingPathComponent("models", isDirectory: true)
    .appendingPathComponent(id, isDirectory: true)
```

**验证**：删掉 `config.json`（7 KB）让库 re-snapshot，确认：(a) 落地路径 = 我代码算出的路径 = MD5 与原文件一致；(b) `find` 全盘只有一个 `model.safetensors` 在正确位置，没下载到别处。

文件：[`QwenASRRecognizer.swift`](../../Sources/VoiceTyping/ASR/QwenASRRecognizer.swift)

### Issue 9: 菜单在下载进度回调下每秒重建数十次，视觉堆叠

**症状**：Qwen 下载时菜单展开着，能看到菜单整体不停重绘/叠影。

**原因**：`state.$recognizerState.sink { rebuildMenu() }` 对每个 progress 值都 fire，`progressHandler` 每秒回调几十次。NSMenu 展开时重建整个 items 数组会有肉眼可见的闪烁。

**修复**：Combine 链加 `.map { stateKind($0) }.removeDuplicates()`，把 4 个 case 映射成 0/1/2/3，同一个 kind 内部 progress 变化不再触发 rebuild。菜单里的 "Preparing… N%" 行只在菜单 open 时（`menuWillOpen`）刷一次，够用。

文件：[`StatusItemController.swift`](../../Sources/VoiceTyping/Menu/StatusItemController.swift)

## 已知限制

- 切换后端会触发 "Preparing…" 状态，如果按 Fn 录音时后端正在切换，会被拒（"Model still loading…"）
- 第一次下载 Qwen-1.7B 需要 ~1.4 GB，取决于网速需要几分钟
- WhisperKit 的下载进度仍是 indeterminate（上游 API 限制），只有 Qwen 后端有百分比
- Metal Toolchain 是一次性 setup，未装时 Qwen 后端选项在菜单可见但选中会失败
