# v0.6.0.2 — Sparkle helper 签名修正

## 背景

v0.6.0.1 真升级链路验证（#72）跑出来 fail：Sparkle UI 弹"An error occurred while running the updater"，DMG 已成功下载到 `~/Library/Caches/.../PersistentDownloads/`，但 Autoupdate 子进程在创建 `Installation` cache 子目录时被系统拒绝（`NSPOSIXErrorDomain Code=1 "Operation not permitted"`）。

## 根因

Sparkle 2 的 XPC IPC 模型要求 Autoupdate / Updater.app / Downloader.xpc / Installer.xpc 与宿主 app **要么共享真实 Apple Team ID，要么 ad-hoc 签名**。

我们用 `codesign --force --deep --sign "VoiceTyping Dev"`（自签证书）一锅端把这 4 个 helper 全部用我们的证书重签——但自签证书的 `TeamIdentifier=not set`，**夹在两条合法路径中间**：

- Team ID 路径：宿主和 helper 都 `not set`，Sparkle 不认（要求 non-empty match）
- Ad-hoc 路径：helper 不是 ad-hoc（被我们的 cert 签了）

Sparkle 启动 Autoupdate launch agent 后，内部 sandbox profile 拒它写 cache 目录 → exit 1 → 宿主等 IPC 超时 → 兜底 generic 错误。

v0.6.0 + v0.6.0.1 都是同一份 broken Makefile 出来的，**两者都不能被 Sparkle 升级**。

## 修复

`Makefile` 的 build 收尾签名段改成两阶段：

1. **Sparkle 4 个 helper 用 `codesign --sign -`（真 ad-hoc）+ `--preserve-metadata=entitlements,runtime` 重签**：
   - `Sparkle.framework/Versions/B/XPCServices/Downloader.xpc`
   - `Sparkle.framework/Versions/B/XPCServices/Installer.xpc`
   - `Sparkle.framework/Versions/B/Updater.app`
   - `Sparkle.framework/Versions/B/Autoupdate`
2. **宿主 app 用我们的自签证书签——`--deep` 必须去掉**（否则会再次覆盖回 helper 的 ad-hoc 签名）

等以后真上 Apple Developer ID 时，这两步可以塌成原来的一条 `--deep` 走 Team ID 路径。

## 这版怎么 ship

- v0.6.0.1 的 Sparkle 链路坏着，**这版不能走 Sparkle 自更新**
- 用户必须**手动拖 v0.6.0.2 DMG 到 /Applications 覆盖**
- 装上之后，**下一个**版本（v0.6.0.3 或 v0.6.1）的 Sparkle 升级链路才能真正测通

## 验证

- 手动安装 v0.6.0.2 后能打开、运行
- 下次发版时 #72 真升级验证从 v0.6.0.2 → 下一版 跑通
