# v0.6.1 开发日志

_Closed: 2026-04-26_

## 做了什么

- **HuggingFace 自动镜像兜底** (`HFEndpointResolver`)：启动时并发跑两个 GET `Range: 0-524287` 探针，比官方与 `hf-mirror.com` 的 KB/s 真实带宽，按决策表选 endpoint，结果写 UserDefaults cache（官方 24h / 镜像 6h），下次启动 sync 命中直接 setenv。后台 race 完成后再 setenv 一次（覆盖默认）。下载首次模型前 `awaitResolutionIfPending(22s)` lazy 兜底
- **零上游修改实现 endpoint 切换**：`swift-transformers` 的 `HubApi` 主动读 `HF_ENDPOINT` 环境变量。`AppDelegate.applicationWillFinishLaunching` 调 `setenv` → 全链路（WhisperKit + speech-swift）天然走对的 endpoint，不需要 fork / wrap
- **首次安装 onboarding**：检测 `state.asrBackend` 未 `isComplete` + 无 `onboardingShownAt` flag → 弹 NSAlert 确认下默认 backend (Qwen3-ASR 1.7B, ~1.4 GB)。"立即下载" 走原有 `activateBackend` 路径；"稍后" 不调用 activateBackend、只 mark flag
- **配套 Settings UI 修复**：deferred 后 recognizer 永远 nil → state 卡 `.unloaded` → 旧 UI 误显示 "Active" / "Preparing…"。改成把 `recognizerState` 显式 surface 为 `.failed("No model downloaded …")`；`statusLabel` / `actionLabel` 在 `.unloaded` / `.failed` 区分模型在/不在盘，按钮文案变 "Download" / "Retry"
- **决策表 + cache TTL 不对称的设计依据**：100 KB/s 阈值 + 5x 倍率（mirror 显著快才切）；mirror 6h TTL 比官方 24h 短，是为了误判时尽快回到 source-of-truth
- **HFEndpointResolverTests 15 个 case** 覆盖决策表 6 行 + cache TTL 边界 + 隐藏 override 优先级
- **隐藏 debug override**：`defaults write voice_typing voice_typing.hfEndpoint <url>` short-circuit 整套自动逻辑。**不进** Settings UI、不写 README

## 遇到什么

- **probe timeout 5s 太短**：dogfood 实测 hf-mirror TTFB 在 CN 慢链路 9.5s，5s 必时；同时本地 `URLSession` 跟系统代理走时被 clash-verge MITM TLS 在 10s 处崩。先把 `probeTimeoutSec` bump 到 20s + 加错误日志（host / 错误描述 / 实际耗时），下次再失败有据可查
- **rm -rf footgun**：dogfood 测试 playbook 给的 `rm -rf ~/Library/Application\ Support/...` 命令，反斜杠转义脆弱，用户复制时多一个空格把整个 `~/Library/Application Support/` 当成第一个 path 去删。所幸 macOS TCC 拦下了通讯录 / iCloud / 通话记录 / 隐私权限等敏感目录，正在运行的 app 文件被进程锁住也保住了；实际只丢了 VoiceTyping 模型 (5.9 GB) + Discord/Notion app 缓存 + Obsidian 插件配置（vault 内容安全）
- **Onboarding "Later" 路径 UI 状态不一致**：dogfood 当场暴露——Settings 显 "Active" / 按 Fn 报 "Model still loading…" / 实际啥都没在 load。根因是旧代码隐含假设 "asrBackend 被设 ⇒ activateBackend 一定调过 ⇒ recognizer 一定 non-nil"，被 v0.6.1 的 deferred 分支打破

## 为什么这么做

- **Settings 不暴露 endpoint 选项**：用户最早期版本里我提议过加 segmented control，但用户砍掉了——心智负担太重。零配置自动判定才是 v0.6.1 主题
- **bandwidth probe 而不是 HEAD ping**：HEAD 只能测延迟，CN 到 HF 的实际场景是"延迟 OK 但带宽 10 KB/s"。Range GET 512KB 直接拿吞吐量
- **5x 倍率 + 不对称 TTL**：mirror 必须显著快于官方才切（避免边界震荡）；mirror cache TTL 短一半，是因为非官方公益镜像 SLA 不确定，需要更频繁验证
- **deferred 后用 `.failed` 而不是新增 enum case**：`.failed` 已经存在且既有 UI 都正确处理过；新增 `.deferred` 要扫所有 `switch` 加 case，价值低
- **不 bundle 模型进 DMG**：模型存 `~/Library/Application Support/`，Sparkle 升级根本不动；bundle 0.6b 单 backend 也要让 DMG 涨 17 倍，每次代码补丁升级都拉 700MB —— 不值
- **不做下载过程中实时切 endpoint**：Hook 进 HubApi 下载流要么 fork swift-transformers 要么 URLProtocol 截流，复杂度爆。先 ship 启动时一次性决策，dogfood 看真实命中率再说
