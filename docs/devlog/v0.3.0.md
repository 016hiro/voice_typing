# v0.3.0 — 智能后处理 + 自定义词典 + 延迟优化

**发布日期**：2026-04-17（进行中，验证阶段）
**状态**：代码完成，端到端验证进行中

## 交付的功能

### 四档 Refiner

`RefineMode` enum：`off` / `conservative`（默认，v0.2 等价）/ `light`（+filler+结巴）/ `aggressive`（+自我纠正识别+列表格式化+语义润色）。

- `off` 直接短路不调用 LLM，`hasCredentials` 独立校验
- `aggressive` prompt 加硬约束 `output between 0.9× and 1.5× input chars`，减 LLM 生成 token
- v0.2→v0.3 迁移：首次启动若无 `refineMode` 键，从 `LLMConfig.enabled` 推断（true → conservative，false → off）

### 自定义词典（Custom Dictionary）

**数据模型**：`DictionaryEntry { id, term, hints[], note?, createdAt, lastMatchedAt? }`

**形态语义分桶**（三层 guard 防空）：
- **只 `term`** → 纯热词：ASR context + LLM glossary 的 `Preserve` 段（"don't paraphrase"）
- **`term + hints`** → 重写规则：ASR context 只用 term 锚定拼写 + LLM glossary 的 `Rewrite` 段（`hint / hint → term`）
- **只 hints / 空** → 非法，`hasContent` 属性 + `upsert` guard + UI disable 三层挡住

**存储**：`~/Library/Application Support/VoiceTyping/dictionary.json`，debounced 5 s flush，损坏时 rename `.corrupted-<ts>.json` 空启动。

**UI**：Settings 新增 Dictionary tab：
- SwiftUI `Table` + 多选 + 右键菜单 + 双击编辑 + 显式 Add / Edit / Delete / Import / Export 工具条
- 注入状态条：Qwen / Whisper / LLM 各自 `N/M entries · T/B tokens`，接近 85% budget 变橙
- 编辑时保留原 `createdAt` / `lastMatchedAt`，不破坏 LRU 排序

### 双层字典注入

两个独立的注入点，配合 `GlossaryBuilder` 做 LRU + 贪心 budget 截断：

| 注入点 | 入口 | 格式 | Budget |
|---|---|---|---|
| Qwen ASR context (zh-\*) | `Qwen3ASRModel.transcribe(context:)` | `热词：X、Y、Z。` | 460 token |
| Qwen ASR context (其他语种) | 同上 | `X, Y, Z`（对齐 Alibaba CLI `-c` 例子） | 460 token |
| Whisper prompt | `DecodingOptions.promptTokens` | 空格分隔 term 列表，经 `WhisperTokenizer.encode` 过滤特殊 token | 200 token |
| LLM glossary | 接在 RefineMode system prompt 后 | Markdown：`## User's Custom Vocabulary` + `Preserve` 段 + `Rewrite` 段 | 1500 token |

**注入算法**：
1. LRU 排序：按 `max(lastMatchedAt, createdAt)` desc
2. 贪心填到 budget × 90% 截断（不淘汰数据，只决定本次注入谁）
3. 匹配检测扫 ASR 输出 + LLM 输出，英文 `\b{term}\b`、中文 substring，命中的 entry 更新 `lastMatchedAt`

### Raw-first 注入（可选）

全局开关（默认关）。开启后 ASR 完成立刻 inject raw，refiner 后台跑；refiner 完成时检查：
1. 前台 app bundle ID 未变
2. 用户未继续打字

同时满足 → 合成 Cmd+Z 撤销 + paste refined。否则丢弃 refined。用户感知到的延迟从 "ASR + LLM + inject" 降到 "ASR + inject"。

### LatencyTracker

三段 ms 埋点（ASR / LLM / inject），写 os_log：
```
latency backend=qwen-asr-1.7b mode=conservative dict=2 rawFirst=false asr_ms=199 llm_ms=-1 inject_ms=144 total_ms=360
```
`llm_ms=-1` 表示未跑（mode=off 或 credentials 缺）。

### Settings 窗口重做（Liquid Glass）

- 菜单结构重整：Settings 升到顶级 `⌘,`（v0.2 藏在 LLM Refinement 子菜单里很别扭）
- 3 个 tab：Models / LLM / Dictionary
- macOS 26+ 用 `.glassEffect(in: .rect(cornerRadius: 22))`，旧系统 fallback `.ultraThinMaterial`
- 所有按钮去蓝色，单一强调靠材质厚度（`.glass` / `.thickMaterial`）
- 自定义 `BorderlessKeyWindow`：圆角窗口 + SwiftUI 阴影（AppKit `hasShadow=true` 会画矩形边框漏色）
- `RefineModeSegmented` 替代 `.pickerStyle(.segmented)`（去系统蓝选中态）

## 设计决策

### 为什么字典只有一张表，不拆 "Hotwords" / "Rewrites" 两栏

Typeless 这类产品也是单字典模型。80% 情况下同一个术语既要 ASR 锚定拼写又要 LLM 重写（Python、Kubernetes、公司名），拆两栏会双倍录入。单字典 + 按 entry shape 分桶处理，心智成本最低。

### 为什么 Qwen context 只注入 term，不注入 `A→B` 发音映射

实测 + 搜索证据：
- 官方 CLI 唯一示例 `qwen3-asr -c "Qwen-ASR, DashScope, FFmpeg"`——裸术语列表
- 知乎官方 blog 转载的中文示例 `热词：科创板、北交所`——同样无 `→` 替换语法
- arXiv 技术报告对 context 只有一句：`"the model learns to utilize the context tokens inside the system prompt as background knowledge"`，**无发音替换示例**
- 我们初版 `Pronunciations: 配森→Python` 格式在 Qwen 1.7B 实测里完全不起作用（`千问 Three ASR` 仍然原样输出，见验证段 log）

结论：Qwen context 是**声学锚定**（"别把 Kubernetes 写成库伯内提斯"），不是**发音替换**（"配森→Python"）。后者是 LLM refiner 的活，落到 glossary 的 Rewrite 段。

### 为什么 LLM glossary 拆 Preserve + Rewrite 两段

单段混写 `- Python (user may say: 配森, 派森)` 是"暗示"而非"指令"：LLM 可能只把它当成"注意拼写提醒"，不会做显式替换。拆两段后：
- Preserve 段动作明确："do NOT paraphrase"
- Rewrite 段动作明确："Rewrite these user pronunciations to the canonical spelling"

同一个 entry 按 `hints.isEmpty` 分桶，只出现在一段。

### 为什么字典用独立 JSON 文件，不存 UserDefaults

- 导入导出就是存储本身，零成本
- 用户想手编辑字典时 `~/Library/Application Support/VoiceTyping/dictionary.json` 可以直接改
- iCloud Drive 同步将来零代码接入

Debounced 5 s flush 避开高频 `updateLastMatched` 导致的 fsync 风暴。

### 为什么软上限 500 条而非硬 cap

真正的限制是注入时的 token budget（Whisper 200、Qwen 460）。500 只是防用户手滑导入一百万条的护栏，且 UI 能用 `Table` 顺滑渲染。

## 实现期遇到的问题与修复

### Issue 1：Qwen 初版 context 格式完全不生效

**症状**：字典里加 `Python`（hint `配森`）和 `Qwen3-ASR` 后，说"我要学配森" → 仍输出"我要学配森"；说"千问三 ASR" → 仍输出"千问 Three ASR"。Console log 确认 context 已传入。

**原因**：初版用的格式是英文指令 + `A→B` 替换语法：
```
Terms: Python, Qwen3-ASR. Pronunciations: 配森→Python.
```
这个格式和 Qwen3-ASR 训练分布不符——官方所有示例都是纯术语列表或自然段落，`→` 语法零先例，英文指令头部对中文场景权重也低。

**修复**：
1. 改为 zh-\* 用 `热词：X、Y。`、其他语种裸逗号列表，对齐 Alibaba CLI 和官方 blog
2. 撤掉 `Pronunciations:` 段——发音替换是 LLM refiner 的职责，不是 ASR
3. `GlossaryBuilder.buildForASR` 新增 `language:` 参数

文件：[`GlossaryBuilder.swift`](../../Sources/VoiceTyping/LLM/GlossaryBuilder.swift)

### Issue 2：按 Fn 立刻松手崩进程

**症状**：pid 崩退出，DiagnosticReports 里 `Swift runtime failure: Index out of range`，栈顶 `AudioPreprocessing.swift:180 WhisperFeatureExtractor.extractFeatures`。

**原因**：用户意外轻点 Fn 释放了一个接近 0 sample 的 buffer，`speech-swift` 的 reflect-padding 循环里：
```swift
let srcIdx = min(padLength - i, audio.count - 1)
paddedAudio[i] = audio[max(0, srcIdx)]
```
`audio.count=0` 时 `srcIdx=-1`，`max(0, -1)=0`，但 `audio[0]` 在空数组上直接崩。

**修复**：在 AppDelegate.stopRecording / QwenASRRecognizer / WhisperKitRecognizer 三层都加 `samples.count >= 400` guard（400 = 25 ms @ 16 kHz = 1 FFT window）。`AppDelegate` 里还 fire 了一个 "no speech" 信息胶囊。

**遗留**：上游 `soniqo/speech-swift` 的 `extractFeatures` 本身应该加 `guard !audio.isEmpty` —— 待报 issue。

文件：[`AppDelegate.swift:233`](../../Sources/VoiceTyping/AppDelegate.swift)、[`QwenASRRecognizer.swift:112`](../../Sources/VoiceTyping/ASR/QwenASRRecognizer.swift)、[`WhisperKitRecognizer.swift:79`](../../Sources/VoiceTyping/ASR/WhisperKitRecognizer.swift)

### Issue 3：WhisperKit `DecodingOptions.prompt` 参数不存在

**症状**：第一版 Whisper bias 用的是 `DecodingOptions(prompt: biasString)`，编译报 no such member。

**原因**：WhisperKit 新版 API 只接 token 串不接 raw 字符串。

**修复**：`DecodingOptions(promptTokens: [Int]?)`，走 `pipe.tokenizer?.encode(text:)` 并过滤 `specialTokens.specialTokenBegin` 以上的特殊 token。

文件：[`WhisperKitRecognizer.swift:86-108`](../../Sources/VoiceTyping/ASR/WhisperKitRecognizer.swift)

### Issue 4：Settings 窗口圆角边漏出矩形阴影

**症状**：Liquid Glass 圆角窗口外侧能看到一圈矩形边框。

**原因**：`NSWindow.hasShadow = true` 画的阴影跟 window frame 走，是直角矩形。

**修复**：`hasShadow = false`，改用 SwiftUI `.shadow()` 跟 view 的圆角走。同时 window 本体比可见面板大 `shadowMargin=28` px，留透明空间给 SwiftUI 的软阴影。

文件：[`SettingsWindow.swift`](../../Sources/VoiceTyping/Menu/SettingsWindow.swift)

### Issue 5：Liquid Glass 蓝色溢出胶囊边框

**症状**：tab pill 选中态的蓝色 halo 跑到 pill capsule 外面。

**原因**：`.glassEffect().interactive()` 在 `GlassEffectContainer` 内把 tint 的辉光延伸到容器边界，不受 capsule shape 约束。

**修复**：选中态改用 `.thickMaterial` 填充 Capsule（无颜色 tint），配合用户"去蓝色"反馈整体把所有强调色从 accent blue 换成材质厚度。

### Issue 6：编辑条目时 createdAt 被覆盖

**症状**：编辑已有 entry 后 `createdAt` 被设为 now，LRU 排序错乱。

**原因**：`commitEdit` 用 `DictionaryEntry(id: editingID ?? UUID(), ..., createdAt: Date())` 构造，`editingID != nil` 时也强制 now。

**修复**：`editingID != nil` 时查现有 entry 拷 `createdAt` + `lastMatchedAt`。

文件：[`SettingsWindow.swift:867`](../../Sources/VoiceTyping/Menu/SettingsWindow.swift)
