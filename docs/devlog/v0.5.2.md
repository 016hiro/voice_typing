# v0.5.2 — Transcription timing UI 公开 + 分析脚本前置 + dogfood 揪出两个 writer bug

> Ship date: 2026-04-23

## 背景

v0.5.1 把 `liveStreamingEnabled` 作为 UserDefaults-hidden 开关 ship 了，承诺 dogfood 一周后加 UI。v0.5.2 兑现这个承诺，同时前置 5 个 Python 分析脚本为 v0.5.3 advancement 做数据基础。

目标：
1. **Live mode 走到 UI**——新老用户都能在 Settings 里明确选择转写时机
2. **分析基础设施就位**——`Scripts/analysis/` 下 5 个 stdlib-only Python 脚本读 debug-captures 产出决策 input
3. **devdoc 规范接入**——一直在写 devlog/todo/backlog，但根目录没有 `docs/STATE.md` / `CHANGELOG.md` / `docs/conventions.md` 等。这版本补齐

## 决策

### 1. 双 Toggle → Picker（/think 回合拍板）

v0.5.1 末尾定的方案是"Streaming" SectionCard 内部两个独立 Toggle（post-record + live experimental）。实现到一半用户反馈"两个 Streaming 会混"——确实：

- `LiveTranscriber` runtime 本来就 bypass post-record 路径，双 Toggle 假装独立
- Off / Postrecord / Live 本质是一维选择（**什么时候出字**），硬拆两个开关反人性
- 用户第一次看到要先搞清 "Post-record vs Live" 的技术差异才能选

拍板改成**三选一 Picker**，标题从 "Streaming" 换成 "Transcription timing"（"when 出字"是核心差异维）：

```
After recording — one shot (default)     [DEFAULT]
After recording — segmented (VAD)        [QWEN]
While speaking — live (experimental)     [EXPERIMENTAL]
```

`AppState` 新增 `enum TranscriptionTiming { oneshot, postrecord, live }` + computed property 映射现有两个 bool。**无新 UserDefaults key、零迁移**：

```
getter:  live=true → .live / streaming=true → .postrecord / else → .oneshot
setter:  逐挡设另一个为 false，避免 legacy "both true" 状态遗留
```

### 2. native `.radioGroup` → 自定义 row-card stack（/design 回合拍板）

Picker 最初用 native `.radioGroup` 实现。截图一看跟项目其它地方的 Liquid Glass 视觉打架——radio 圆点是 Aqua 时代语言，SectionCard + RefineModeSegmented 是 macOS Big Sur+ 材质语言。

重设计成 3 行可点击 row card，视觉词汇**直接抄 `RefineModeSegmented`**：

- **选中**：不透明白渐变 + inverse 文字 + drop shadow（同 Refine 选中态）
- **未选**：半透明白容器
- **chip** (DEFAULT / QWEN / EXPERIMENTAL) 右对齐：属性标注 + disabled 提前预告
- **per-row disabled**：`one-shot` 永远可用（所有 backend）；`segmented` / `live` 只在 Qwen 可用。比整组灰更诚实
- **动画**：0.15s easeOut 切换 + button press 默认 scale

### 3. 600px 面板放不下怎么办（/think 实测回合）

新 SectionCard + 3 个 row card 让 Models tab 超出 600px 面板约 104px，Done 按钮被挤出界面。

实测量化：
- Panel 固定 760×600
- 可用 content area ~454px
- 内容总高 ~558px

选择不走 ScrollView（用户审美理由："包 ScrollView 有点丑"），改三管齐下省空间：
- 去掉 `Speech Recognition Model` SectionCard 标题（3 个 ASR 名字自解释）
- 去掉 `Downloads cached under ~/Library/...` footer（路径进 runbook.md）
- row card 去 description Text，改成 `.help(description)` hover tooltip

三项合计节省 ~138px，剩 ~420px 内容高 + ~34px buffer。

### 4. 4 脚本 → 5 脚本（反思回合拍板）

scope doc 原定 4 个脚本覆盖 dogfood 基础分析。反思一轮发现缺关键一个：

- `live_drain.py` 只量"末段后延迟"，dominated by 最后一段
- 但用户体感由**每段 ASR 延迟 + RTF** 决定——中间如果一段卡 800ms，drain 再短也救不回来
- 补 `segment_latency.py`：per-segment `transcribeMs` + RTF (= transcribeMs / audio_ms) + first-vs-rest cold/warm 对比

最终 5 个脚本：

| 脚本 | 答什么问题 |
|---|---|
| `summary.py` | dogfood pool 够不够？live 占比？哪个 backend / 语言主流？|
| `hallucination_review.py` | HallucinationFilter 是否过激？per-backend rate 差异？|
| `live_drain.py` | Fn↑ → 末段 inject 完成延迟分布 |
| `focus_drop.py` | focusChanged + skipped 双重 drop，per-app 排序 |
| `segment_latency.py` | 每段 ASR 延迟 + RTF + cold/warm 差 |

脚本共用 `_common.py`（iter_sessions / load_* / parse_iso / percentiles / histogram_ascii / human_bytes）。**无第三方依赖**——`python3 summary.py "$ROOT"` 直接跑。

### dogfood 数据跑 5 脚本（2026-04-23）

本版收尾前拿 303 个累积 session 跑全套：

- **37 个有效 session**（Qwen 0.6B / zh-CN / 36 live + 1 batch）
- **Qwen 0.6B 表现优秀**：per-segment p50 **108ms** / p95 235ms；RTF p95 **0.27**（4× 实时）；first vs rest delta **-9%**（dl_init 修已消除 warmup）
- **Focus drop 3.3%**（Warp 1 次）—— 低于 5% 阈值

但也**揪出 3 个 writer bug**（留 v0.5.3 修）：

1. **87% session 缺 meta.json**（266 / 303）—— `DebugCaptureWriter.begin()` 只创目录不写 partial meta，任何 crash/force-quit/early-bail 都丢 metadata
2. **timestamp 秒精度**—— `JSONEncoder.dateEncodingStrategy = .iso8601` 默认无 fractional seconds，`live_drain.py` 完全失效（endedAt == last_inject 同秒 → delta = 0）
3. **Qwen 0.6B prompt echo 率 40%**—— 拦掉的 20 段**全部**是 dictionary 列表 `'热词：Python、Qwen3-ASR、...'` 的字面回显。filter 工作正常但频率远高于 5% baseline（多短音频触发）

以上 3 个挪到 `docs/todo/v0.5.3.md`。分析脚本基础设施证明价值——一轮跑就把藏在数据里的 bug 挖出来。
