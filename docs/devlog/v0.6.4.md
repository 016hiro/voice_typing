# v0.6.4 — ASR keep-alive (anti-compressor)

> Ship date: TBD
>
> Branch: `v0.6.4` (并行于 v0.6.3 主线 refiner 工作)

## 背景

v0.6.3 #R4 spike 排查 mlx-swift-lm 集成路径时无意诊断出：1.7B Qwen ASR 闲置 1-2 小时后第一次 Fn 按下要等 9-30 s（基线 99-550 ms，慢 17-56 倍）。根因是 macOS 在 unified memory 上跑的 compressor 会把长期未 access 的物理页（包括 mmap 的 2.5 GB MLX 权重）压成冷页，下次 access 必须先解压。Whisper backend 不受影响——CoreML 把权重塞 ANE 池，不走 RSS / compressor 路径。

这是从 v0.6.0 切到 MLX backend 那天就存在的存量 bug，只是当时没人闲置那么久才回头按 Fn。

## 做了什么

- **`Sources/VoiceTyping/ASR/ASRKeepAlive.swift`**：90 s 周期 `Timer` 驱动 200 ms 静音 dummy `transcribeSegmentSync`。每次 tick 让 MLX runtime 真实触摸全部权重 page，compressor 看到 "recently accessed" 就不会瞄准这些页
- **`KeepAliveTarget` protocol**：把 `var state` + `transcribeSegmentSync` 抽成最小接口，让 unit test 拿 fake 跑而不需要 MLX shaders + 1.4 GB 模型下载。`QwenASRRecognizer` 直接 conform，零侵入
- **AppDelegate wire-up**：observe recognizer state stream → `.ready` 时 `start(target: qwen)`，任何 non-ready 状态 `stop()`，backend swap 在 `activateBackend` 入口处主动 `stop()`，`applicationWillTerminate` 也 `stop()`
- **`ASRKeepAliveTests.swift`**：12 个 case，覆盖常量 pin（cadence + 200 ms 长度 + ≥400 sample FFT 下限 + all-zero）、state-aware skip 四种状态、target replacement、stop 后 idempotent、真 timer 跑 350 ms 看 tick 数

## 为什么这么做

- **dummy transcribe vs `mlock()`**：`mlock` 需要 root + 物理 RAM 钉死 2.5 GB，24 GB Mac 都嫌占。dummy transcribe 让 OS 自己决定保不保活，零特权
- **90 s cadence**：compressor 的 decision window 经验观察是 5-10 分钟，90 s 远比这窄。CPU 成本约 0.3% 持续。30 s 也行但收益边际，180 s 开始可能失效。**写常量不入 Settings**——心智负担不值，dogfood 数据如显示需要调就改 const
- **200 ms 静音**：太短（< 100 ms）某些 backend 可能短路 path 不真过 weights；太长（> 500 ms）占 ANE 队列阻挡用户真实请求。200 ms 走完整推理 path 又短到不会感知发热
- **不保活 refiner**：用户明确决策——refine 本来就有"等"心智模型，第一次冷解压几秒可接受。Settings 后续会加一句说明
- **App 后台 / 失焦不停**：用户切去 IDE 写代码、回头 Fn 必须命中 warm。NSTimer 默认进 sleep 时 OS 暂停、wake 自动恢复，不需要额外处理
- **state-aware skip**：tick 里检查 `case .ready = state`。loading / failed / unloaded 都跳过——不同状态机阶段下 dispatch 一个 dummy transcribe 风险/收益不对称

## 遇到什么

- **state 在 transcribe 期间不变**：v0.6.4.md 设计稿说 "用户 Fn 录音期间 tick 因 state != .ready 跳过"。重读 `QwenASRRecognizer.transcribe` 发现它实际上不改 state，所以 tick 不会因为 state 跳过——只会在 `transcribeLock` 后排队。最坏多等 200 ms，可接受。代码注释里没复述这条错误的设计稿假设
- **`SourceKit "No such module 'Sparkle'"`**：编辑 `AppDelegate.swift` 时 SourceKit 报 Sparkle 找不到。预存量 indexing 警告，`swift build` 真编译不影响。无关 v0.6.4 改动
