# v0.1.0 — 首个可用版本

**发布日期**：2026-04-16
**状态**：基础功能可用，开发期内部使用

## 交付的功能

- 全局 Fn 键监听（按住录音、松开转录），通过 CGEventTap 抑制 Fn 默认行为（emoji picker）
- 五语言切换：English / 简体中文 / 繁體中文 / 日本語 / 한국어，默认 zh-CN
- 菜单栏图标 + 动态菜单（语言子菜单、LLM Refinement 子菜单、About、Quit）
- LSUIElement 模式，无 Dock 图标
- 屏幕底部居中浮动胶囊 HUD：
  - NSPanel (.nonactivatingPanel) + NSVisualEffectView (.hudWindow) + 圆角蒙版
  - 5 根真实 RMS 驱动的波形条（权重 [0.5, 0.8, 1.0, 0.75, 0.55]，attack 40% / release 15%，±4% 抖动）
  - 文字弹性宽度 160-560 px，spring 0.25 s 平滑过渡
  - 入场 0.35 s alpha，退场 0.22 s alpha
- WhisperKit `openai_whisper-large-v3` 本地识别，首次启动自动下载到 `~/Library/Application Support/VoiceTyping/models/`
- `SpeechRecognizer` 协议化，未来可平替其他 ASR 后端
- 剪贴板 + 合成 Cmd+V 注入；CJK IME 检测后临时切到 ABC 键盘布局，注入完成后还原 IME 和剪贴板
- 可选 LLM 后处理（OpenAI 兼容 chat completions），保守纠错 system prompt
- LLM 设置窗口：API Base URL / API Key（含眼睛切换 + 清空按钮）/ Model / Test / Save
- 麦克风 + 辅助功能权限请求，缺权限时菜单顶部出现授权引导项，2 s 轮询自动恢复

## 开发期遇到的问题与修复

### Issue 1：WhisperKit `modelFolder` 误用

**症状**：菜单显示 `Model load failed: Model file not found at /Users/.../models/MelSpectrogram.mlmodelc`

**原因**：`WhisperKitConfig` 的 `modelFolder` 字段是"已存在的模型文件所在目录的精确路径"，不会在里面下载子目录。我把它当成了"下载的根目录"。

**修复**：改用 `downloadBase: URL`，WhisperKit 会自动建 `argmaxinc/whisperkit-coreml/openai_whisper-large-v3/` 子树并下载完整模型。

文件：[`WhisperKitRecognizer.swift`](../../Sources/VoiceTyping/ASR/WhisperKitRecognizer.swift)

### Issue 2：模型大小估算错误

**症状**：以为 `large-v3` 是 1.5 GB，提前在 1.4 GB 时认为下载完成并 kill 了进程，导致 AudioEncoder 的 weights 文件不完整。

**原因**：`large-v3` 的 CoreML 格式实际占 ~3 GB（AudioEncoder ~1.2 GB + TextDecoder ~1.5 GB + MelSpectrogram + 配置）。OpenAI 的原始 checkpoint 是 ~1.5 GB，但 CoreML 编译后体积接近翻倍。

**修复**：知识更新；下次启动 WhisperKit 自动续传补全；README/docs 中标注真实大小。

### Issue 3：`CGEventType.systemDefined` 不存在

**症状**：编译错误 `type 'CGEventType' has no member 'systemDefined'`

**原因**：NX_SYSDEFINED (类型 14) 只在 `NSEvent.EventType` 中暴露，CGEventType Swift 桥接没有这一项。

**修复**：移除 `systemDefined` 相关 mask 和 case，仅监听 `kCGEventFlagsChanged`（对内置 + 大多数 Apple 键盘的 Fn 检测足够）。详见 [`FnHotkeyMonitor.swift:39-42`](../../Sources/VoiceTyping/Hotkey/FnHotkeyMonitor.swift)。

### Issue 4：`main.swift` 调用 `@MainActor` 初始化器

**症状**：编译错误 `call to main actor-isolated initializer 'init()' in a synchronous nonisolated context`

**原因**：`main.swift` 顶层代码不自动是 MainActor，但 `AppDelegate` 标了 `@MainActor`，从顶层直接 `AppDelegate()` 跨越了 actor 边界。

**修复**：用 `MainActor.assumeIsolated { … }`（macOS 14+）包住 NSApplication 启动代码。我们知道 main.swift 在主线程跑，这个 API 让编译器同意。

文件：[`main.swift`](../../Sources/VoiceTyping/main.swift)

### Issue 5：胶囊圆角处的彩色直角边框

**症状**：`.hudWindow` 材质胶囊在圆角四角外有可见的彩色边缘漏光。

**原因**：`NSVisualEffectView` 对 `layer.cornerRadius + masksToBounds` 支持不干净，模糊材质的渲染路径不完全遵守图层圆角裁剪。

**修复**：改用 `effect.maskImage = capsuleMaskImage(radius: 28)`，构造一张 9-slice 可拉伸（`NSImage.capInsets` + `.resizingMode = .stretch`）的圆角图作为形状蒙版。这是 NSVisualEffectView 自定义形状的官方推荐做法。

文件：[`CapsuleWindow.swift`](../../Sources/VoiceTyping/UI/CapsuleWindow.swift)

### Issue 6：第一次转录后崩溃（SIGTRAP / EXC_BREAKPOINT）⚠️ 严重

**症状**：录音→转录→注入流程到注入阶段时进程崩溃。crash report 显示：

```
dispatch_assert_queue_fail
TSMCurrentKeyboardInputSourceRefCreate
InputSourceManager.currentSource()
TextInjector.inject(_:)
```

**原因**：Carbon Text Input Services API（`TISCopyCurrentKeyboardInputSource` 等）内部用 `dispatch_assert_queue` 强制要求主队列，否则直接 trap。我把整个 `inject` 放在 `Task { … }`（cooperative 队列）里执行了。

**修复**：`TextInjector` 类标 `@MainActor`。AppDelegate 中 `await self.injector.inject(finalText)` 会自动跳到主线程执行。CGEvent.post 也是主线程敏感，顺带受益。

文件：[`TextInjector.swift:3`](../../Sources/VoiceTyping/Inject/TextInjector.swift)

**教训**：所有 Carbon/CoreFoundation Toolbox API 默认假设主线程。下次接触老 C API 时先查文档线程要求。

### Issue 7：重新构建后辅助功能权限失效

**症状**：用户已经在 System Settings 中授予了辅助功能权限，但每次 `make build` 后菜单又显示 "Grant Accessibility Permission…"。

**原因**：ad-hoc codesign（`codesign --sign -`）每次重建产生不同的 cdhash，TCC 把每次构建当成"新应用"，旧授权不认。

**当前缓解**：

- 让用户在 System Settings 里移除并重新添加 VoiceTyping
- 或 `tccutil reset Accessibility com.voicetyping.app`

**长期方案**（未实现，进 backlog）：用本地自签名证书（`security create-keychain` + `security create-certificate`）签名，公钥固定后 designated requirement 跨重建稳定。
