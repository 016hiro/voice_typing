_Closed: 2026-05-02_

## 做了什么

- Refine 输出从"沉默几秒 → 一次性 paste"变成"边生成边 paste"。Cloud SSE 把累积改 yield；本地 mlx-swift-lm 换 `streamResponse(to:)`，前 30 字 head buffer 兜 Qwen3 `<think>\n\n</think>` bookend
- 新 `TextInjector.injectIncremental(stream:)` 路径：单次 pasteboard snapshot/restore + IME bypass，按标点 / 换行 / ≥8 char + whitespace / 32 char 硬上限四条 flush 边界（11 个 unit tests 覆盖）
- `RefineDelivery` enum 替换 `rawFirstEnabled` Bool。Settings UI 从 toggle 改 segmented picker
- Live mode 按 backend 二分流：local → per-segment streaming refine，跨段共享一个 mlx `ChatSession` 让历史段做后段的 prompt context；cloud → session-end batch refine + Cmd+Z×N replace
- Capsule 流式状态显示 `Refining (N chars)`，Esc / focus-loss 取消（pipelineTask 把 cancellation 传到 stream pipeline）
- Notion bundle ID `notion.id` 加 deny-list，命中 streaming 自动降 batch（详见 `decisions/0001-stream-inject-path.md`）
- v0.7.0 dogfood 收尾时把 raw-first case 整段砍掉（详见下方"为什么"）

## 遇到什么

- **R1 spike 决定一切**：流式 inject 三个候选（Cmd+V incremental / CGEvent keystroke / NSAccessibility insertText）实测下来只有 Cmd+V 在 7 个目标 app 全部 lossless。CGEvent 在 Notion 直接吞 6+ 字符（autocomplete eats chunks at boundaries），AX 在 Electron / Terminal 全废。Cmd+V 在 Notion 也有 block-splitting 副作用，但内容 lossless 可以 Cmd+Z 整段撤销，于是 Notion 加 deny-list 走 batch fallback
- **R9 design 拍板错了一次**：think skill 阶段我替用户拍 "live mode 用户不期望 progressive，期待录完出整段干净文本" → R9 v1 实装 session-end batch refine + Cmd+Z×N replace。dogfood 时用户立刻反馈 "我期待 per-segment refine"。R9 redo 改为按 backend 二分流。教训写在了下方"为什么"段
- **mlx-swift-lm `ChatSession` 多轮 history 复用**是这次最优雅的发现——同一 live session 跨段共享 ChatSession，历史段自动作为后段的 prompt context，跨段指代（"它"/"that"）解决得很自然。KV cache 复用让段 N+1 比 N 还快
- **诊断 bug 走过两个弯路**：
  - 用户报 "live + refine `llm_ms=-1` 进不了 refine 阶段"。我跳到 "willRefine gate 关掉了"，commit 了一个 `refinerReady` fix（独立的 latent bug，但不是用户报告的根因）。后来意识到 R9 redo per-segment refine 在 `liveInjectTask`（detached）跑，pipelineTask 的 tracker 看不见 → `llm_ms=-1` 是 logging gap 不是 refine 没跑。修法是把 per-segment infer_ms 累计回 tracker
  - 用户报 raw-first "refined 不替换 raw 反而 append 在后面"。诊断是 CJK 输入法拦截 Cmd+Z 当 composition undo。修法没做完（写到一半 `import` 报错没编译过），用户决定直接砍 raw-first feature——streaming 已经覆盖了它的低延迟用例，Cmd+Z + IME 这条脆弱路径不值得维护

## 为什么这么做

- **流式 inject 选 Cmd+V incremental**：spike 测出来唯一 lossless across all targets 的方案。CGEvent / AX 在 Cmd+V 失败的 app（Notion / Electron / Terminal）上也都失败，没有 fallback 价值，只增复杂度。详见 `decisions/0001-stream-inject-path.md`
- **R9 redo 二分流而不是统一 design**：Cloud 调用贵（API 钱 + 网络 RTT），Local 调用免费。同样的 progressive UX trade-off 给两种 backend 算账结果不一样——cloud 保 batch 省钱，local 走 per-segment 享受 KV cache 复用。一个 `state.localRefinerEnabled` 的判断分流，简单且语义对齐用户预期
- **raw-first 砍掉**：这个 feature 是 v0.3 引入的"先 paste raw 后台 refine 再 Cmd+Z 替换"，目的是降感知延迟。v0.7.0 streaming 在感知延迟上严格更好（不闪烁、不需要替换、refine 边产边 paste），raw-first 的存在意义被覆盖。Cmd+Z + CJK IME 那条路径本来就脆（IME 拦截 Cmd+Z 当 composition undo），dogfood 暴露后没人愿意修一个用户已经不需要的 feature。enum 从三段降两段，老 `rawFirstEnabled=true` 用户迁移到 streaming
- **Live 模式的 `RefineDelivery` 设置故意被忽略**：live 模式跟 one-shot 是不同的 ASR 模式，混淆 picker 语义会误导用户。live 路径完全由 `localRefinerEnabled` 决定，这个判断更直接对应"我有没有本地模型可用"
- **拍 design 前没问用户**这次踩了坑。R9 在 think skill 阶段我替用户拍 "用户不期待 progressive"，dogfood 第一天就被打脸。教训：design 涉及"用户期待 X 还是 Y"这种主观偏好时，**先问，不替**。后续做 ADR 类设计选择记得加这条 sanity check
