## 做了什么

- **#B1 inline refine telemetry bug 修复**：v0.7.0 以来 `LocalLiveSegmentSession` 路径根本没写 `refines.jsonl`——5 天 dogfood 数据全丢。给 `LocalLiveSegmentSession` 加 `appendRefine` 闭包，`runSegment` 末尾构造 `RefineRecord` 写回。session-end batch refine 也补齐 MB 字段。
- **#B2 中文"点 → ASCII ."规则**：refine prompt（light + aggressive）加 1a 规则——`com 点 → .com`、`V 一点二点三 → V1.2.3` 这类版本号 / 域名 / 文件后缀里的"点"明确改成 ASCII。Glossary 对量词搭配（"三点了" / "锚定点"）做反例，避免误改。
- **#B3 VAD minSilenceDuration 0.7 → 1.5 s**：live mode 短停顿被切成两段、inline refine 各跑一次的问题。1.5 s 稍激进但 force-split（25 s 上限）兜底，平均段长升 ~30%，inline refine 调用次数降。
- **#B5 Refiner pin toggle (Settings → LLM)**：复用 ADR-0002 的 ASR `WiredMemoryTicket` 模式扩到 refiner（2.5 GB ceiling for Qwen3.5-4B 4-bit）。lazy semantics：toggle on 时若 container 未 load 仅 set flag；下次 `ensureLoaded()` pin。Combine 把 `AppState.localRefinerPinned` 接到 `LocalMLXRefiner.setPinned` 做单一数据源。16 GB Mac 上加 warning（系统可能 clamp ticket 到低于 2.5 GB）。
- **#B6 HallucinationFilter false-positive on single hotwords + reason telemetry**：用户短按只说 "Qwen"/"VAD"/"Rust"，ASR 原样输出，但 Layer 2b 的 `normalizedCtx.contains(normalized)` 把它当 prompt-echo 干掉。短词 normalize 4-5 字符必然被长 context 包含。去掉 `contains(normalized)` 分支只保留反向 `normalized.contains(normalizedCtx)`，加 `filterReason: String?` 字段透到 `SegmentRecord` 区分 `.blacklist / .promptEchoChinesePrefix / .promptEchoSubstring`。
- **#B8a per-refine MLX memory snapshot telemetry**：`LocalMLXRefiner.memSnapshotMb()` 拿 `MLX.Memory.snapshot()`，refine 跑完往 `refines.jsonl` 写 `mlxActiveMb / mlxCacheMb / mlxPeakMb` 三个 Int?。Optional 保证旧 jsonl 解析 + cloud-only 路径仍可用。同步往 `Log.llm.notice` 落一条便于实时观察。
- **#B8b MLX cache pool clamp at 1 GB**：`main.swift` 一行 `MLX.Memory.cacheLimit = 1_000_000_000`，在 `AppDelegate` 构造前生效。决策依据见 `decisions/0003-bound-mlx-cache-pool.md`。
- **ADR-0003**：把 pin（守 active）和 cacheLimit（守 cache pool）两个机制的职责划清——两条性能 regression 路径来自不同 MLX 内存区域，正交，不能互相替代。

## 验证数据

cacheLimit OFF vs ON 端到端（同样的 telemetry 看两遍）：

| 状态 | n_refines | cacheMemory 区间 | 进程 RSS (uptime) | swapped |
|---|---|---|---|---|
| 老 build 跑 2 天（无 cacheLimit） | 7 | 2.4 → 5.6 GB ⬆️ | **16.1 GB** (49 h) | 13.6 GB |
| cacheLimit=1 GB | 6 | **986 → 1000 MB 横走** | **5.6 GB** (9.5 h) | 112 KB |

ASR 延迟（老 build 在 IOAccelerator 15.6 GB 状态下，per-segment `transcribeMs`）：

| 进程 uptime | n | p50 | p90 |
|---|---|---|---|
| 0-6 h | 17 | **340 ms** | 883 ms |
| 48-54 h | 37 | **831 ms** | 1804 ms |

修后 9.5 h+ uptime 新进程 p50 = 333 ms——和 0-6h baseline 持平，长跑漂移曲线被消除。

## 遇到什么

- **#B5 scope 失控（意外但及时）**：原始 #B5 是"用 #B4 dogfood 数据决定要不要 pin refiner"。调研期间发现真正的 silent killer 不是 cold-after-idle 而是 long-uptime drift——MLX cache pool 无上限，2 天涨到 15 GB。pin（守 active）和 cacheLimit（守 cache pool）必须**同时**上才完整。原本一个 ticket 变成 #B5 (pin) + #B8a (telemetry) + #B8b (cacheLimit) + ADR-0003 四个产物。
- **`WiredSumPolicy` 命名冲突**：`MLX` 和 `MLXLMCommon` 都 export 这个类型。`LocalMLXRefiner` import 了两者，必须 `MLX.WiredSumPolicy()` 显式 qualify。ASR 文件只 import `MLX` 自动选对——所以这坑只在 refiner 触发。
- **#B1 修完才暴露 #B8**：没有 per-refine telemetry 之前长跑 latency drift 看不见——只看到 oslog 里偶尔一条慢 transcribe，没人能定量它在涨。jsonl 流入 + MB 快照后，第一次 refine 就 cache = 2.4 GB 把假设打实。教训：observability 修复永远是先决条件，不是顺手做的事。
- **首版 cacheLimit 估值（512 MB）偏紧**：基于 mlx-swift docs "2 MB 都常常够"的直觉。#B8a 数据出来后看到单次 refine working-set ~1.3 GB，调到 1 GB 才合理。docs 的经验值是给固定 shape 工作流，可变长度语音段不适用。

## 为什么这么做

- **pin + cacheLimit 双机制不冗余而是正交**：两条性能 regression 路径（idle eviction / uptime drift）来自不同的 MLX 内存区域。详见 `decisions/0003-bound-mlx-cache-pool.md`。
- **pin 做 runtime toggle 不做 always-on**：refiner 是 opt-in feature（默认 cloud），2.5 GB wired 不能强加给没启用 local refiner 的用户。toggle UI 给 power user 一个明确的内存换延迟开关。16 GB Mac 上 warn 是因为系统可能拒绝完整 ticket size。
- **cacheLimit 进 `main.swift` 不进 `LocalMLXRefiner`**：process-global setting，必须早于任何 MLX 调用。放进 actor 初始化里风险是有人 `MLX.Memory.snapshot()` 提前触发分配。`main.swift` 顶层最稳。
- **`refines.jsonl` 加 Optional Int? 字段不加新 schema**：向后兼容旧 jsonl 解析；cloud-only 路径仍能写出干净记录。Codable 默认 skip nil，老 reader 看不到字段也不报错。
